# P2 — TUI App restructure

**Status:** in-progress **Started:** 2026-04-30 **Landed:** —

P2 splits today's god-struct `App` into a headless-safe `Core` plus
`TuiApp` / `HeadlessApp` frontends, carves the subsystems out as
their own types (Session, Confirms, Clipboard, Timers, Cells,
LuaRuntime, ToolRuntime, EngineBridge, AppConfig, WellKnown), installs
the `Host` + `UiHost` traits (absorbing P1.e's trait + supporting
types), and collapses the event surface onto a single `select!` loop
publishing through `Cells`. Sub-phases land independently, each green
at the boundary. Detail per sub-phase lives in the commit message;
this file is an index.

## Sub-phases landed

- **P2.a.1** (`b07f803`) — `WellKnown { transcript, prompt, statusline }` carves the well-known `WinId`s off App; `Ui::win_buf{,_id,_mut}` resolves `WinId` → backing `Buffer`.
- **P2.a.2** (`03bd06e`) — `WellKnown.cmdline: Option<WinId>` absorbs today's `App::cmdline_win`.
- **P2.a.3a** (`b75a599`) — `Confirms` data carve-out: `pending` map + `next_handle` counter move off App into a typed subsystem with `register / get / take`; `ConfirmEntry` relocates from `dialogs/confirm.rs` to `app/confirms.rs`.
- **P2.a.6** (`a8cbe34`) — `AppConfig` data carve-out: 16 fields (`model` / `api_base` / `api_key_env` / `provider_type` / `available_models` / `model_config` / `cli_*override` / `mode` / `mode_cycle` / `reasoning_effort` / `reasoning_cycle` / `settings` / `multi_agent` / `context_window`) lift off `App` into `app::app_config::AppConfig`; call sites read through `self.config.*`.

## Decisions made

- **WellKnown shape** (`b07f803`) — three stable WinIds (prompt /
  transcript / statusline) only; cmdline (currently
  `App::cmdline_win: Option<WinId>`) folds in next sub-phase. The
  redundant `input_display_buf` / `transcript_display_buf` /
  `status_buf` `BufId` fields retire entirely — buffers are reached
  through the new `Ui::win_buf{,_id,_mut}` helpers, matching the
  ARCHITECTURE.md target where `WellKnown` carries only WinIds.
- **P2.a sub-phase split** (`b07f803`) — P2.a is too large for one
  session; split into `a.1` … `a.12` with independent landings.
  Order is data-only carve-outs first (WellKnown, Confirms,
  AppConfig, Session), then new reactive primitives (Cells, Timers),
  then bridges (LuaRuntime/Cells fold, ToolRuntime, EngineBridge),
  then the Core / TuiApp / HeadlessApp aggregate.
- **P2.a.3 split into a.3a / a.3b** (`b75a599`) — the data carve-out
  (struct + register/get/take helpers) is mechanical; the
  oneshot::Sender swap needs `Cells` for the request-payload route.
  a.3a lands the carve-out; a.3b lands the channel rewrite +
  `is_clear()` gate alongside `EngineBridge` once a.4 is in. No
  consumers exist for `is_clear()` until the bridge fold, so adding
  it now would be premature.
- **AppConfig holds runtime-mutable fields too** (`a8cbe34`) — `mode`,
  `reasoning_effort`, `model`, `model_config` flip during a session
  (Shift+Tab cycles `mode`; `/model` rewrites `model`; pricing
  resolves into `model_config`). They live on `AppConfig` anyway
  because the eventual `Cells` migration (P2.a.4) reads them as cell
  setters' source of truth — segregating "immutable startup" from
  "runtime-mutated" inside this struct would just split one logical
  unit. `startup_auth_error` stays on `App` (one-shot transient,
  drained on first frame).
- **Data-only carve-outs precede a.4 (Cells)** (`a8cbe34`) — REFACTOR's
  P2.a sub-phase listing is numerical (a.4 → a.5 → a.6 …), but the
  text body says "data-only carve-outs first, then the new reactive
  primitives." a.6 / a.7 / a.8 are pure data lifts and can land
  before the Cells primitive without touching its design. Pick the
  data lifts now so a.4's migration step (built-in cells fed by
  these fields) reads against a clean target shape.
