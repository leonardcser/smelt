# P2 — TUI App restructure

**Status:** in-progress **Started:** 2026-04-30 **Landed:** —

P2 splits today's god-struct `App` into a headless-safe `Core` plus
`TuiApp` / `HeadlessApp` frontends, carves the subsystems out as
their own types (Session, Confirms, Clipboard, Timers, Cells,
LuaRuntime, ToolRuntime, EngineBridge, AppConfig, WellKnown), installs
the `Host` + `UiHost` traits (absorbing P1.e's trait + supporting
types), and collapses the event surface onto a single `select!` loop
publishing through `Cells`. Sub-phases land independently, each green
at the boundary. Detail per sub-phase lives in the commit message;
this file is an index.

## Decisions made

- **WellKnown shape** (`b07f803`) — three stable WinIds only; redundant `BufId` shadows retire, buffers reach via `Ui::win_buf{,_id,_mut}`.
- **P2.a sub-phase split** (`b07f803`) — split into a.1…a.12: data lifts first, reactive primitives next, bridges, then the Core / TuiApp / HeadlessApp aggregate.
- **P2.a.3 split into a.3a / a.3b** (`b75a599`) — data carve-out is mechanical; oneshot::Sender swap + `is_clear()` gate need Cells (a.4) and EngineBridge.
- **AppConfig holds runtime-mutable fields too** (`a8cbe34`) — `mode` / `reasoning` / `model` flip during a session and feed Cells migration; segregating "immutable" from "runtime-mutated" would split one logical unit.
- **Data-only carve-outs precede a.4 (Cells)** (`a8cbe34`) — REFACTOR's text body ("data lifts first") trumps its numerical listing; a.6 / a.7 / a.8 are pure data and can land before the primitive.
- **P2.a.7 split into a.7a / a.7b** (`db9fecc`) — snapshots is small + contained; `history → session.messages` is ~25 rename sites, too large to combine.
- **`Session.session_cost_usd` is persisted** (`db9fecc`) — `#[serde(default)]` field rather than computed from `cost_snapshots.last()`; load path stays correct without a schema migration.
- **Timers lands before Cells** (`90df0fa`) — Timers is independent of Cells; landing first means `now` / `spinner_frame` drivers are ready when Cells arrives.
- **a.8 fits the data-only carve-out tier** (`2db595a`) — routing the two clipboard bypass paths through `app.clipboard.*` is mechanical, finishes the data tier ahead of a.4.
- **P2.a.4 splits into a.4a / a.4b / a.4c** (`63e7ff9`) — primitive + bindings (a.4a); userdata-handle ergonomics + glob/au alias (a.4b); built-in cell migrations + Rust subscribers (a.4c).
- **`Rc<dyn Any>` over a custom `CellValue` trait** (`63e7ff9`) — blanket-impl `as_any` produced wrong TypeId; `Rc<dyn Any>::downcast_ref` works directly with no trait dispatch.
- **Single Lua-only `SubscriberKind` variant for now** (`63e7ff9`) — Rust variant warned dead (no built-in subscriber yet); a.4c adds `Rust(...)` when the first Rust-typed subscriber lands.
- **`PendingCallback { kind, is_glob }` over a new SubscriberKind variant** (`9f6596e`) — keeps the kind enum to one shape; the drain pump branches on `is_glob` to call `fn(value)` vs `fn(name, value)`.
- **Glob subs share the SubscriptionId space** (`9f6596e`) — one `next_id` counter for direct + glob keeps the bookkeeping uniform; `unsubscribe_glob(id)` is named distinctly so callers can keep their existing `(name, id)` direct-unsubscribe shape.
- **Strict `smelt.au.*` aliasing — no auto-declare** (`9f6596e`) — `au.on` / `au.fire` return `nil` / `false` for undeclared cells rather than auto-declaring with `nil`; built-ins are declared at startup in a.4c, plugin authors keep using `smelt.cell.new`.
- **a.4c splits into a.4c.1 / a.4c.2 / a.4c.3** (`94d6a2f`) — machinery + simple-typed setters; timer-driven cells (`now`, `spinner_frame`, `tokens_used`); event-shaped cells (some gated on EngineBridge a.11). REFACTOR.md split body lists the per-sub-phase scope.
- **Per-`TypeId` `LuaProjector` over per-cell projectors** (`94d6a2f`) — projector keyed by `TypeId::of::<T>()` not by cell name. One `String` projector serves every `String`-typed cell; new types register once. `(*value).type_id()` on `&dyn Any` returns the inner concrete TypeId so the lookup happens at fire / read time without a stored type tag.
- **`EventStub` placeholder over deferred declaration** (`94d6a2f`) — declare every event-shaped cell at startup with an `EventStub` value (projector returns `nil`) so `smelt.au.on("turn_complete", fn)` succeeds today; the un-migrated setter just never fires until a.4c.3 lands its publish call. Avoids "cell not declared" errors while a.4c.3 ramps in.
- **Diff-publish for `vim_mode` / `confirms_pending`** (`94d6a2f`) — `Cells::publish_if_changed` short-circuits when the new value equals the slot's current `T`; `App::publish_diff_cells` runs once per tick before `drain_cells_pending`. The alternative (every `vim_state.set_mode` call site publishing) would scatter `cells.set_dyn` across `ui::Window`, `ui::vim`, App mouse / events / lua_handlers — the per-tick diff stays in one App-level method.
- **`now` is unix seconds u64, not a chrono `DateTime`** (`49f0745`) — no chrono / time crate is in the tree and the loop is idle-CPU-zero; publishing a u64 keeps the projector primitive (already registered) and lets Lua format with `os.date("%H:%M:%S", v)`. ARCHITECTURE.md says `DateTime`; that's the semantic intent — storage shape is u64 today and a chrono swap is one declaration / projector if a real consumer wants typed dates.
- **`tokens_used` excludes background requests** (`49f0745`) — auxiliary requests (title, compaction, btw, predict) carry `background = true` so a subscriber on `tokens_used` sees only the user-visible context flow, matching the existing displayed-context behaviour. Sub-agent token usage routes through `agent.context_tokens` separately and never touches this cell.
- **`spinner_frame` rolls back to 0 when no turn is animating** (`49f0745`) — `publish_diff_cells` reads `working.is_animating()` and falls back to 0 outside a live turn so a subscriber sees a clean rollover at turn-end and never fires again until the next turn — instead of latching at the last live frame and re-emitting every time `working.elapsed()` hands back the archived `last` duration.
- **`EventStub` declarations stay; typed projectors route by stored TypeId** (`dd997df`) — keeping the `EventStub` placeholder declared at startup means a `:get()` before the first publish reads `nil` rather than a synthetic default. Once `set_dyn` writes a typed payload, `project_to_lua` keys on the stored value's `TypeId`, so the typed projector takes over with no special "first-fire" path. The alternative — declaring with `TurnMeta::default()` etc. — would surface zero-elapsed / empty-message synthetic events that aren't real publishes.
- **`ConfirmChoice` projects as a stable short string, not the Rust enum** (`dd997df`) — `decision_label` returns `"yes" | "no" | "always_session" | "always_workspace" | "always_pattern_session" | "always_pattern_workspace" | "always_dir_session" | "always_dir_workspace" | "auto_allow"`. Plugins branch on the label rather than re-deriving the enum shape; pattern / dir payloads aren't included in the cell since the resolved decision doesn't need them (workspace store already has them) and the cell stays a flat 2-field table.
- **`confirm_requested` cell carries the full request snapshot** (`af62947`) — ARCHITECTURE.md's earlier `Cell<HandleId>` sketch becomes `Cell<ConfirmRequested>`. Lua reads tool / desc / args / options / approval patterns / outside dir / cwd label off the cell instead of a separate `_get(handle_id)` call, eliminating the only handle-keyed Rust primitive that was a pure data poll. `_render_title` and `_back_tab` stay handle-keyed for now because each reaches into Rust-only state (span-level bash highlight composition, mode toggle + auto-allow check); they collapse onto the cell payload only once those Rust internals grow Lua surfaces (P3.b / P4.f).
- **`is_clear()` over `is_empty()` for the engine-drain gate** (`af62947`) — same predicate, but the canonical name signals intent at call sites (the EngineBridge gate "is this clear?" reads more naturally than "is the pending map empty?"). Single rename now means a.11's gate hookup grafts onto the right name straight away.
- **mode_change / model_change autocmd events fold into `agent_mode` / `model` cells** (`7a2efec`) — both autocmds today emit `{ from, to }` at the same callsite where the cell publishes the new value. After the migration the cell is the only signal; subscribers wanting `from` track it locally. Net simpler than carrying a `{ prev, current }` payload struct just to mirror the old shape.
- **turn_end stays a separate cell from turn_complete** (`7a2efec`) — `turn_complete` only fires on natural end (carrying `TurnMeta`); `turn_end` fires on every termination (natural / cancel / error) with `{ cancelled }`. Predict.lua needs the cancel signal so it doesn't eat its own input on Esc; statusline / cost surfaces want the meta so they wait for `turn_complete`. Two distinct events, both as cells.
- **smelt.on retires entirely** (`7a2efec`) — kept as an alias was tempting but adds a stringly-typed event-name registry on top of cells. Migrating the two plugin callers (plan_mode, predict) is one-line each; no third-party plugins exist yet that we'd be breaking. `smelt.au.on` is the only event-subscribe surface.
- **Engine-drain gate uses `confirms.is_clear()`, not `blocks_agent`** (`2f8a1e8`) — the focused-overlay `blocks_agent` flag mixes "tool confirm pending" with "any modal UI absorbs input". Only the first is a real reason to pause engine streaming — the engine doesn't care about UI dialogs, only about tool-permission round-trips. `Confirms::pending` already tracks exactly the tool-confirm set, so `is_clear()` is the precise predicate. Pure-UI overlays (Permissions, Resume, /ps, /agents, /help, /btw) keep streaming under them — the hidden regression today's `blocks_agent` gate caused.
- **EngineBridge field stays named `engine` for now** (`2f8a1e8`) — ARCHITECTURE.md lists `Core { engine_bridge: EngineBridge }` as the target field name, but pre-Core-split today it lives on App alongside `clipboard` / `session` / `cells` / `lua` / `tools` (no `_subsystem` suffix anywhere). Renaming to `engine_bridge` now would churn ~40 callsites for zero structural benefit; the rename rides the `Core` aggregate (P2.a.12) where every subsystem field renames at once.
- **a.12 splits into a.12a / a.12b** (`5df837c`) — the subsystem aggregation (~350 callsite renames) and the frontend split (App → `TuiApp` + new `HeadlessApp`) are each natural single-session units. a.12a wraps the 8 subsystems in `Core` while `App` still owns the rest of today's struct surface; a.12b carves `HeadlessApp` over today's headless coordinator, renames `App` → `TuiApp`, and updates the dispatch in `src/main.rs`.
- **`Core` lives behind `App.core: Core` for now, not the inverse** (`5df837c`) — the target shape (`TuiApp { core: Core, … }`) is one rename of `App` away (a.12b), but switching the wrapping direction this session would force every UI-only field on today's App to re-home in the same commit. Wrapping with `App.core: Core` instead means the 8 subsystem fields move (and their callsites flip) without touching the UI fields, splitting the work along the natural seam.
- **a.10 (ToolRuntime carve-out) deferred** (`5df837c`) — REFACTOR's "may defer (depends on P5.a's trait shape)" applies. Lifting `LuaShared.plugin_tools` into a `ToolRuntime` struct without the `engine::ToolDispatcher` trait it impls would split the carve-out from its only consumer-facing API; the data lift waits until P5.a defines the dispatcher trait. `Core.tools: ToolRuntime` slot stays vacant until then.
- **`pub` vs `pub(crate)` on Core fields mirrors the prior App layout** (`5df837c`) — `Confirms`, `Timers`, `Cells`, `EngineBridge` are `pub(crate)` types; their Core fields are `pub(crate)` to match (compiler "more private than the item" rule). `AppConfig`, `Session`, `ui::Clipboard`, `LuaRuntime` are `pub` and their Core fields stay `pub`. Promoting everything to `pub` or demoting to `pub(crate)` would touch types outside Core; matching the existing visibility per type keeps the change scoped.
- **a.12b splits into a.12b1 / a.12b2** (`d94a3e0`) — REFACTOR's a.12b is two natural session-sized units: the HeadlessApp carve (~600 LOC moved + new `Core::new` + sink encapsulation + `src/main.rs` branch flip) and the `App` → `TuiApp` rename (~300+ callsites). Pre-splitting because each is a self-contained move; landing one and exiting matches the "one sub-phase per session, ish" rule better than packing both into one commit.
- **`Core::new(config, engine)` is the only Core construction path** (`d94a3e0`) — both `App::new` (TUI) and `HeadlessApp::new` (headless / subagent) call it. The cells `BuiltinSeeds` build pulls `cwd` / `mode` / `model` / `reasoning` from the passed-in `AppConfig` so headless and TUI start with the same cell shape. Keeping the seed assembly inside `Core::new` (rather than pushing it onto callers) means the eight subsystem fields are a closed unit — no construction-order subtleties leak past the call site.
- **Headless skips saved-state seeding** (`d94a3e0`) — `App::new` reads `state::State::load()` to seed `mode` / `reasoning_effort` / `accent_color` from the user's last interactive session; `HeadlessApp::new` does *not*. Headless / subagent invocations must be predictable from the CLI alone — bringing in `~/.local/state/smelt/state.toml` would surprise scripted callers. `mode_override` (CLI / config) wins; otherwise `Mode::Normal` is the explicit default. Reasoning is whatever main.rs computed.
- **`HeadlessSink::new_subagent` is JSON + verbose, no flags** (`d94a3e0`) — subagents always emit every engine event as a JSON line for the parent to parse, never branch on `format` / `verbose`. Modeling subagent as "JSON + verbose=true" rather than a separate `SubagentSink` type means one sink struct, one set of log helpers, and the variant lives at construction not at runtime; the few subagent-specific differences (always `emit_json`, no token-summary print) live in `run_subagent` itself, not in the sink.
- **`build_headless_config` lives in `src/main.rs`, not in `tui::app`** (`d94a3e0`) — the helper is one caller's assembly, mirroring how `App::new` already takes its 17 args from main. Promoting it to `tui::app::AppConfig::from_resolved_startup(...)` would force `tui` to depend on `tui::config::ResolvedModel` and `tui::state::ResolvedSettings` at the construction-API surface, when they're already inputs main owns. Keeps `tui::app::AppConfig` a plain data struct.
- **Color resolution stays a process-wide static** (`d94a3e0`) — `COLOR_OVERRIDE: OnceLock<Option<bool>>` and the `stderr_supports_color` cache live at module scope, set once via `init_color_mode`. Threading them through `&self` on `HeadlessSink` is possible but pointless — there's only ever one sink per process, and `stderr_supports_color` honours `NO_COLOR` / `TERM=dumb` / `FORCE_COLOR` / TTY-detection in addition to the `--color` flag, none of which is per-sink state.
- **P2.b splits into b.1 / b.2 / b.3 / b.4 / b.5** (`2204c0d`) — REFACTOR's P2.b body is too big for one session. Five natural single-session units: (b.1) `Host` trait in tui; (b.2) `UiHost` trait in ui; (b.3) supporting `ui::Event` / `Status` / `WinEvent` / `FocusTarget` types; (b.4) `Window::handle_key` + `handle_mouse` collapse into `Window::handle(Event, ctx, &mut dyn UiHost) -> Status` plus the P1.f.6b mouse fold; (b.5) Lua `with_host` / `with_ui_host` TLS pointer split + headless UiHost-binding errors. Splitting matches the "session lands one or more consecutive sub-phases" rule.
- **Host is `pub(crate)`, not `pub`** (`2204c0d`) — `pub trait Host` would emit `more_private_than_public_item` warnings because `Confirms` / `EngineBridge` / `Cells` / `Timers` are `pub(crate)` types. Promoting return types to `pub` is wrong (they're internal subsystems with no external consumers). The trait stays internal to the `tui` crate; that's where every consumer lives. If a future tier wants Host across crates, the return types promote together.
- **Skip `tools()` until P2.a.10 lands** (`2204c0d`) — ARCHITECTURE.md lists `tools(&mut self) -> &mut ToolRuntime` on `Host`, but `Core.tools` is vacant pending P5.a's `engine::ToolDispatcher` trait shape. Adding a placeholder `tools()` would force a `ToolRuntime` stub or `Option<ToolRuntime>` in Core that no caller exercises; better to grow the trait when the consumer lands. The other seven methods are concrete today.
- **First-consumer migration covers every method** (`2204c0d`) — landing the trait without consumers triggers `dead_code`. Tests-only consumption (a generic `_touch_all<H: Host>` helper) doesn't clear the lib-target lint. Migrating one production callsite per method (~7 sites total) is the smallest move that proves the seam compiles end-to-end and keeps `-D warnings` green.
- **Frontend impls delegate, not re-aggregate** (`2204c0d`) — `impl Host for TuiApp` / `for HeadlessApp` is mechanical: every method is `self.core.X()`. The trait impl on `Core` returns `&mut self.X` directly; the frontends inherit through their inner `Core`. This avoids duplicating field access logic and means a future Host method addition only needs to extend Core's impl + the trait — frontend impls follow mechanically.
- **`UiHost` is `pub`, not `pub(crate)`** (`5099ffb`) — sibling to `Host`'s `pub(crate)`, but the trait lives in the `ui` crate and its return types (`Ui`, `Buffer`, `Window`, `Overlay`, `BufId`, `WinId`, `OverlayId`) are all `pub`. A `pub(crate)` `UiHost` wouldn't be reachable from `tui`'s impl; promoting to `pub` is the only choice that compiles. No `more_private_than_public_item` warning because every return type is already `pub`.
- **`set_focus` over architecture-sketch `focus(&mut self, win)`** (`5099ffb`) — ARCHITECTURE.md sketches the trait with `fn focus(&mut self, win: WinId)` as a setter. `Ui` already publishes `focus() -> Option<WinId>` as the read accessor and `set_focus(win) -> bool` as the write; matching that split keeps the trait method names aligned with `Ui`'s inherent surface. The README rule "diagram is canonical for structure, ARCHITECTURE for intent" applies — the sketch is intent, the trait spelling matches `Ui`.
- **`Ui` impls `UiHost` for tests + direct callers** (`5099ffb`) — parallel to `Core` impl `Host`. Body uses explicit `Ui::method(self, …)` path syntax to disambiguate from the same-named inherent method. Lets tests exercise compositor-touching code through `&mut dyn UiHost` without spinning up a `TuiApp`. The new `ui_host_dispatch_round_trips_through_dyn` test does exactly that.
- **No `Host` supertrait** (`5099ffb`) — REFACTOR.md mandates "no supertrait bound" because `ui` cannot reference tui-defined `Host`. Bindings that need both reach through TLS for each independently (`with_host` / `with_ui_host` in P2.b.5). The trait split is by *crate boundary*, not by feature surface — `ui::UiHost` is the slice that lives where `Ui` lives, `tui::Host` is the slice that lives where `Core` lives.
- **Skip `fire_win_event` / `overlay_close` / `ui()` migration this commit** (`5099ffb`) — every production `fire_win_event` callsite (events.rs, lua_bridge.rs) holds a borrow on `self.core.lua` for its closure that conflicts with the `&mut self`-shaped trait dispatch; the borrow split would require a real refactor (e.g. moving the queue into the trait method). `overlay_close` is only invoked from `&mut Ui` callback contexts (`text_modal.rs`) where method resolution already chooses `Ui`'s inherent path. `ui()` is a meta-accessor that pays off when bindings dispatch through the trait. The new ui-side test exercises all three through `&mut dyn UiHost` end-to-end so the trait shape is validated; production migrations follow naturally in P2.b.4 / P2.b.5 where the borrow patterns flatten.
- **`Status` collapses `DispatchOutcome` + `MouseAction` into one enum** (`c91b983`) — `Status::{ Consumed | Capture | Ignored }` is the union. Key dispatch never returns `Capture` (no in-flight key gesture concept); mouse dispatch never returns `Ignored` from the *Ui-level* pre-flight (it returns `Consumed` for absorbed wheel/modal-outside cases and falls through to host routing otherwise). Two semi-overlapping enums become one; the call-site discrimination is which method you called, not which type you matched on. `Window::handle_mouse` keeps its three-way return; the host folds `Capture` into `Ui::set_capture` exactly as before.
- **`From<crossterm::event::Event>` over a host-side translator fn** (`c91b983`) — the conversion is a 1-1 enum re-wrap; an `impl From` keeps the host call-site a single `.into()` and lets future code that wants to feed an already-typed crossterm event through the Ui dispatch path do so without reaching for a helper. The three tui callsites that exist today don't need it (they already construct the crossterm payload separately), so they go straight to `ui::Event::Key(_)` etc., but the conversion is there for the next consumer that walks an `EventStream`.
- **`FocusTarget` is a single-variant enum, not a type alias** (`c91b983`) — ARCHITECTURE.md sketches it as a future-proof distinction from `HitTarget`. A bare `WinId` *would* type-check today, but it conflates "id of the window with keyboard focus" and "id used as a hit-test result for clicks on a window". One variant is cheap; the future where `FocusTarget` grows (e.g. `OverlayChrome` for a modal title bar that absorbs Tab / Shift+Tab) is exactly the case the type was created for. No production consumer migrates yet — it lands as a public type so the consumers that need it next don't pay a renaming tax.
- **P2.b.4 splits into b.4a / b.4b / b.4c** (`62ef662`) — REFACTOR's P2.b.4 absorbs both the `Window::handle` signature collapse *and* the deferred P1.f.6b mouse fold; each is session-sized. Three natural single-session units: (b.4a) `handle_key` returns `ui::Status` so the two per-event methods share a return type ahead of the collapse; (b.4b) the unified `Window::handle(ui::Event, ctx, &mut dyn UiHost) -> Status` entry point with today's per-event methods as private helpers; (b.4c) the pre-P2 mouse routing fold from tui into `Ui::dispatch_event` reaching through `UiHost` (multi-session; further split in REFACTOR).
- **Kill-ring inspection moves to the caller, not behind a richer Status** (`62ef662`) — `Window::handle_key`'s old `Option<Option<String>>` shape leaked the transcript-display-rendering hack (raw → display re-render before clipboard write) into the Window primitive's signature. The cleanest collapse is `Status::{ Consumed | Ignored }` and the single production caller (`handle_content_vim_key`) reads `kill_ring.{current,source_range}` after dispatch. The clear (`set_with_linewise(empty)`) moves to the caller too — Window no longer touches kill-ring state, only vim does (through `clipboard` in its `VimContext`). Aligns with `handle_mouse`'s return shape (already `Status`) without inventing a yank-payload variant that would only ever be relevant on the transcript path.
- **P2.b.5 splits into b.5a / b.5b / b.5c** (`645a08b`) — full TLS-slot generalization (carrying either `TuiApp` or `HeadlessApp`) plus the bulk-migration of 92 `with_app` callsites is too big for one session, and HeadlessApp doesn't drive Lua bindings yet so the runtime-error path stays untested in production until P2.c surfaces a headless Lua callback driver. Three units: (b.5a) trait-typed dispatchers `with_host` / `with_ui_host` over today's TLS pointer plus a few first consumers so each new fn is reached from production; (b.5b) the slot-shape change (Tui-or-Headless) with UiHost from headless raising — lands alongside the first headless Lua driver; (b.5c) bulk migration plus dropping the `with_app` alias.
- **`with_host` / `try_with_host` are `pub(crate)`, `with_ui_host` / `try_with_ui_host` are `pub`** (`645a08b`) — sibling to the trait visibilities (`Host` is `pub(crate)`, `UiHost` is `pub`). A `pub fn with_host(&mut dyn Host)` would warn `more_private_than_public_item`; matching the trait visibility per-fn keeps `-D warnings` green without promoting the trait.
- **P2.b.4c splits into c.1…c.6** (`ce0d3c4`) — REFACTOR's "multi-session, further split" resolves outside-in: drop redundant viewport mirrors first (c.1), then click-count tracking (c.2), scrollbar drag (c.3), wheel scroll routing (c.4), per-pane Down/Drag/Up onto `Window::handle` (c.5 = the b.4b collapse), drag autoscroll (c.6). Each is one session; the host's TuiApp-side mouse code peels away in layers rather than collapsing in one big move.
- **`Window::viewport` is canonical, not the TuiApp shadows** (`ce0d3c4`) — render-loop already writes `win.viewport` on the same WinId immediately after computing the geometry; the parallel `TuiApp.{prompt,transcript}_viewport` fields are pure cache. Routing reads through `Ui::win(id)?.viewport` makes Window the single source of truth for last-painted geometry, which is what mouse code actually wants. `WindowViewport: Copy` makes the lookup-then-deref ergonomic — no borrow lifetime extends across `&mut self` mouse handlers.
- **Scrollbar capture-aware routing in `Ui::dispatch_event` keeps host-side mirror state via post-dispatch read-back** (`444ddb2`) — the cleanest split for c.3 was: Ui owns the gesture (Down latches capture and snaps `Ui::wins[owner].scroll_top`; Drag re-snaps; Up releases), host reads back the new `scroll_top` from `Ui::win(owner)` and propagates to its parallel `transcript_window.scroll_top` / `input.win.scroll_top` plus the transcript-specific re-anchor + follow-tail recompute. The alternative — passing a host-side callback through `dispatch_event` — would have grown the closure surface without buying anything (the math is pure, the side effects are bounded to two well-known WinIds, the propagate helper stays a single host-side fn). The post-dispatch read-back stays cheap because `Ui::win` is a `HashMap::get` and `WindowViewport: Copy`.
- **`Ui::hit_test` returns `Scrollbar` for splits leaves now** (`444ddb2`) — the doc-comment's "never returns Scrollbar yet" carve-out lifts. The split-leaf branch checks the painted viewport's scrollbar geometry first (`bar.contains(vp.rect, row, col)`) and only falls through to `Window` when the cell is content. Overlay leaves still return `Window` even on their painted scrollbars — no overlay scrollbar drag exists in production today; that extension waits for a real consumer.
- **The `begin_scrollbar_drag_if_hit(_, _, owner)` two-call dance retires** (`444ddb2`) — the host's two Down branches (prompt-viewport-then-transcript) each called the helper with a hard-coded `owner: WinId`. After folding the gesture into Ui, the discriminator becomes `Ui::hit_test(row, col, None) == HitTarget::Scrollbar { owner }`, which is *the* canonical query — no parallel "is this point on owner's bar?" callsite.
- **Wheel coalescer tracks the column too** (`130fb13`) — the existing `absorb` closure folded `ScrollUp`/`ScrollDown` events into `(scroll_delta, scroll_row)` and discarded the column because `LayoutState::hit_test(row, 0)` only needed the row. After switching to `Ui::hit_test(row, col, None)` the column is significant — overlays and chrome straddling the same row resolve differently from a content cell. Coalescing onto `(scroll_delta, scroll_row, scroll_col)` keeps the last wheel event's column as the apply-site coordinate; that matches today's behaviour where the *most recent* wheel event's pointer position drives the routing decision.
- **Routing the wheel through `Ui::hit_test` over `LayoutState::hit_test` for c.4** (`130fb13`) — the parallel `LayoutState::hit_test` is a row-strip discriminator that knows only transcript / prompt / status / outside; `Ui::hit_test` is the canonical splits + overlays + scrollbar query. Switching means a non-focused overlay covering part of the transcript will resolve as `Window(overlay_leaf)` and fall through to the "content scroll" default — same outcome as today (transcript scrolls), one fewer parallel routing surface. `LayoutState::hit_test` keeps its remaining callers (the click-on-status early-out, in-content snap math) until c.5 retires them.
- **c.5 splits into c.5a / c.5b / c.5c** (`636ef5e`) — REFACTOR's "(b.4b)" parenthetical conflated three distinct moves: (c.5a) Window-capture + `Ui::resolve_split_mouse` so Down/Drag/Up share the scrollbar-style capture pattern; (c.5b) host accessors on `UiHost` (`rows_for_window` / `viewport_for(WinId)`) so per-pane row + viewport + break data flow through the trait; (c.5c) the unified `Window::handle(Event, ctx, host)` entry consuming the accessors — this is b.4b proper. Splitting matches "session lands one or more consecutive sub-phases"; c.5a lands today, c.5b/c.5c next sessions.
- **`Ui::resolve_split_mouse` over a `mouse_invoke` closure callback** (`636ef5e`) — a `dispatch_event(ev, lua_invoke, mouse_invoke)` callback variant would have forced the host's per-pane router into a closure capturing `&mut self`, conflicting with the `&mut self.ui` borrow held during dispatch. Returning `Option<(WinId, u8)>` from a separate method instead lets the host call `Ui::dispatch_event(...)` first (capture/scrollbar/wheel/modal absorb) and then `Ui::resolve_split_mouse(me)` after — no closure borrow gymnastics, and the captured-Window state lives entirely on `Ui::capture` (same slot scrollbar drag uses, never both at once because Down latches one or the other).
- **Window capture replaces `mouse_drag_active`-as-a-flag for routing** (`636ef5e`) — `mouse_drag_active` survives this sub-phase only as a tick_drag_autoscroll input (c.6's territory). The "is this drag still attached to its origin pane?" question that used to flow through `dispatch_focused_mouse(app_focus)` now flows through `Ui::capture == HitTarget::Window(win)` — the same answer, but durable across pointer drift and read directly from the capture slot rather than reconstructed from a focus enum.
