# P2 — TUI App restructure

**Status:** in-progress **Started:** 2026-04-30 **Landed:** —

P2 splits today's god-struct `App` into a headless-safe `Core` plus
`TuiApp` / `HeadlessApp` frontends, carves the subsystems out as
their own types (Session, Confirms, Clipboard, Timers, Cells,
LuaRuntime, ToolRuntime, EngineBridge, AppConfig, WellKnown), installs
the `Host` + `UiHost` traits (absorbing P1.e's trait + supporting
types), and collapses the event surface onto a single `select!` loop
publishing through `Cells`. Sub-phases land independently, each green
at the boundary. Detail per sub-phase lives in the commit message;
this file is an index.

## Sub-phases landed

- **P2.a.1** (`b07f803`) — `WellKnown { transcript, prompt, statusline }` carves the well-known `WinId`s off App; `Ui::win_buf{,_id,_mut}` resolves `WinId` → backing `Buffer`.
- **P2.a.2** (`03bd06e`) — `WellKnown.cmdline: Option<WinId>` absorbs today's `App::cmdline_win`.
- **P2.a.3a** (`b75a599`) — `Confirms` data carve-out: `pending` map + `next_handle` counter move off App into a typed subsystem with `register / get / take`; `ConfirmEntry` relocates from `dialogs/confirm.rs` to `app/confirms.rs`.
- **P2.a.6** (`a8cbe34`) — `AppConfig` data carve-out: 16 fields (`model` / `api_base` / `api_key_env` / `provider_type` / `available_models` / `model_config` / `cli_*override` / `mode` / `mode_cycle` / `reasoning_effort` / `reasoning_cycle` / `settings` / `multi_agent` / `context_window`) lift off `App` into `app::app_config::AppConfig`; call sites read through `self.config.*`.
- **P2.a.7a** (`db9fecc`) — Session snapshots data lift: drop App-side `context_tokens` / `token_snapshots` / `cost_snapshots` / `turn_metas` / `session_cost_usd` shadows; reads/writes route through `self.session.*`; `save_snapshots_to_session` / `restore_snapshots_from_session` / `clear_snapshots` / `truncate_snapshots_to` retire.
- **P2.a.7b** (`8a686ea`) — `App.history: Vec<Message>` collapses into `self.session.messages`; `set_history` writes the session field directly; `sync_session_snapshot` drops the `messages = self.history.clone()` line; ~25 call sites across `app/{mod,agent,history,cmdline,commands,lua_handlers}.rs` + `lua/api/state.rs` reroute through `self.session.messages.*`. Session is now the single source of truth for in-session message history.
- **P2.a.5** (`90df0fa`) — `Timers` carve-out: `LuaShared.timers` storage + `LuaRuntime::tick_timers` retire; `app::timers::Timers` on `App` holds the queue with `set` / `every` / `cancel` + `drain_due`; `App::tick_timers` borrows `timers` + `lua` disjointly. Lua bindings: `smelt.timer.{set,every,cancel}` (returning cancellable ids) plus `smelt.defer` kept as a thin alias over `Timers::set`. Recurring timers re-arm in place via `retain_mut`; the callback list materialises before `&mut Timers` releases so a fire that re-enters `set` / `every` / `cancel` composes cleanly with `with_app`.
- **P2.a.8** (`2db595a`) — Clipboard formalised: every clipboard text I/O routes through `app.clipboard.{read,write}`. Lua `smelt.clipboard(text)` writes via `with_app(|app| app.clipboard.write(...))` (matches `smelt.notify`); `KeyAction::ClipboardImage`'s text fallback reads via `clipboard.read()`. `commands::{copy_to_clipboard, paste_from_clipboard}` drop `pub(crate)` — `SystemSink` is their only caller now, and vim / emacs / transcript yank / prompt paste / Lua all flow through the App-level subsystem.

## Decisions made

- **WellKnown shape** (`b07f803`) — three stable WinIds (prompt /
  transcript / statusline) only; cmdline (currently
  `App::cmdline_win: Option<WinId>`) folds in next sub-phase. The
  redundant `input_display_buf` / `transcript_display_buf` /
  `status_buf` `BufId` fields retire entirely — buffers are reached
  through the new `Ui::win_buf{,_id,_mut}` helpers, matching the
  ARCHITECTURE.md target where `WellKnown` carries only WinIds.
- **P2.a sub-phase split** (`b07f803`) — P2.a is too large for one
  session; split into `a.1` … `a.12` with independent landings.
  Order is data-only carve-outs first (WellKnown, Confirms,
  AppConfig, Session), then new reactive primitives (Cells, Timers),
  then bridges (LuaRuntime/Cells fold, ToolRuntime, EngineBridge),
  then the Core / TuiApp / HeadlessApp aggregate.
- **P2.a.3 split into a.3a / a.3b** (`b75a599`) — the data carve-out
  (struct + register/get/take helpers) is mechanical; the
  oneshot::Sender swap needs `Cells` for the request-payload route.
  a.3a lands the carve-out; a.3b lands the channel rewrite +
  `is_clear()` gate alongside `EngineBridge` once a.4 is in. No
  consumers exist for `is_clear()` until the bridge fold, so adding
  it now would be premature.
- **AppConfig holds runtime-mutable fields too** (`a8cbe34`) — `mode`,
  `reasoning_effort`, `model`, `model_config` flip during a session
  (Shift+Tab cycles `mode`; `/model` rewrites `model`; pricing
  resolves into `model_config`). They live on `AppConfig` anyway
  because the eventual `Cells` migration (P2.a.4) reads them as cell
  setters' source of truth — segregating "immutable startup" from
  "runtime-mutated" inside this struct would just split one logical
  unit. `startup_auth_error` stays on `App` (one-shot transient,
  drained on first frame).
- **Data-only carve-outs precede a.4 (Cells)** (`a8cbe34`) — REFACTOR's
  P2.a sub-phase listing is numerical (a.4 → a.5 → a.6 …), but the
  text body says "data-only carve-outs first, then the new reactive
  primitives." a.6 / a.7 / a.8 are pure data lifts and can land
  before the Cells primitive without touching its design. Pick the
  data lifts now so a.4's migration step (built-in cells fed by
  these fields) reads against a clean target shape.
- **P2.a.7 split into a.7a / a.7b** (`db9fecc`) — `Session` already
  owns 4 of the 5 snapshot fields and `App.history` mirrors
  `session.messages`; lifting both is one logical "Session SoT"
  move but ~80+ rename sites is too much for one session. a.7a
  (snapshots: context_tokens / token_snapshots / cost_snapshots /
  turn_metas / session_cost_usd) lands now — a small, contained
  carve-out that deletes the sync helpers entirely. a.7b
  (history → session.messages) lands next as a mechanical sweep
  across transcript / agent / history modules.
- **`Session.session_cost_usd` is persisted** (`db9fecc`) — added
  as a `#[serde(default)]` field rather than computing it on demand
  from `cost_snapshots.last()`. Old session files load with `0.0`
  and the `restore_snapshots_from_session` defensive scrub recomputes
  the running cost from the last cost snapshot, so the field is
  effectively cached but the load path stays correct without a
  schema migration.
- **Timers lands before Cells** (`90df0fa`) — REFACTOR's numerical
  ordering puts a.4 (Cells) before a.5 (Timers), but Timers is fully
  independent of Cells: it's a storage move from `LuaShared` onto
  `App` with two small ergonomic adds (`every` + `cancel`). Landing
  it now means the `now` / `spinner_frame` cell drivers are ready
  the moment Cells lands, matching ARCHITECTURE.md's "Timers feeds
  these built-ins" model. Going Cells-first would have left the
  built-in cells without their drivers until Timers caught up.
- **a.8 fits the data-only carve-out tier** (`2db595a`) — REFACTOR
  framed a.8 as "formalise Clipboard if any wiring is still loose."
  The two real bypasses were the Lua `smelt.clipboard` binding
  (called `commands::copy_to_clipboard` directly) and
  `KeyAction::ClipboardImage`'s text fallback (called
  `commands::paste_from_clipboard` directly). Routing both through
  `app.clipboard.{write,read}` is mechanical, doesn't touch Cells,
  and lets the freestanding helpers go file-private — `SystemSink`
  becomes the single seam between the App-level subsystem and the
  platform subprocesses. Landing it now finishes the data-only
  carve-out tier ahead of a.4 (Cells), matching the same "data
  lifts first, reactive primitives next" ordering a.6/a.7 followed.
