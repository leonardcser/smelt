# P5 — Tools to Lua

**Status:** in-progress **Started:** 2026-05-01 **Landed:** P5.a, P5.b, P5.c, P5.d, P5.e

P5 moves the 15 Rust tool implementations in `engine/tools/` into
`runtime/lua/smelt/tools/`. Engine becomes schema + dispatcher only.
Mode gating becomes a Lua-tool `hooks` concern. The `engine/permissions/`
module pulls out to `tui::permissions`; engine ends policy-free.

Sub-phases:

- **P5.a** — Tool dispatcher trait shape. Engine takes
  `Box<dyn ToolDispatcher>` at startup. `ToolSchema` slims; the
  `Tool` trait drops `needs_confirm` / `preflight` /
  `approval_patterns` / `modes` / `decide_override`.
- **P5.b** — Migrate core tools to Lua. 15 tools relocate to
  `runtime/lua/smelt/tools/`; intricate logic stays in Rust as
  `tui::*` capabilities the Lua tools FFI into. ✅ Completed.
- **P5.c** — Engine cleanup. `engine/permissions/` → `tui::permissions`.
  Drop the multi-agent concept (`Role::Agent`, `EngineEvent::AgentMessage`,
  etc.). `EngineHandle` becomes channels-only. Drop
  `EngineConfig.interactive`. ✅ Completed.
- **P5.d** — Drop `config.yaml`; all config in `init.lua`. Drop
  `serde_yml` dep + the YAML loader. ✅ Completed.
- **P5.e** — Protocol rename pass. `Mode → AgentMode`,
  `ExecutePluginTool → ToolDispatch`, etc. Drop the `Plugin` prefix
  from tool defs (one registry now). ✅ Completed.

## Decisions made

- **P5.b formally done** (`<this session>`) — all 15 core tools now live in
  `runtime/lua/smelt/tools/`; `crates/engine/src/tools/mod.rs` is strictly
  trait + registry surface with zero tool implementations. No remaining Rust
  tool impls in engine.
- **P5.d — tui-side `serde_yml` removal + Lua-driven config** (`<this session>`) —
  stripped `Deserialize` / `serde_yml` from `tui::config` types (`Config`,
  `ModelConfig`, `ProviderConfig`, etc.); replaced YAML test harness with
  Lua `init.lua` generation in `tests/common/harness.rs`; added
  `smelt.provider.register`, `smelt.mcp.register`, `smelt.permissions.set_rules`,
  `smelt.settings.set` Lua APIs that populate `LuaShared` registries;
  `LuaRuntime::load_user_config()` executes `init.lua` before engine start
  and `Config::from_lua_shared()` builds the Rust `Config` from those
  registries; `main.rs` startup sequence loads Lua config before calling
  `startup::resolve`. Setup wizard (`run_initial_setup`) writes a seed
  `init.lua`; `smelt auth` for non-OAuth prints the Lua block for manual
  paste (never edits user `init.lua`). `--config` repurposed to point to
  a custom `init.lua` path. `serde_yml` removed from both `engine/Cargo.toml`
  and `tui/Cargo.toml`.

- **Drop dead `Tool` trait methods** (`9c356d8`) — `interactive_only` /
  `modes` / `decide_override` retire; `definitions(interactive: bool)`
  arg drops.
- **Lift `Tool::is_mcp` to a `ToolEntry` data flag** (`030aecd`) — set
  at registration via `register` / `register_mcp`.
- **Collapse `needs_confirm` / `approval_patterns` / `preflight` into
  `Tool::evaluate_hooks(&args) -> protocol::PluginToolHooks`**
  (`a26a42e`) — three trait methods fold into one returning the
  shape plugin tools already use.
- **Extract `Turn::resolve_decision` shared by core+plugin paths**
  (`c27dca1`) — the `permissions.decide → runtime-approvals upgrade`
  block was duplicated; folded into one associated fn.
- **Introduce `ToolDispatcher` trait + route engine through it**
  (`db2ed43`) — engine never touches `&dyn Tool` directly; every
  per-call decision goes through `definitions / contains / is_mcp /
  evaluate_hooks / dispatch`.
- **Drop `&Permissions, Mode` from `ToolDispatcher::definitions`**
  (`48e742d`) — dispatcher returns the unfiltered list; engine
  applies mode-permission filtering externally via `is_mcp`.
- **P5.e protocol rename pass** (`2320338`) — `Mode → AgentMode`,
  `SetMode → SetAgentMode`, `ExecutePluginTool → ToolDispatch`,
  `EvaluatePluginToolHooks → ToolHooksRequest`,
  `PluginTool{Result,HooksResult} → Tool{Result,HooksResponse}`,
  `Plugin` prefix drops from tool def types (one registry now).
- **P5.e internal identifier sweep** (this session) — `plugin_tools` → `tools`,
  `PluginToolEnv → ToolEnv`, `PluginToolHandles → ToolHandles`,
  `evaluate_plugin_hooks → evaluate_hooks`, `build_plugin_ctx → build_tool_ctx`,
  `pending_plugins → pending_tools`, `sequential_plugins → sequential_tools`,
  `pending_plugin_hooks → pending_tool_hooks`, `pending_plugin_perms → pending_tool_perms`,
  `PendingPluginCall → PendingToolCall`, `plugin_results → tool_results`,
  `wait_for_plugin_result → wait_for_tool_result`, and all "plugin tool" comments
  updated to "tool". No `Plugin` prefix remains in the tool/hook/dispatch path.
- **Remove `summary` from `EngineEvent::ToolStarted`** — engine no longer
  pre-computes a UI-facing summary string. Summary ownership moves to
  Lua-tool metadata: `smelt.tools.register({ summary = function(args) ... end })`
  defines the one-line transcript / headless label, and `_bootstrap.lua`
  installs a Lua-side default summarizer for tools that omit it. Adding a
  new Lua tool never requires a Rust change for its summary.
- **Keep tool presentation metadata on the Lua registration** — labels like
  the one-line summary live beside the tool schema / hooks in
  `smelt.tools.register(...)`, not in Rust-side name switches. Rust consumes
  that metadata generically; the default product presentation remains plugin-owned.
- **Move `engine/tools/{file_state,notebook,result_dedup,web_cache}.rs`
  out of `tools/`** — `file_state` → `tui::fs`; `notebook` →
  `tui::notebook`; `result_dedup` → `engine::result_dedup`;
  `web_cache` inlined into `engine::pricing`. `engine/tools/mod.rs`
  shrinks to the tool surface only (`ToolResult`, `ToolContext`,
  `ToolFuture`, `Tool`, `ToolEntry`, `ToolDispatcher`, `ToolRegistry`).
- **Extract `trim_tool_output` + `MAX_TOOL_OUTPUT_LINES` to
  `engine::trim`** — provider serializers consume it; not a tool concern.
- **Move `try_flock` / `FlockGuard` to `tui::fs`** — filesystem primitive,
  consumed by Lua tools via `smelt.fs.try_flock`.

### P5.b Lua tool migrations

- **`glob` → Lua** (`9be6f95`) — composes new `tui::fs::glob`
  capability via `smelt.fs.glob`. `tools/` directory introduced.
- **`grep` → Lua** (`72fc354`) — composes `smelt.grep.run` (ripgrep)
  with `smelt.process.run("grep", ...)` fallback.
- **`load_skill` → Lua** (`5f5e841`) — composes new `smelt.skills.content`
  over a shared `Arc<SkillLoader>` clone parked on `Core.skills`.
- **`list_agents` → Lua** (`48536df`) — composes new
  `smelt.agent.{discover,my_pid}` over `engine::registry` (since removed).
- **`stop_agent` → Lua** (`53909e3`) — composes new
  `smelt.agent.{find_by_id,is_in_tree,kill,my_pid}`. Owner-only check
  preserved.
- **`ask_user_question` relocated `plugins/` → `tools/`** (`2954db1`) —
  embedded module path renamed to `smelt.tools.ask_user_question`.
- **`peek_agent` + `message_agent` → Lua** (`1d977a2`) — composes new
  `smelt.agent.{send_query,send_message,my_id,my_slug}` over
  `engine::socket::send_{query,message}_blocking` blocking wrappers
  (since removed).
- **`web_search` → Lua + `tui::http` POST/cache/UA** (`17ba19e`) —
  composes new `smelt.http.{post,cache.{get,put},random_user_agent}`
  + `smelt.html.parse_ddg_results`. `tui::http::cache` relocates from
  `engine/tools/web_cache.rs` (both implementations read/write the
  same `<cache_dir>/web/<hash>` files until `web_fetch` retires).
- **`FileStateCache` plumbed through `engine::start` + parked on Core**
  (`711492d`) — lifts `FileStateCache` / `FileState` /
  `file_mtime_ms` / `staleness_error` from `pub(crate)` to `pub`;
  `engine::start(config, files)` takes the cache as a param so engine
  + Lua see the same observations. `smelt.fs.file_state.*` exposes
  the cache to Lua.
- **`write_file` → Lua** (`25e31eb`) — composes
  `smelt.fs.{exists,mkdir_all,write,try_flock,file_state.*}` +
  `smelt.notebook.is_notebook_path` + new `smelt.path.display`. Lifts
  `try_flock` / `FlockGuard` / `display_path` to `pub`. Lua-side
  flock guard is a userdata wrapper with `:release()` + `__gc`.
- **`edit_file` → Lua** (`da42a63`) — composes
  `smelt.fs.{read,write,try_flock,file_state.*}`; pure-Lua
  `replace_first` / `replace_all` over `string.find(..., 1, true)`.
- **`read_file` → Lua** (`47cc6d3`) — composes new
  `smelt.image.{is_image_file,read_as_data_url}` (new `lua/api/image.rs`)
  + new `smelt.notebook.read` (over public
  `engine::tools::notebook::render_notebook_text` wrapper) +
  `smelt.fs.{read,file_state.*}`. Pure-Lua `dedup_stub` /
  `format_text_window` reproduce the engine semantics. Drops dead
  `tools::int_arg` and `notebook::is_notebook` helpers.
- **`edit_notebook` → Lua** (`991b070`) —
  `runtime/lua/smelt/tools/notebook_edit.lua` thin wrapper over a new
  `smelt.notebook.apply_edit(args)` FFI (over a new `pub`
  `engine::tools::notebook::apply_edit` returning
  `NotebookEditOutcome { message, metadata }`). Intricate JSON cell
  munging stays Rust; Lua owns schema, hook, flock, metadata
  passthrough via a new `super::json_to_lua_value` helper. Drops
  `ToolContext.file_locks` + `engine::tools::FileLocks` (last
  consumer gone — flock alone serializes concurrent edits).
- **`exit_plan_mode` extracted to `tools/exit_plan_mode.lua`** (this session) —
  the Plan-mode tool body (schema, plan-name generator, `save_plan`,
  confirm dialog) relocates verbatim from `plugins/plan_mode.lua` into
  a `register / unregister` module the plugin requires. Plan-mode hook
  driver shrinks to ~60 LOC.
- **`bash` → Lua + `tui::process::run_streaming` + cross-thread
  Lua resume infrastructure** (this session) —
  `runtime/lua/smelt/tools/bash.lua` composes `smelt.shell.*` with
  the new `smelt.process.run_streaming(task_id, call_id, command,
  timeout_ms)` that spawns `sh -c command` on a tokio task, fires
  `EngineEvent::ToolOutput` per stdout/stderr line via the engine
  injector, and resumes the parked Lua coroutine on exit with a JSON
  `{ content, is_error, timed_out }` payload. The cross-thread resume
  goes through a new `LuaResumeSink` (a `Send`-safe handle holding a
  `Vec<(u64, serde_json::Value)>` JSON inbox + an `mpsc` wakeup
  sender); the main loop drains the JSON inbox into the runtime's
  `task_inbox` (as a new `TaskEvent::ExternalResolvedJson` variant)
  and `pump_task_events` converts JSON → `mlua::Value`. New main-loop
  `select!` arm on `lua_wakeup_rx` ensures parked coroutines resume
  promptly when nothing else is happening. `engine/tools/bash.rs`
  deleted; `check_interactive` /
  `check_shell_background_operator` relocated inline into
  `lua/api/shell.rs` (last consumer); `DEFAULT_BASH_ALLOW` lifted to
  `pub` and exposed via `smelt.shell.is_default_bash_allow`.
  `ToolContext` shed `event_tx` + `call_id` (no remaining Tool impl
  consumes them after BashTool's removal). Note: plugin tools still
  lack cancel propagation (engine-level cancel doesn't reach the
  tokio task driving `run_streaming`); same regression as every
  other migrated tool, future P6 work.
- **`engine/tools/background.rs` → `tui::process`** (this session) —
  P3.a / P5.c capability fold. `ProcessRegistry` / `ProcessInfo` /
  `Process` (228 LOC) move from engine into `tui::process` alongside
  the existing `run` / `run_streaming` foreground primitives.
  `kill_group_sigkill` (SIGKILL — for explicit `stop_process`) sits
  next to the existing `kill_process_group` (SIGTERM — used by
  `run_streaming` timeout) so the registry's stop semantics survive
  the move. Engine no longer references background-process state in
  any form. `engine::tools::ProcessRegistry` import in `Core` rewires
  to `crate::process::ProcessRegistry`.
- **`EngineHandle.runtime_approvals` field retired** (this session) —
  P5.c step toward "EngineHandle becomes channels-only": drop the
  `Arc<RwLock<RuntimeApprovals>>` field and the public
  `runtime_approvals()` accessor. `TuiApp::new` now takes the
  original `Arc` directly from main.rs (which already constructed
  it). Engine's internal use stays via `EngineConfig.runtime_approvals`
  until the full policy-free move.
- **`EngineHandle.processes` field retired → `Core.processes`** (last session) —
  P5.c step toward "EngineHandle becomes channels-only": engine no
  longer constructs `ProcessRegistry` (no engine-side consumer
  remains after `bash`'s migration to Lua); `Core::new` builds it
  directly. `EngineBridge::processes()` accessor + the `_processes`
  arg on `tools::build_tools` retire; the 7 `app.core.engine.processes()`
  / `self.core.engine.processes()` callsites in
  `lua/api/{statusline,process}.rs` + `app/history.rs` rewrite to
  `app.core.processes` / `self.core.processes`. `runtime_approvals` /
  `EngineConfig.permissions` / `runtime_approvals` dropped in P5.c.
  `EngineConfig.interactive` dropped (field no longer exists on `EngineConfig`).
- **`spawn_agent` → Lua** (transitional, since deleted) — was
  `runtime/lua/smelt/tools/spawn_agent.lua` composing
  `smelt.agent.{spawn,wait_for_message,subagent_meta,list,my_pid}` over
  `EngineHandle::spawn_subagent`. All engine-side multi-agent plumbing
  (`spawn_subagent`, `subagent_meta`, `AgentMessageNotification`,
  `engine/registry.rs`, `engine/socket.rs`) and the five agent tools
  deleted in P5.c. Target shape (if multi-agent returns) is an optional
  plugin over `smelt.subprocess` with no engine involvement.
- **`web_fetch` → Lua + `tui::html::to_markdown`** (`c1c9de9`) —
  `runtime/lua/smelt/tools/web_fetch.lua` composes
  `smelt.http.{get,cache.{get,put},random_user_agent}` +
  `smelt.html.{title,links,to_text,to_markdown}` +
  `smelt.image.data_url_from_bytes` + `smelt.engine.ask` (with
  `smelt.task.alloc/wait` parking the coroutine until
  `EngineAskResponse` fires). The full HTML→markdown converter
  (`ParsedHtml::to_markdown` + helpers) relocates from
  `engine/tools/web_shared.rs` into `tui::html` as
  `Markdown { title, content, links }`. The `smelt.http.get` body
  switches from lossy UTF-8 to a byte-safe Lua string so binary
  responses (image content-types) survive the boundary unchanged.
  LLM extraction routes through `AuxiliaryTask::Btw` — falls back
  to the primary model when no btw aux is configured (was the
  primary unconditionally before). `engine/tools/{web_fetch,
  web_shared}.rs` retire; `web_cache.rs` keeps its TTL setter for
  pricing's metadata cache. `ToolContext` drops the
  `provider / model / api` fields (web_fetch was their only
  consumer); `Provider::extract_web_content` retires.
