# P5 — Tools to Lua

**Status:** in-progress **Started:** 2026-05-01 **Landed:** —

P5 moves the 15 Rust tool implementations in `engine/tools/` into
`runtime/lua/smelt/tools/`. Engine becomes schema + dispatcher only.
Mode gating becomes a Lua-tool `hooks` concern. The `engine/permissions/`
module pulls out to `tui::permissions`; engine ends policy-free.

Sub-phases:

- **P5.a** — Tool dispatcher trait shape. Engine takes
  `Box<dyn ToolDispatcher>` at startup. `ToolSchema` slims; the
  `Tool` trait drops `needs_confirm` / `preflight` /
  `approval_patterns` / `modes` / `decide_override`.
- **P5.b** — Migrate core tools to Lua. 15 tools relocate to
  `runtime/lua/smelt/tools/`; intricate logic stays in Rust as
  `tui::*` capabilities the Lua tools FFI into.
- **P5.c** — Engine cleanup. `engine/permissions/` → `tui::permissions`.
  Drop the multi-agent concept (`Role::Agent`, `EngineEvent::AgentMessage`,
  etc.). `EngineHandle` becomes channels-only. Drop
  `EngineConfig.interactive`.
- **P5.d** — Drop `config.yaml`; all config in `init.lua`. Drop
  `serde_yml` dep + the YAML loader.
- **P5.e** — Protocol rename pass. `Mode → AgentMode`,
  `ExecutePluginTool → ToolDispatch`, etc. Drop the `Plugin` prefix
  from tool defs (one registry now).

## Decisions made

- **Drop dead `Tool` trait methods** (`9c356d8`) — `interactive_only` /
  `modes` / `decide_override` were defaults nothing overrode; the
  filter checks they fed in `ToolRegistry::definitions` were dead
  branches. Removed alongside the unused `interactive: bool` arg on
  `definitions`. Pre-P5.a-proper slim of the `Tool` trait surface;
  `needs_confirm` / `preflight` / `approval_patterns` (the methods
  with real overrides) ride P5.a-proper when their per-tool logic
  moves into a `BuiltinDispatcher::evaluate_hooks` impl.
- **Lift `Tool::is_mcp` to a `ToolEntry` data flag** (`030aecd`) — the
  trait method had one real override (`McpTool` → `true`) and was
  consumed by two callers (`ToolRegistry::definitions` for ruleset
  selection; `agent.rs` for `permissions.decide(..., is_mcp)`). The
  flag now lives on `ToolEntry { tool, is_mcp }` set at registration
  via `register` (false) / `register_mcp` (true). One step closer
  to a dispatch-only `Tool` trait.
- **Collapse `Tool::needs_confirm` + `approval_patterns` + `preflight`
  into a single `Tool::evaluate_hooks(&args) -> protocol::PluginToolHooks`**
  (this session) — three trait methods doing per-call hook evaluation
  fold into one returning the same `PluginToolHooks` shape plugin
  tools already use. `agent.rs`'s core-tool launch path now reads
  `hooks.preflight_error` / `.needs_confirm` / `.approval_patterns`
  off one struct, mirroring the plugin-tool branch (lines 1476+) byte
  for byte. Tests that called `tool.preflight(&args)` /
  `tool.approval_patterns(&args)` directly switch to
  `tool.evaluate_hooks(&args).{preflight_error, approval_patterns}`.
  The trait surface is now `name / description / parameters /
  execute / evaluate_hooks` — the exact shape of the future
  `ToolDispatcher::{evaluate_hooks, dispatch}` trait, just keyed
  on `&self` instead of `&str` name.
- **Extract `Turn::resolve_decision` shared between core+plugin
  paths** (this session) — the `permissions.decide(...) → runtime
  approvals upgrade Ask→Allow` block was duplicated verbatim across
  the core-tool and plugin-tool launch paths. Folded into one
  associated fn taking `(permissions, runtime_approvals, mode, name,
  args, is_mcp, hooks)`. Associated rather than `&self` because both
  call sites live inside or alongside a `select!` loop where
  `self.cmd_rx` is mutably borrowed. Drift insurance for the imminent
  dispatcher inversion.
- **Introduce `ToolDispatcher` trait + route engine through it**
  (`db2ed43`) — `pub(crate) trait ToolDispatcher` carries
  `definitions / contains / is_mcp / evaluate_hooks / dispatch`,
  impl'd for `ToolRegistry`. `Turn::registry: &'a ToolRegistry`
  becomes `dispatcher: &'a dyn ToolDispatcher`; `ToolSlot` drops its
  `tool: &'a dyn Tool` field and dispatches by name through the
  trait. `entry.is_mcp` access folds onto `dispatcher.is_mcp(name)`,
  the `entry.tool.evaluate_hooks(args)` call onto
  `dispatcher.evaluate_hooks(name, args)?`, and both
  `entry.tool.execute(args, ctx)` sites onto
  `dispatcher.dispatch(name, args, ctx).expect(...)`. Trait
  visibility is `pub(crate)` for now; the lift to `pub` +
  `engine::start` injection lands in the next sub-phase alongside
  `tui::ToolRuntime` first impl. The `Tool` trait surface is
  unchanged.
- **Drop `&Permissions, Mode` from `ToolDispatcher::definitions`**
  (`48e742d`) — the dispatcher trait is now permission-free:
  `definitions()` returns the unfiltered tool list and engine applies
  its per-mode `permissions.check_{tool,mcp}(mode, name)` filter
  externally over the result, using `dispatcher.is_mcp(name)` to pick
  the right ruleset. The inherent `ToolRegistry::definitions(&Permissions, Mode)`
  method retires; only the trait impl carries the schema-list logic.
  The trait surface no longer references `crate::permissions::*` —
  one notch closer to the P5.c shape where `tui::permissions` owns
  policy and engine emits `RequestPermission` only.
- **P5.e protocol rename pass** (`2320338`) — five mechanical
  renames at the wire boundary, landed in one commit:
  - `protocol::Mode` → `protocol::AgentMode` (every consumer across
    engine, tui, ui, and the binary updated to the new name; bare
    `Mode` imports collapse to `AgentMode`).
  - `UiCommand::SetMode` → `UiCommand::SetAgentMode` (matches the
    cell name and Lua-side `smelt.mode` namespace).
  - `EngineEvent::ExecutePluginTool` → `ToolDispatch`,
    `EvaluatePluginToolHooks` → `ToolHooksRequest`. The variant
    names no longer carry the "plugin" qualifier — engine sees one
    registry shape regardless of whether the impl is engine-side or
    tui/Lua-side.
  - `UiCommand::PluginToolResult` → `ToolResult`,
    `PluginToolHooksResult` → `ToolHooksResponse`. Variant `ToolResult`
    coexists fine with `engine::tools::ToolResult` — different
    namespaces, different concepts.
  - `protocol::{PluginToolDef, PluginToolHookFlags, PluginToolHooks}`
    → `{ToolDef, ToolHookFlags, ToolHooks}`. Tui-side names like
    `PluginToolEnv` / `PluginToolHandles` stay (those track the
    Lua-vs-Rust split inside the runtime, which is a tui-internal
    concern until P5.b migrates the Rust impls to Lua).
  Tests + clippy + format clean across the workspace; 968 passing.

- **`ask_user_question` relocates `plugins/` → `tools/`** (this session)
  — already-Lua tool moved to live alongside the rest of the migrated
  built-ins. Embedded module path renames `smelt.plugins.ask_user_question`
  → `smelt.tools.ask_user_question`. Tool body, schema, and
  `execution_mode = "sequential"` flow unchanged. `plan_mode.lua`
  deferred — its `exit_plan_mode` tool is registered/unregistered at
  mode-change time, and splitting that into a separate file would
  duplicate the activate/deactivate dance across two files for no
  win. Extraction reopens when the Lua keymap-registry / mode-hook
  refactor surfaces a cleaner shape (P3.b/P4 tail).
- **`stop_agent` migrates to Lua** (this session) —
  `engine/tools/stop_agent.rs` retires;
  `runtime/lua/smelt/tools/stop_agent.lua` registers as an
  `override = true` plugin tool composing
  `smelt.agent.{find_by_id,is_in_tree,kill,my_pid}` (new FFI over
  `engine::registry::*`, with `find_by_id` / `is_in_tree` lifted from
  `pub(crate)` → `pub`). Gated on `smelt.engine.multi_agent()`; same
  owner-only check the Rust impl enforced via
  `is_in_tree(target_pid, my_pid)`.
- **`list_agents` migrates to Lua** (this session) — `engine/tools/list_agents.rs`
  retires; `runtime/lua/smelt/tools/list_agents.lua` registers as an
  `override = true` plugin tool composing two new bindings:
  `smelt.agent.discover(scope?)` over `engine::registry::discover` (now
  `pub`) and `smelt.agent.my_pid()` over `std::process::id`. The Lua
  body reproduces the workspace-scope discovery + owned-vs-peer split
  + aligned columns. Gated on `smelt.engine.multi_agent()` so the LLM
  doesn't see the tool in single-agent runs (mirrors the
  `if let Some(ma) = ma` gate the Rust `ListAgentsTool` lived behind).
- **`load_skill` migrates to Lua** (this session) — `engine/tools/load_skill.rs`
  retires; `runtime/lua/smelt/tools/load_skill.lua` registers as an
  `override = true` plugin tool composing a new `smelt.skills.content`
  binding over `engine::SkillLoader`. The loader is shared: engine still
  consumes `prompt_section()` for the system prompt via
  `EngineConfig.skills`; tui parks an additional `Arc<SkillLoader>`
  clone on `Core.skills` so the Lua FFI reads the same data without
  rescanning. `SkillLoader::content` + a new `names()` accessor are now
  `pub`. `tools::build_tools` drops its `skills` param (no longer used
  engine-side); `engine::start` calls it with two args. The Rust
  `LoadSkillTool` and its registration retire entirely.
- **First P5.b tool migrations: `glob` + `grep` to Lua** (previous session) —
  `engine/tools/glob.rs` retires; `runtime/lua/smelt/tools/glob.lua`
  registers as an `override = true` plugin tool composing the new
  `tui::fs::glob` capability (globset + ignore::WalkBuilder). New
  `smelt.fs.glob(pattern, path?, { max })` Lua binding sorts mtime
  descending in the binding layer; the capability returns matches
  unsorted. `tui` Cargo gains `globset` + `ignore`; `tools/` directory
  introduced under `runtime/lua/smelt/`.
  `engine/tools/grep.rs` retires next; `tools/grep.lua` registers as
  an `override = true` plugin tool over `smelt.grep.run` (ripgrep)
  with a `smelt.process.run("grep", ...)` fallback when `rg` is
  missing. Lua-side `offset` / `head_limit` slicing + stdout+stderr
  concatenation reproduce the retired `GrepTool` semantics; schema
  preserves the `-i` / `-n` / `-A` / `-B` / `-C` / `context` /
  `multiline` / `output_mode` knobs. `engine::tools::run_command_with_timeout`
  loses its last caller and retires alongside.
  Both tools reuse the existing plugin-tool dispatch path (engine
  emits `ToolDispatch`, tui runs the Lua coroutine, replies with
  `ToolResult`) — the unified-dispatcher fold is still ahead.
- **`web_search` migrates to Lua + `tui::http` POST/cache/UA** (this session) —
  `engine/tools/web_search.rs` retires;
  `runtime/lua/smelt/tools/web_search.lua` registers as an
  `override = true` plugin tool composing four new Lua bindings:
  `smelt.http.post(url, body, opts)` over a new `tui::http::post`,
  `smelt.http.cache.{get,put}` over a new `tui::http::cache`
  submodule (relocated from `engine/tools/web_cache.rs` —
  same on-disk format / hash / TTL, both implementations now read &
  write the same `<cache_dir>/web/<hash>` files until `web_fetch`
  also retires from engine), `smelt.http.random_user_agent()` over a
  new `tui::http::random_user_agent` (USER_AGENTS list lifted off
  `engine/tools/web_shared.rs`), and `smelt.html.parse_ddg_results`
  over a new `tui::html::parse_ddg_results` (DOM walk + `uddg=`
  href un-wrap lifted off the retired Rust impl). 15-minute cache
  keyed `search:<query>`, rotated UA, top-20 results, formatted
  numbered list — all behaviour ports verbatim. `engine::paths::cache_dir`
  flips `pub(crate)` → `pub` so `tui::http::cache` can reach the
  same `<cache_dir>/web/` directory the engine cache uses; the rest
  of `engine/tools/web_cache.rs` stays put because `pricing.rs` and
  `web_fetch.rs` still call it (retires when `web_fetch` migrates).
- **`read_file` migrates to Lua** (this session) —
  `engine/tools/read_file.rs` retires;
  `runtime/lua/smelt/tools/read_file.lua` registers as an
  `override = true` plugin tool composing three new bindings plus the
  shared file-state cache: `smelt.image.{is_image_file,read_as_data_url}`
  (new `lua/api/image.rs` over `engine::image`) for the
  `.png/.jpg/...` short-circuit into a base64 `data:` URL,
  `smelt.notebook.read(path, offset, limit)` (over a new public
  `engine::tools::notebook::render_notebook_text` wrapper) for the
  line-numbered `.ipynb` projection, plus `smelt.fs.{read,
  file_state.{get,record_read,mtime_ms}}`. Pure-Lua `effective_range`
  / `dedup_stub` / `format_text_window` reproduce the retired
  `ReadFileTool` semantics: `FILE_UNCHANGED_STUB` on exact-range
  repeat against unchanged mtime, default `(1, 2000)` window,
  2000-byte per-line truncation, `"%4d\t%s"` line numbering. The
  unused `tools::int_arg` and `engine::tools::notebook::is_notebook`
  helpers retire alongside (last consumers — read/write/edit_file —
  all gone).
- **`edit_file` migrates to Lua** (previous session) —
  `engine/tools/edit_file.rs` retires;
  `runtime/lua/smelt/tools/edit_file.lua` registers as an
  `override = true` plugin tool composing `smelt.fs.{read,write,try_flock,
  file_state.{staleness_error,record_write}}` + `smelt.notebook.is_notebook_path`
  + `smelt.path.display`. Find/replace runs in pure Lua over
  `string.find(..., 1, true)` (plain-text, no regex) with
  `replace_first` / `replace_all` helpers — byte-exact match of the
  retired `EditFileTool` (reject identical old/new, reject not-found,
  reject ambiguous without `replace_all`, write under flock, record
  the write into the shared cache, output `"edited {display_path}"`).
- **`write_file` migrates to Lua** (this session) —
  `engine/tools/write_file.rs` retires;
  `runtime/lua/smelt/tools/write_file.lua` registers as an
  `override = true` plugin tool composing `smelt.fs.{exists,mkdir_all,write}`
  + the new `smelt.fs.try_flock(path)` userdata wrapper over
  `engine::tools::try_flock` (`:release()` + `__gc` drop the lock) +
  `smelt.fs.file_state.{has,staleness_error,record_write}` (parked on
  the same `Core.files` cache the engine still uses) +
  `smelt.notebook.is_notebook_path` + the new `smelt.path.display`
  binding over `engine::tools::display_path`. `WriteFileTool` /
  `mod write_file;` retire from `engine/tools/`. Behaviour ports
  verbatim — same `UNREAD_OVERWRITE_ERR`, same staleness preflight,
  same notebook redirect, same `wrote {len} bytes to {display_path}`
  ok message.
- **`FileStateCache` plumbed through `engine::start` + parked on Core**
  (this session) — `engine::tools::FileStateCache` (and the
  `FileState` / `file_mtime_ms` / `staleness_error` helpers) lift from
  `pub(crate)` to `pub`; `engine::start(config, files)` now takes the
  cache as a parameter (`tools::build_tools` fans it into the
  read/write/edit/notebook tools as before, no longer constructing one
  itself). `Core::new(..., files)` stores a clone on `Core.files` so
  Lua tools see the same observations the engine-side impls populate.
  `smelt.fs.file_state.{has,get,record_read,record_write,staleness_error,mtime_ms}`
  Lua bindings expose the cache. Sets up the next four P5.b sub-phases
  (write_file → edit_file → read_file → notebook_edit) so each can
  migrate without losing the cross-tool dedup + staleness coupling.
- **`peek_agent` + `message_agent` migrate to Lua** (previous session) —
  `engine/tools/peek_agent.rs` and `engine/tools/message_agent.rs`
  retire; the matching `runtime/lua/smelt/tools/*.lua` register as
  `override = true` plugin tools composing four new bindings:
  `smelt.agent.send_query` and `smelt.agent.send_message` over new
  `engine::socket::send_{query,message}_blocking` wrappers
  (`block_in_place` + `Handle::current().block_on(...)` so
  synchronous Lua coroutines reach the async Unix-socket I/O), plus
  `smelt.agent.my_id()` and `smelt.agent.my_slug()` over
  `engine::registry::read_entry(my_pid)`. `find_by_id`'s row gains
  `socket_path`. Both tools gate on `smelt.engine.multi_agent()`.
  The third-person framing in `peek_agent` and the partial-delivery
  reporting in `message_agent` port verbatim. With the `my_id` /
  `my_slug` arguments now sourced from the registry, the Rust
  shared-state plumbing on `MultiAgentToolConfig`
  (`agent_id: String`, `slug: Arc<Mutex<Option<String>>>`) drops,
  and `MultiAgentConfig.agent_id: Option<String>` (used only to
  seed those fields) drops with it; the registry entry created in
  `main.rs::run_normal` remains the single source of truth for
  identity.
