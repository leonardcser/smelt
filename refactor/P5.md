# P5 — Tools to Lua

**Status:** in-progress **Started:** 2026-05-01 **Landed:** —

P5 moves the 15 Rust tool implementations in `engine/tools/` into
`runtime/lua/smelt/tools/`. Engine becomes schema + dispatcher only.
Mode gating becomes a Lua-tool `hooks` concern. The `engine/permissions/`
module pulls out to `tui::permissions`; engine ends policy-free.

Sub-phases:

- **P5.a** — Tool dispatcher trait shape. Engine takes
  `Box<dyn ToolDispatcher>` at startup. `ToolSchema` slims; the
  `Tool` trait drops `needs_confirm` / `preflight` /
  `approval_patterns` / `modes` / `decide_override`.
- **P5.b** — Migrate core tools to Lua. 15 tools relocate to
  `runtime/lua/smelt/tools/`; intricate logic stays in Rust as
  `tui::*` capabilities the Lua tools FFI into.
- **P5.c** — Engine cleanup. `engine/permissions/` → `tui::permissions`.
  Drop the multi-agent concept (`Role::Agent`, `EngineEvent::AgentMessage`,
  etc.). `EngineHandle` becomes channels-only. Drop
  `EngineConfig.interactive`.
- **P5.d** — Drop `config.yaml`; all config in `init.lua`. Drop
  `serde_yml` dep + the YAML loader.
- **P5.e** — Protocol rename pass. `Mode → AgentMode`,
  `ExecutePluginTool → ToolDispatch`, etc. Drop the `Plugin` prefix
  from tool defs (one registry now).

## Decisions made

- **Drop dead `Tool` trait methods** (`9c356d8`) — `interactive_only` /
  `modes` / `decide_override` were defaults nothing overrode; the
  filter checks they fed in `ToolRegistry::definitions` were dead
  branches. Removed alongside the unused `interactive: bool` arg on
  `definitions`. Pre-P5.a-proper slim of the `Tool` trait surface;
  `needs_confirm` / `preflight` / `approval_patterns` (the methods
  with real overrides) ride P5.a-proper when their per-tool logic
  moves into a `BuiltinDispatcher::evaluate_hooks` impl.
- **Lift `Tool::is_mcp` to a `ToolEntry` data flag** (`030aecd`) — the
  trait method had one real override (`McpTool` → `true`) and was
  consumed by two callers (`ToolRegistry::definitions` for ruleset
  selection; `agent.rs` for `permissions.decide(..., is_mcp)`). The
  flag now lives on `ToolEntry { tool, is_mcp }` set at registration
  via `register` (false) / `register_mcp` (true). One step closer
  to a dispatch-only `Tool` trait.
- **Collapse `Tool::needs_confirm` + `approval_patterns` + `preflight`
  into a single `Tool::evaluate_hooks(&args) -> protocol::PluginToolHooks`**
  (this session) — three trait methods doing per-call hook evaluation
  fold into one returning the same `PluginToolHooks` shape plugin
  tools already use. `agent.rs`'s core-tool launch path now reads
  `hooks.preflight_error` / `.needs_confirm` / `.approval_patterns`
  off one struct, mirroring the plugin-tool branch (lines 1476+) byte
  for byte. Tests that called `tool.preflight(&args)` /
  `tool.approval_patterns(&args)` directly switch to
  `tool.evaluate_hooks(&args).{preflight_error, approval_patterns}`.
  The trait surface is now `name / description / parameters /
  execute / evaluate_hooks` — the exact shape of the future
  `ToolDispatcher::{evaluate_hooks, dispatch}` trait, just keyed
  on `&self` instead of `&str` name.
