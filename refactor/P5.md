# P5 — Tools to Lua

**Status:** in-progress **Started:** 2026-05-01 **Landed:** —

P5 moves the 15 Rust tool implementations in `engine/tools/` into
`runtime/lua/smelt/tools/`. Engine becomes schema + dispatcher only.
Mode gating becomes a Lua-tool `hooks` concern. The `engine/permissions/`
module pulls out to `tui::permissions`; engine ends policy-free.

Sub-phases:

- **P5.a** — Tool dispatcher trait shape. Engine takes
  `Box<dyn ToolDispatcher>` at startup. `ToolSchema` slims; the
  `Tool` trait drops `needs_confirm` / `preflight` /
  `approval_patterns` / `modes` / `decide_override`.
- **P5.b** — Migrate core tools to Lua. 15 tools relocate to
  `runtime/lua/smelt/tools/`; intricate logic stays in Rust as
  `tui::*` capabilities the Lua tools FFI into.
- **P5.c** — Engine cleanup. `engine/permissions/` → `tui::permissions`.
  Drop the multi-agent concept (`Role::Agent`, `EngineEvent::AgentMessage`,
  etc.). `EngineHandle` becomes channels-only. Drop
  `EngineConfig.interactive`.
- **P5.d** — Drop `config.yaml`; all config in `init.lua`. Drop
  `serde_yml` dep + the YAML loader.
- **P5.e** — Protocol rename pass. `Mode → AgentMode`,
  `ExecutePluginTool → ToolDispatch`, etc. Drop the `Plugin` prefix
  from tool defs (one registry now).

## Decisions made

- **Drop dead `Tool` trait methods** (`9c356d8`) — `interactive_only` /
  `modes` / `decide_override` were defaults nothing overrode; the
  filter checks they fed in `ToolRegistry::definitions` were dead
  branches. Removed alongside the unused `interactive: bool` arg on
  `definitions`. Pre-P5.a-proper slim of the `Tool` trait surface;
  `needs_confirm` / `preflight` / `approval_patterns` (the methods
  with real overrides) ride P5.a-proper when their per-tool logic
  moves into a `BuiltinDispatcher::evaluate_hooks` impl.
- **Lift `Tool::is_mcp` to a `ToolEntry` data flag** (`030aecd`) — the
  trait method had one real override (`McpTool` → `true`) and was
  consumed by two callers (`ToolRegistry::definitions` for ruleset
  selection; `agent.rs` for `permissions.decide(..., is_mcp)`). The
  flag now lives on `ToolEntry { tool, is_mcp }` set at registration
  via `register` (false) / `register_mcp` (true). One step closer
  to a dispatch-only `Tool` trait.
- **Collapse `Tool::needs_confirm` + `approval_patterns` + `preflight`
  into a single `Tool::evaluate_hooks(&args) -> protocol::PluginToolHooks`**
  (this session) — three trait methods doing per-call hook evaluation
  fold into one returning the same `PluginToolHooks` shape plugin
  tools already use. `agent.rs`'s core-tool launch path now reads
  `hooks.preflight_error` / `.needs_confirm` / `.approval_patterns`
  off one struct, mirroring the plugin-tool branch (lines 1476+) byte
  for byte. Tests that called `tool.preflight(&args)` /
  `tool.approval_patterns(&args)` directly switch to
  `tool.evaluate_hooks(&args).{preflight_error, approval_patterns}`.
  The trait surface is now `name / description / parameters /
  execute / evaluate_hooks` — the exact shape of the future
  `ToolDispatcher::{evaluate_hooks, dispatch}` trait, just keyed
  on `&self` instead of `&str` name.
- **Extract `Turn::resolve_decision` shared between core+plugin
  paths** (this session) — the `permissions.decide(...) → runtime
  approvals upgrade Ask→Allow` block was duplicated verbatim across
  the core-tool and plugin-tool launch paths. Folded into one
  associated fn taking `(permissions, runtime_approvals, mode, name,
  args, is_mcp, hooks)`. Associated rather than `&self` because both
  call sites live inside or alongside a `select!` loop where
  `self.cmd_rx` is mutably borrowed. Drift insurance for the imminent
  dispatcher inversion.
- **Introduce `ToolDispatcher` trait + route engine through it**
  (`db2ed43`) — `pub(crate) trait ToolDispatcher` carries
  `definitions / contains / is_mcp / evaluate_hooks / dispatch`,
  impl'd for `ToolRegistry`. `Turn::registry: &'a ToolRegistry`
  becomes `dispatcher: &'a dyn ToolDispatcher`; `ToolSlot` drops its
  `tool: &'a dyn Tool` field and dispatches by name through the
  trait. `entry.is_mcp` access folds onto `dispatcher.is_mcp(name)`,
  the `entry.tool.evaluate_hooks(args)` call onto
  `dispatcher.evaluate_hooks(name, args)?`, and both
  `entry.tool.execute(args, ctx)` sites onto
  `dispatcher.dispatch(name, args, ctx).expect(...)`. Trait
  visibility is `pub(crate)` for now; the lift to `pub` +
  `engine::start` injection lands in the next sub-phase alongside
  `tui::ToolRuntime` first impl. The `Tool` trait surface is
  unchanged.
- **Drop `&Permissions, Mode` from `ToolDispatcher::definitions`**
  (`48e742d`) — the dispatcher trait is now permission-free:
  `definitions()` returns the unfiltered tool list and engine applies
  its per-mode `permissions.check_{tool,mcp}(mode, name)` filter
  externally over the result, using `dispatcher.is_mcp(name)` to pick
  the right ruleset. The inherent `ToolRegistry::definitions(&Permissions, Mode)`
  method retires; only the trait impl carries the schema-list logic.
  The trait surface no longer references `crate::permissions::*` —
  one notch closer to the P5.c shape where `tui::permissions` owns
  policy and engine emits `RequestPermission` only.
- **P5.e protocol rename pass** (`2320338`) — five mechanical
  renames at the wire boundary, landed in one commit:
  - `protocol::Mode` → `protocol::AgentMode` (every consumer across
    engine, tui, ui, and the binary updated to the new name; bare
    `Mode` imports collapse to `AgentMode`).
  - `UiCommand::SetMode` → `UiCommand::SetAgentMode` (matches the
    cell name and Lua-side `smelt.mode` namespace).
  - `EngineEvent::ExecutePluginTool` → `ToolDispatch`,
    `EvaluatePluginToolHooks` → `ToolHooksRequest`. The variant
    names no longer carry the "plugin" qualifier — engine sees one
    registry shape regardless of whether the impl is engine-side or
    tui/Lua-side.
  - `UiCommand::PluginToolResult` → `ToolResult`,
    `PluginToolHooksResult` → `ToolHooksResponse`. Variant `ToolResult`
    coexists fine with `engine::tools::ToolResult` — different
    namespaces, different concepts.
  - `protocol::{PluginToolDef, PluginToolHookFlags, PluginToolHooks}`
    → `{ToolDef, ToolHookFlags, ToolHooks}`. Tui-side names like
    `PluginToolEnv` / `PluginToolHandles` stay (those track the
    Lua-vs-Rust split inside the runtime, which is a tui-internal
    concern until P5.b migrates the Rust impls to Lua).
  Tests + clippy + format clean across the workspace; 968 passing.

- **First P5.b tool migrations: `glob` + `grep` to Lua** (this session) —
  `engine/tools/glob.rs` retires; `runtime/lua/smelt/tools/glob.lua`
  registers as an `override = true` plugin tool composing the new
  `tui::fs::glob` capability (globset + ignore::WalkBuilder). New
  `smelt.fs.glob(pattern, path?, { max })` Lua binding sorts mtime
  descending in the binding layer; the capability returns matches
  unsorted. `tui` Cargo gains `globset` + `ignore`; `tools/` directory
  introduced under `runtime/lua/smelt/`.
  `engine/tools/grep.rs` retires next; `tools/grep.lua` registers as
  an `override = true` plugin tool over `smelt.grep.run` (ripgrep)
  with a `smelt.process.run("grep", ...)` fallback when `rg` is
  missing. Lua-side `offset` / `head_limit` slicing + stdout+stderr
  concatenation reproduce the retired `GrepTool` semantics; schema
  preserves the `-i` / `-n` / `-A` / `-B` / `-C` / `context` /
  `multiline` / `output_mode` knobs. `engine::tools::run_command_with_timeout`
  loses its last caller and retires alongside.
  Both tools reuse the existing plugin-tool dispatch path (engine
  emits `ToolDispatch`, tui runs the Lua coroutine, replies with
  `ToolResult`) — the unified-dispatcher fold is still ahead.
