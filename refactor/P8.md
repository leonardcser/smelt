## Decisions made

- **Lua FFI tier split (`core/lua/api/` vs `lua/api/`).** Host-tier bindings
  (subsystems, capabilities, config registries) live in `core/src/lua/api/`;
  UiHost-tier bindings (buffer/window/overlay operations, render primitives,
  theme access) stay in `tui/src/lua/api/`. The split is now physical across
  crates, not just organizational. `core/lua/api/` files use `try_with_host`
  (via the `host_read!` macro) which resolves through a TLS `CORE_PTR` holding
  `&mut dyn Host`. `register_host_api` receives both `smelt` and `smelt_keymap`
  because `smelt.keymap` is a Host-tier namespace that writes to a dedicated
  sub-table.

- **`app/` renames to `core/` and capability modules consolidate (`37c153be`).**
  The `app/` module (P2 aggregate) is renamed to `core/` to match the target
  crate name. All generic capability modules (`fs`, `fuzzy`, `grep`, `html`,
  `http`, `notebook`, `path`, `permissions`, `process`) move inside `core/` so
  the eventual physical split only moves one directory. Terminal-specific files
  (`events.rs`, `mouse.rs`, `render_loop.rs`, `cmdline.rs`, etc.) stay in
  `core/` for now and move to `tui/src/` in P8.a.

- **Create `crates/core` now — not deferred.** The previous plan gated the
  physical split on a third frontend (GUI / server). That gate is removed; the
  split happens in P8 regardless. Two crates (`core` + `tui`) are strictly
  better than one internal module split, because they enforce the dependency
  arrow at the compiler level.

- **`crossterm` lives in `tui` only.** `protocol`, `engine`, and `core` compile
  without it. `ui` (now `tui/src/ui/`) is a terminal UI primitives module —
  Buffer, Window, Grid, LayoutTree, VimMode — all inherently terminal-specific.
  A GUI or web frontend uses `core` directly and builds its own widgets; it has
  no use for `ui`. `core` has zero `crossterm` and zero `ui` imports.

- **`ui` absorbs into `tui`.** `ui` was factored out as a separate crate under
  the theory it could ship as a standalone TUI library. In practice it is too
  specialized (nvim-style extmarks, vim key dispatch, cell-grid rendering) and
  is tightly coupled to `tui` (529 `ui::` references, the `UiHost` trait
  implemented only by `TuiApp`). Four crates (`protocol`, `engine`, `core`,
  `tui`) is simpler than five. `ui` becomes `tui/src/ui/`.

- **`app.rs` is the module root, not `app/mod.rs`.** Rust 2018 supports both
  `foo.rs` + `foo/` and `foo/mod.rs`. The `app.rs` + `app/` pattern is chosen
  because `app.rs` carries the `TuiApp` struct and event loop, while `app/`
  carries the 12 `impl TuiApp` submodules. This keeps the module root file
  meaningful rather than a thin shim.

- **`pub mod app` over `pub(crate) mod app`.** The `tui` library's binary entry
  point lives in the workspace-root `src/main.rs`, not inside `crates/tui/`.
  When `app` is `pub(crate)`, rustc's dead-code lint considers `TuiApp::run`
  dead because no code inside the `tui` library calls it. Making `app` `pub`
  promotes `TuiApp::run` to the crate's public API surface from the lint's
  perspective, eliminating ~340 false-positive dead-code warnings.

- **`term/` dissolves — it is a transitional module.** `tui` is already the
  terminal crate; an extra `term/` layer inside it is redundant. The split that
  matters is `core` (headless) vs `tui` (terminal chrome), not `tui` vs
  `tui::term`. During P8:
  - `term/content/` splits: headless-safe display model (`DisplayBlock`,
    `LayoutContext`, `SpanCollector`, `stream_parser.rs`, `highlight/`) moves to
    `core/content/`; terminal rendering adapters (`to_buffer.rs`,
    `prompt_data.rs`, `status.rs`) stay in `tui`.
  - `term/input/` splits: data-model pieces (`History`, `PromptState`) move to
    `core`; terminal editing chrome (`completer_bridge.rs`, `vim_bridge.rs`,
    crossterm key dispatch in `mod.rs`) stays in `tui`.
  - `term/window.rs` stays in `tui`. This eliminates the circular dependency
    smell where `core` imports `crate::term::content::*` — a module that lives
    inside the `tui` crate.

## Deferrals

- **`LuaRuntime` core/tui split — done.** Core-safe skeleton (`Lua` state,
  shared registries, command/keymap dispatch, config/permission extraction,
  bootstrap loading) lives in `core/src/lua/runtime.rs`. Tui's `LuaRuntime`
  wraps it via `Deref`/`DerefMut` and adds UI-specific methods (`tick_statusline`,
  `prepare_invocation`, `queue_invocation`, `drain_invocations`).

- **Orphaned UiHost-tier modules rewired.** `engine`, `history`, `keymap`,
  `metrics`, `model`, `permissions`, `session`, `settings`, `transcript`, `vim`
  were present in `tui/src/lua/api/` but not declared in `mod.rs` after P8.c,
  causing `smelt.*` nil errors at runtime. All are now wired into
  `tui/src/lua/api/mod.rs` and compile. Host-tier migration of these modules
  to `core` is deferred to future work if a genuine headless use case appears.

- **Scenario snapshot tests.** 5 `smelt-agent::scenarios` tests fail due to
  snapshot mismatches (pre-existing, unrelated to P8).

## Open questions — resolved

- **Clipboard.** `Clipboard`, `KillRing`, and the `Sink` trait move from `ui` to
  `core`. They are text-manipulation primitives, not UI rendering primitives —
  `KillRing` is pure `std`, `Sink` is a two-method platform abstraction, and
  `ui` itself (Buffer, Window, Grid) never touches them. `SystemSink` /
  `Osc52Sink` moved from `core/commands.rs` to `core/clipboard.rs` and implement
  `core::Sink`. This removes the `core → ui` dependency cleanly without
  inventing a wrapper trait.

- **VimMode.** `Core` stores `vim_mode` as a plain `String` cell (seeded
  `"Insert"`). `TuiApp` owns the typed `ui::VimMode` and publishes
  `format!("{:?}", vim_mode)` into the cell. Lua reads the cell string via
  `smelt.vim.mode`; it never sees the enum. This keeps the typed vim state
  machine in `ui` where it belongs while letting `core` stay ignorant of it.
