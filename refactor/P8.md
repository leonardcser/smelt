# P8 — Crate extraction

**Status:** in-progress **Started:** 2026-05-03 **Landed:**

## Decisions made

- **`app/` renames to `core/` and capability modules consolidate (`37c153be`).** The `app/` module (P2 aggregate) is renamed to `core/` to match the target crate name. All generic capability modules (`fs`, `fuzzy`, `grep`, `html`, `http`, `notebook`, `path`, `permissions`, `process`) move inside `core/` so the eventual physical split only moves one directory. Terminal-specific files (`events.rs`, `mouse.rs`, `render_loop.rs`, `cmdline.rs`, etc.) stay in `core/` for now and move to `tui/src/` in P8.a.

- **Create `crates/core` now — not deferred.** The previous plan gated the
  physical split on a third frontend (GUI / server).  That gate is removed;
  the split happens in P8 regardless.  Two crates (`core` + `tui`) are
  strictly better than one internal module split, because they enforce the
  dependency arrow at the compiler level.

- **`crossterm` lives in `ui` and `tui`.** `protocol` and `engine` compile
  without it.  `ui` is a terminal UI primitives crate — Buffer, Window,
  Grid, LayoutTree, VimMode — all inherently terminal-specific.  A GUI or
  web frontend uses `core` directly and builds its own widgets; it has no
  use for `ui`.  Keeping `crossterm` in `ui` makes `ui` a reusable TUI
  library rather than an awkward abstraction that satisfies no one.
  `core` has zero `crossterm` and zero `ui` imports.

- **`ui` absorbs into `tui`.** `ui` was factored out as a separate crate
  under the theory it could ship as a standalone TUI library.  In practice
  it is too specialized (nvim-style extmarks, vim key dispatch, cell-grid
  rendering) and is tightly coupled to `tui` (529 `ui::` references, the
  `UiHost` trait implemented only by `TuiApp`).  Four crates
  (`protocol`, `engine`, `core`, `tui`) is simpler than five.  `ui`
  becomes `tui/src/ui/`.

- **`term/` dissolves — it is a transitional module.** `tui` is already the
  terminal crate; an extra `term/` layer inside it is redundant.  The split
  that matters is `core` (headless) vs `tui` (terminal chrome), not `tui`
  vs `tui::term`.  During P8:
  - `term/content/` splits: headless-safe display model (`DisplayBlock`,
    `LayoutContext`, `SpanCollector`, `stream_parser.rs`, `highlight/`) moves
    to `core/content/`; terminal rendering adapters (`to_buffer.rs`,
    `prompt_data.rs`, `status.rs`) stay in `tui`.
  - `term/input/` splits: data-model pieces (`History`, `PromptState`) move
    to `core`; terminal editing chrome (`completer_bridge.rs`,
    `vim_bridge.rs`, crossterm key dispatch in `mod.rs`) stays in `tui`.
  - `term/window.rs` stays in `tui`.
  This eliminates the circular dependency smell where `core` imports
  `crate::term::content::*` — a module that lives inside the `tui` crate.

## Deferrals

- Lua FFI tier split (P8.c) — can start once `core` has no `ui` imports;
  blocked until P8.b completes.

## Open questions — resolved

- **Clipboard.** `Clipboard`, `KillRing`, and the `Sink` trait move from
  `ui` to `core`.  They are text-manipulation primitives, not UI
  rendering primitives — `KillRing` is pure `std`, `Sink` is a two-method
  platform abstraction, and `ui` itself (Buffer, Window, Grid) never
  touches them.  `SystemSink` / `Osc52Sink` stay in `tui` and implement
  `core::Sink`.  This removes the `core → ui` dependency cleanly without
  inventing a wrapper trait.

- **VimMode.** `Core` stores `vim_mode` as a plain `String` cell (seeded
  `"Insert"`).  `TuiApp` owns the typed `ui::VimMode` and publishes
  `format!("{:?}", vim_mode)` into the cell.  Lua reads the cell string
  via `smelt.vim.mode`; it never sees the enum.  This keeps the typed
  vim state machine in `ui` where it belongs while letting `core` stay
  ignorant of it.
