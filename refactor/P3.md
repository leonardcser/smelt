# P3 — Rust capabilities + Lua API split

**Status:** done **Started:** 2026-04-30 **Landed:** 2026-04-30

P3 puts the generic Rust capabilities behind named modules under
`tui::*` (parse / process / subprocess / fs / http / html / notebook /
grep / path / fuzzy / permissions) and splits the Lua bindings to one
file per `smelt.*` namespace under `crates/tui/src/lua/api/<name>.rs`.
Mostly mechanical; unblocks P4 (Lua takes the UX) and P5 (tools to
Lua). Sub-phases land independently.

## Decisions made

- **`tui::fuzzy` is a folder module, not a flat file** — `mod.rs` keeps `fuzzy_score`; `score.rs` carries the completer ranking helpers. They share the same ranking surface and the completer is the only consumer of `score::*` today.
- **`tui::fs` capability shells without engine absorption** — P3.a's plan called for absorbing engine tool files, but the engine→tui dependency arrow blocks that until those tools migrate to Lua in P5.b. `tui::fs` lands as a sync std::fs wrapper module. Same shape applies to the rest of P3.a engine moves (`http`, `process`, `grep`, `notebook`, `permissions`).
- **`smelt.os` has no backing Rust capability module** — every binding is a one-liner over `std::env` / `std::process` / `dirs`; introducing a `tui::os` module would add indirection without consolidating logic.
- **`smelt.html` ships read-only `title` / `links` / `to_text`, not `to_markdown`** — the engine's HTML→markdown converter lives entwined with `web_shared.rs` redirect / cache / UA-rotation logic; 400+ LOC. The full converter migrates with `web_fetch` in P5.b.
- **`tui::http` is sync (`reqwest::blocking`), not async** — Lua bindings run on the main thread inside the tokio runtime, so calling an async client's `block_on` deadlocks. Adding `reqwest`'s `blocking` feature lets `smelt.http.get` look like every other Host-tier binding.
- **Per-language Lua bindings (`smelt.diff` / `smelt.syntax` / `smelt.bash` / `smelt.notebook`) stay top-level** — per Decision in `9d40423`, rather than folding under a single `smelt.parse`.

## Deferrals

- **P2.a.10 (ToolRuntime carve-out)** — lands alongside P5.a's `engine::ToolDispatcher` trait shape.
- **P2.b.5b** — TLS-slot generalization (Tui-or-Headless); lands with the first headless Lua driver in P5.b.
- **P2.b.5c** — bulk-migrate remaining `with_app` callsites; lands after b.5b.
- **P2.d.2** — full EngineBridge target fold (TextDelta → Buffer::append, on_block fan-out at semantic boundaries); gated on P1.a tail (`Buffer::attach`).
- **`tui::parse` + `smelt.parse`** — relocating the parse code from `content/highlight/*` + `content/stream_parser` into a `tui::parse::*` namespace requires a clean parse/present split. Lands when transcript renderers migrate onto `BufferParser` in P4.b (gated on P1.a-tail `Buffer::attach`). Per Decision in `9d40423`, per-language Lua bindings stay top-level and supersede `smelt.parse`.
- **`tui::subprocess` + `smelt.subprocess`** — future generic child-process primitive; gated on demand.
- **`tui::permissions` full absorption** — P3.a shipped only the workspace JSON store half; P5.c absorbs `engine/permissions/{approvals,bash,rules,workspace,tests}` when engine becomes policy-free.
