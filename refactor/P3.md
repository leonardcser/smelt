# P3 — Rust capabilities + Lua API split

**Status:** done **Started:** 2026-04-30 **Landed:** 2026-04-30

P3 puts the generic Rust capabilities behind named modules under
`tui::*` (parse / process / subprocess / fs / http / html / notebook /
grep / path / fuzzy / permissions) and splits the Lua bindings to one
file per `smelt.*` namespace under `crates/tui/src/lua/api/<name>.rs`.
Mostly mechanical; unblocks P4 (Lua takes the UX) and P5 (tools to
Lua). Sub-phases land independently.

## Decisions made

- **`tui::fuzzy` is a folder module, not a flat file** (`b537d1a`) — `mod.rs` keeps `fuzzy_score`; `score.rs` carries the completer ranking helpers (`query_match_score` / `split_words` / `recency_bonus`). They share the same ranking surface and the completer is the only consumer of `score::*` today; co-locating keeps the future `smelt.fuzzy` Lua binding (P3.c) one import away.
- **`tui::fs` capability shells without engine absorption** (`5de3054`) — P3.a's plan called for `tui::fs` to absorb `engine/tools/{read_file, write_file, edit_file, glob, file_state}.rs`, but the engine→tui dependency arrow blocks that until those tools migrate to Lua in P5.b. `tui::fs` lands now as a sync std::fs wrapper module so plugins and future Lua tools have a stable surface; the engine tool absorption rides P5.b alongside each tool's migration. Same shape applies to the rest of the P3.a engine moves (`tui::http`, `tui::process`, `tui::grep`, `tui::notebook`, `tui::permissions`).
- **`smelt.os` has no backing Rust capability module** (`5de3054`) — every binding (`getenv` / `setenv` / `platform` / etc.) is a one-liner over `std::env` / `std::process` / `dirs`; introducing a `tui::os` module would add a layer of indirection without consolidating any logic.
- **`smelt.html` ships read-only `title` / `links` / `to_text`, not `to_markdown`** (`16bf9d7`) — the engine's HTML→markdown converter lives entwined with `web_shared.rs` redirect / cache / UA-rotation logic; lifting it whole means moving 400+ LOC plus its caller. The fresh tui::html shell ships the pieces a Lua plugin can use today (page title, link extraction, plain-text projection); the full converter migrates with `web_fetch` in P5.b.
- **`tui::http` is sync (`reqwest::blocking`), not async** (`16bf9d7`) — Lua bindings run on the main thread inside the tokio runtime, so calling an async client's `block_on` deadlocks. Adding `reqwest`'s `blocking` feature lets `smelt.http.get` look like every other Host-tier binding (no coroutine yield plumbing). Async / streaming variants ride P5.b alongside `web_fetch`.

## Deferrals

- **P2.a.10 (ToolRuntime carve-out)** — lands alongside P5.a's
  `engine::ToolDispatcher` trait shape.
- **P2.b.5b** — TLS-slot generalization (Tui-or-Headless); lands
  with the first headless Lua driver in P5.b.
- **P2.b.5c** — bulk-migrate remaining `with_app` callsites; lands
  after b.5b.
- **P2.d.2** — full EngineBridge target fold (TextDelta →
  Buffer::append, on_block fan-out at semantic boundaries); gated on
  P1.a tail (`Buffer::attach`).
- **`tui::parse` + `smelt.parse`** — relocating the parse code from
  `content/highlight/{inline,diff,syntax,util}` + `content/stream_parser`
  into a `tui::parse::*` namespace requires a clean parse/present split
  (parse output as typed data; presenters consume from there). The
  current files mix parse with presenter code (`print_inline_diff`,
  `BashHighlighter`, `print_syntax_file`, `render_code_block`). The
  split lands when transcript renderers migrate onto `BufferParser`
  in P4.b (gated on P1.a-tail `Buffer::attach`). Per Decision in
  `9d40423`, the per-language Lua bindings (`smelt.diff` / `smelt.syntax`
  / `smelt.bash` / `smelt.notebook`) stay top-level and supersede
  `smelt.parse`; the namespace lands in P4.b only if a generic
  block-level parser surface is needed for `Buffer::attach { parser =
  "markdown" }`.
- **`tui::subprocess` + `smelt.subprocess`** — gated on P5.c removing
  the multi-agent concept from engine. `engine/socket.rs` +
  `engine/registry.rs` + the multi-agent IPC pieces of `engine/agent.rs`
  relocate to `tui::subprocess::*` then.
- **`tui::permissions` full absorption** — P3.a shipped only the
  workspace JSON store half (`tui::permissions::store`, `e34da06`);
  P5.c absorbs `engine/permissions/{approvals,bash,rules,workspace,tests}`
  when engine becomes policy-free.
