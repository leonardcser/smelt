# P1 — UI primitives

**Status:** in-progress **Started:** 2026-04-29 **Landed:** —

P1 rebuilds `ui` around the target primitives (`Buffer`, `LayoutTree`,
`Overlay`, `Window`, `Theme`, `UiHost`, `Ui` facade) and absorbs the
structural deletions P0 deferred. Sub-phases land independently, each
green at the boundary.

## Decisions made

- **Theme registry owns is_light/accent/slug as fields, not atomics** — no global mutable state path; `Ui` is single owner.
- **Syntect light/dark hint stays as a module-local atomic** in `content/highlight/mod.rs` — threading `&Theme` through 14 syntect call sites for one boolean was disproportionate.
- **P1.a closes as foundation; structural completion deferred to P1.a-tail** — gated on transcript-pipeline migration onto `BufferParser` (multi-session keystone).
- **`YankSubst` and `Buffer::wrap_at` shipped without consumers** — original picks (hidden-thinking elision, attachment expansion, WindowView wrap) turned out wrong fits; real consumers come with the transcript migration.
- **Builder constructors instead of struct-literal field-defaults for B.2** — `Direction` had no sensible Default; builders also became the eventual `Vbox`/`Hbox` constructors needed in B.4.
- **Single `SeparatorStyle` (in `layout`), no `Option<…>` wrapper** — dialog's same-named 2-variant enum deleted; consolidated on 3-variant layout-side enum (`None / Solid / Dashed`).
- **C.5 stops at design note before code ships** — first float migration needed either pulling `Window::render` forward from P1.d or a temporary Component-carrier on Overlay; chose path 1 (extract a small Window-render helper scoped to read-only viewer windows).
- **Test markers (`A`/`B`/`C` constants) instead of synthetic `WinId::new`** — layout tests need `WinId` values that don't collide with `PROMPT_WIN(0)` / `TRANSCRIPT_WIN(1)`.
- **List/input keymap recipes live in Rust until P3.b** — target shape says recipes are Lua, but list-Enter and input-printable are fundamental enough to bind in Rust until the keymap-recipe registry takes shape.
- **`Anchor::ScreenBottom { above_rows }` instead of a layout-side `dock_bottom_full_width` builder** — anchor positions, layout sizes; the legacy `Placement::DockBottom` mixed both. Anchor takes the layout's natural size (full-width via `Percentage(100)` outer hbox), clamps height to `term_h - above_rows`, pins to the bottom edge.
- **C.9 splits into C.9a / C.9b / C.9c** — too large for one session. C.9a lands the dock-bottom anchor + overlay-path collapse mechanism + dead-code deletion (foundation). C.9b flips `confirm.lua` (the only legacy consumer). C.9c deletes `FloatConfig` / `Placement` / `PanelWidget` / `dialog.rs` multiplexing.
- **`Overlay::blocks_agent` is independent of `modal`** — `modal` scopes input (focus + Esc-dismiss + Tab cycling); `blocks_agent` gates engine-event drain. `/help` and `/btw` are modal but never `blocks_agent`; confirm.lua is both. `focused_float_blocks_agent` checks `focused_overlay`'s `blocks_agent` field before falling back to the legacy `FloatConfig` check.
- **List leaves fire `SelectionChanged` on cursor move** — confirm.lua needs to know the option panel's selection at submit time even when Submit fires from the reason input leaf. The recipe now returns `CallbackResult::Event(SelectionChanged, Selection { index })` from `move_cursor` so subscribers track the live selection. Cursor-no-op moves (already at first/last row) stay `Consumed`.
