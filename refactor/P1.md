# P1 — UI primitives

**Status:** in-progress **Started:** 2026-04-29 **Landed:** —

P1 rebuilds `ui` around the target primitives (`Buffer`, `LayoutTree`,
`Overlay`, `Window`, `Theme`, `UiHost`, `Ui` facade) and absorbs the
structural deletions P0 deferred. Sub-phases land independently, each
green at the boundary.

## Decisions made

- **Theme registry owns is_light/accent/slug as fields, not atomics** — no global mutable state path; `Ui` is single owner.
- **Syntect light/dark hint stays as a module-local atomic** in `content/highlight/mod.rs` — threading `&Theme` through 14 syntect call sites for one boolean was disproportionate.
- **P1.a closes as foundation; structural completion deferred to P1.a-tail** — gated on transcript-pipeline migration onto `BufferParser` (multi-session keystone).
- **`YankSubst` and `Buffer::wrap_at` shipped without consumers** — original picks (hidden-thinking elision, attachment expansion, WindowView wrap) turned out wrong fits; real consumers come with the transcript migration.
- **Builder constructors instead of struct-literal field-defaults for B.2** — `Direction` had no sensible Default; builders also became the eventual `Vbox`/`Hbox` constructors needed in B.4.
- **Single `SeparatorStyle` (in `layout`), no `Option<…>` wrapper** — dialog's same-named 2-variant enum deleted; consolidated on 3-variant layout-side enum (`None / Solid / Dashed`).
- **C.5 stops at design note before code ships** — first float migration needed either pulling `Window::render` forward from P1.d or a temporary Component-carrier on Overlay; chose path 1 (extract a small Window-render helper scoped to read-only viewer windows).
- **Test markers (`A`/`B`/`C` constants) instead of synthetic `WinId::new`** — layout tests need `WinId` values that don't collide with `PROMPT_WIN(0)` / `TRANSCRIPT_WIN(1)`.
- **List/input keymap recipes live in Rust until P3.b** — target shape says recipes are Lua, but list-Enter and input-printable are fundamental enough to bind in Rust until the keymap-recipe registry takes shape.
- **`Anchor::ScreenBottom { above_rows }` instead of a layout-side `dock_bottom_full_width` builder** — anchor positions, layout sizes; the legacy `Placement::DockBottom` mixed both. Anchor takes the layout's natural size (full-width via `Percentage(100)` outer hbox), clamps height to `term_h - above_rows`, pins to the bottom edge.
- **C.9 splits into C.9a / C.9b / C.9c** — too large for one session. C.9a lands the dock-bottom anchor + overlay-path collapse mechanism + dead-code deletion (foundation). C.9b flips `confirm.lua` (the only legacy consumer). C.9c deletes `FloatConfig` / `Placement` / `PanelWidget` / `dialog.rs` multiplexing.
- **`Overlay::blocks_agent` is independent of `modal`** — `modal` scopes input (focus + Esc-dismiss + Tab cycling); `blocks_agent` gates engine-event drain. `/help` and `/btw` are modal but never `blocks_agent`; confirm.lua is both. `focused_float_blocks_agent` checks `focused_overlay`'s `blocks_agent` field before falling back to the legacy `FloatConfig` check.
- **List leaves fire `SelectionChanged` on cursor move** — confirm.lua needs to know the option panel's selection at submit time even when Submit fires from the reason input leaf. The recipe now returns `CallbackResult::Event(SelectionChanged, Selection { index })` from `move_cursor` so subscribers track the live selection. Cursor-no-op moves (already at first/last row) stay `Consumed`.
- **C.9c is split into C.9c.1 / C.9c.2 / C.9c.3 / C.9c.4 / C.9c.5** — the original C.9c.2 ("migrate cmdline + notification + picker dropdown + delete the legacy float layer") was bigger than fits one session. C.9c.1 lands the panel-multiplexing demolition; C.9c.2 migrates notification only; C.9c.3 takes cmdline; C.9c.4 takes the picker dropdown; C.9c.5 deletes `FloatConfig` / `Placement` / `WinConfig::Float` / the legacy float compositor layer + retires `Component` / `WidgetEvent` once their last consumers move.
- **`Anchor::Win` gains `row_offset` / `col_offset`** mirroring `Anchor::Cursor` — needed so the notification can attach to the prompt's NW corner and shift up one row without each frame re-computing a `ScreenAt` rect. Generic addition: any future overlay that wants "above / beside the target by N rows" composes via the offset.
- **Notification renders as Buffer extmarks, not a `Component`** — one line, two highlight extmarks (label `Bold` + optional theme `ErrorMsg.fg`; message `Dim`). The `Notification` / `NotificationLevel` / `NotificationStyle` types die; `is_error: bool` is the only level distinction the new path needs.
- **`PanelSpec.pad_left` and per-panel `separator` were already dead under the overlay path** (`open_dialog_via_overlay` parsed them but never applied). C.9c.1 drops both. The `pad_left = 2` in `agents.lua` and `separator = "dashed"` in `confirm.lua` are silently ignored — same behaviour as before C.9c.1, just honest about it. Visual restoration belongs to P1.d (Window-level pad) + a per-leaf separator on the LayoutTree's vbox items (overlay surface feature).
- **Slim `dialog.rs` keeps `PanelSpec` / `PanelHeight` only** — pure data types consumed by `tui::lua::ui_ops::open_dialog_via_overlay`. `PanelContent` enum collapses to a `BufId` field on `PanelSpec` (no widget variant left). The file rename to `panel.rs` (or fold into `ui_ops.rs`) is deferred to P1.f when the `Ui` facade rewrite touches the public surface anyway.
- **`WidgetEvent::{Submit, Cancel, SelectDefault}` retire** — no remaining emitters after the legacy widget files (`TextInput`, `OptionList`, `Dialog`) deleted. `Submit` was emitted only by them; `Cancel` only by `TextInput`; `SelectDefault` was never emitted. The remaining variants (`Dismiss`, `Select(idx)`, `SubmitText`, `TextChanged`) are still needed by `Cmdline` and `Picker`.
- **Cmdline keeps a literal `:` prefix in the buffer** — alternative was a 2-leaf Hbox (one cell content leaf for `:` + input leaf), or a virtual-text extmark prefix (not yet a thing on `Buffer`). Keeping `:` as line-0 byte 0 with a cursor clamp to `>= PREFIX_LEN` is the smallest representation, and submit just `strip_prefix`s the literal before dispatch. Bonus: a future `widgets/cmdline.lua` recipe inherits the same shape.
- **No leaf-level callbacks for the cmdline; `App::cmdline_handle_key` runs ahead of compositor dispatch** — Rust callbacks only see `&mut Ui` via `CallbackCtx`, but every cmdline action (submit running a command, history push, completer rebuild, Quit propagation) needs `&mut App`. `with_app` from inside a callback would alias the live `&mut self.ui` borrow. Pre-intercepting in `events.rs` keeps the existing safe `&mut self` pattern from the legacy widget; the leaf carries no recipe so unmatched keys silently swallow (safer than the legacy widget's `KeyResult::Ignored` fallback that allowed split keymaps to fire on top of an open cmdline).
- **Compositor `render_with` closure returns `Option<(u16, u16)>` instead of a separate `cursor_override` parameter** — the closure already runs after layer paint and before cursor placement; threading the override through its return value keeps the call sites symmetric (`Ui::render` computes overlay leaf cursor in the same scope it computes overlay rects). Overlay > layer takes precedence so the focused overlay leaf's caret wins over an unfocused layer's `Component::cursor()`.
- **`paint_overlay` clears the overlay rect before painting** — overlays are conceptually opaque (modal input claims the surface), so leaking lower-layer cells through gap rows or short buffer lines is a bug. `grid.clear(area)` once at the top of `paint_overlay` covers every container shape (vbox / hbox, with or without chrome, gap > 0, narrow buffer lines). Existing dialog overlays were never short enough to expose the bug; the cmdline's single-glyph `:` line surfaced it immediately.
- **Cmdline pre-intercept also gates global chord layer** — `events.rs` previously gated Shift+Tab / Ctrl+T / Ctrl+L on `focused_float().is_none()` so global chords didn't fire while a focused float owned input. With cmdline migrating to overlay, `focused_float()` becomes `None` even with cmdline open. Targeted fix: add `&& self.cmdline_win.is_none()` to the existing gate. Broadening to `active_modal().is_none()` (which would also cover confirm / `/help` / `/btw`) is a behaviour change for those dialogs and stays out of scope for C.9c.3.
- **Picker module lives in `tui`, not `ui`** — the new `crates/tui/src/picker.rs` is a thin compose-helper over generic primitives (`Buffer`, `cursor_line_highlight`, overlay anchor). It carries no Component, no per-pixel paint logic; everything is buffer + extmark + cursor. So it sits at the same layer as the cmdline / notification overlay-builders in `app/`, not as a primitive in `ui`. P3.b moves the recipe behind `widgets/picker.lua`; until then `tui::picker` is the call surface.
- **Drop the `&& ctx.focused` gate on `cursor_line_highlight`** — the prompt-docked completer picker is non-focusable (keys flow to the prompt) but still needs to show the selected row. Selection paint regardless of focus also gives mixed dialogs a free win: a list panel + input panel dialog (e.g. confirm with reason input) now keeps the option highlight visible while focus is in the input. Existing list panels that opted in (resume, agents, options) keep their behaviour because they're focused at open. The only case that materially changed is "list panel never receives focus" — desired everywhere it occurs.
- **Legacy float-resolution tests delete with the picker Component** — `lib.rs::open_stub_float` was the only float-opener in the ui crate; with `Picker` gone there's no Component left to instantiate. The 7 tests it powered (default config / manual placement / SE anchor / clamping / dock-bottom-full-width / z-ordering / resolve-rects) cover machinery that retires in C.9c.5 anyway. Re-adding a fake Component to keep them alive across one phase added churn for no gain; coverage of the surviving overlay paths already lives next door.
- **`Ui::natural_layer_height` reduces to a stub** — the picker was the only float kind that exposed a natural size; with it gone the function has nothing to peek. Kept as an `_win_id`-prefixed stub returning `None` so `resolve_float_rects` callers compile; the function deletes alongside `FloatConfig` in C.9c.5.
- **`WinConfig` collapses to `SplitConfig`** — only the `Split` variant survived after C.9c.4. Inlining the field as `pub config: SplitConfig` removes the only-one-variant enum noise; `Window::is_float` / `is_split` / `zindex` retire with it (the matches no longer differentiate anything).
- **Mouse modal gate moves to `overlay_hit_test`** — `float_at` returned the topmost float whose rect covered the pointer; replacing it with `Ui::overlay_hit_test(row, col, None)` filtered by `active_modal()` keeps the same "absorb clicks outside the modal's rect" behaviour without the float layer. Non-modal overlays don't gate clicks (parity with the legacy "no focused float" path).
- **Tui renames** — `focused_float_blocks_agent` → `focused_overlay_blocks_agent`, `close_focused_non_blocking_float` → `close_focused_non_blocking_overlay`, `close_float` → `close_overlay_leaf`, `sync_completer_float` → `sync_completer_overlay`. "float" leaving the lexicon is part of the deletion — leaving the names alone would have shipped phase-stale identifiers (the README "code stays phase-agnostic" rule).
- **`Component` / remaining `WidgetEvent` variants stay alive** — the original C.9c.5 plan paired their retirement with this sub-phase, but the StatusBar / BufferView / WindowView consumers still need them. Move the `Component` + `WidgetEvent::{Dismiss, Select}` deletion forward to the P1.d Window rewrite where those consumers actually migrate.
- **P1.d splits into six sub-phases** — D.1 (status) / D.2 (prompt) / D.3 (transcript) / D.4 (Component + WidgetEvent retirement) / D.5 (vim decompose) / D.6 (completer decompose). The single P1.d sketch in REFACTOR.md was too big for one session (3 Component impls, vim 3000+ LOC, completer state machine all touched at once). Splitting along consumer boundaries keeps each session at one-architectural-move size.
- **Status line goes first because it's the smallest wedge** — 1-line buffer, no cursor, no selection, no keyboard, no mouse. Lets the `painted_splits` mechanism land in isolation; D.2/D.3 reuse it for prompt/transcript.
- **`painted_splits` is a `Vec<WinId>`, not a flag on `Window`** — keeping the registry on `Ui` (next to `splits` / `split_rects`) means the painting decision is host-policy, not Window-state. Insertion order is paint order; rect comes from `split_rects` (set per frame via `set_window_rect`). No new `Window` field, no new `WinConfig` variant.
- **Painted splits paint *before* overlays** — preserves the prior z order: status was a compositor layer at z=500; overlays were painted in `render_with`'s closure (after the compositor's layer paint, including z=500). With status now in the closure too, painting splits first → overlays second matches "compositor layers + status, then overlays on top".
- **Status-line spans cover every cell with exactly one span** — `Buffer::highlights_at` iterates a HashMap so span paint order is unstable (`Window::render` only calls `slice.set` once per span; last span wins per cell). Layering "bg fill + segment fg" via two overlapping spans would be racy. Solution: emit non-overlapping spans (segment + gap fillers) at line build time, each with the full merged `Style` (fg + bg + attrs). Bg fallback (`status_bg`) folds into each segment's `Style::bg` via `style_to_grid`; gaps get a fill-only span.
- **Status buffer is `BufType::Nofile`, modifiable** — non-modifiable would block `refresh_status_bar` from rewriting via `set_all_lines`. Marking the Window non-focusable (`win.focusable = false`) keeps Tab cycling out of the status row, which is the focus-control end of the same invariant.
- **Status line stays in Rust, Lua migration is P4.c** — the Rust composer (mode pill, vim mode, throbber, permissions/proc/agent badges, position) generates the per-frame spans. P4.c's job is converting those spans to a Lua-driven cell-binding spec. P1.d.1 only changes *how* the spans land on the screen (Buffer extmarks vs `StatusBar` Component), not *what* drives them.
- **P1.d.2 splits into D.2a / D.2b** — the rendering primitive (Window scrollbar paint + cursor surface + painted-split focus on Ui) needs to land before the prompt migration can flip both compositor layers in one move. D.2a ships the primitive; D.2b drops `"prompt"` + `"prompt_input"` and projects chrome + input into a unified Buffer. Splitting along the "primitive vs migration" seam keeps each session at one-architectural-move size; the prompt migration also unblocks D.3 (transcript) which reuses the same primitive.
- **`CursorKind` is a kind enum, position is in `Window`** — the focused-overlay-leaf path already reads `(cursor_col, cursor_line)` straight off the focused leaf's Window. Mirroring that for painted splits keeps cursor coordinates as Window state (one place to mutate) and CursorKind purely about *how* to paint. Two variants suffice: `Hardware` (terminal caret, surfaced through `render_with`'s closure return) and `Block { glyph, style }` (paint cell in `Window::render`, no caret). Soft-cursor cases (vim Normal cell-style) all reduce to `Block`; hardware-caret cases (Insert mode, dialog inputs, cmdline) all reduce to `Hardware`.
- **`viewport: Option<WindowViewport>` lives on `Window`, not pushed in via render args** — overlay leaves and painted splits both need scrollbars surfaced from the same render path; threading viewport through render's signature would force every caller (overlay paint, painted-split paint, future tests) to compute and pass it. Storing on Window matches `cursor_kind`'s call-site shape — hosts repopulate per frame.
- **`painted_split_focus` is a third focus slot** — overlay_focus + compositor focus already make focus a layered read; adding `painted_split_focus` continues the pattern. Alternative was unifying all three into a single `Ui::focus: Option<WinId>` and dropping compositor-focus entirely, but that ripples through every compositor layer (transcript still uses one until D.3) and isn't necessary for D.2a's scope. The unification belongs to P1.f's facade rewrite.
- **Cursor priority is overlay > painted-split > compositor-layer** — overlays are modal-on-top in every other lookup (focus cycling, hit testing, modal Esc), so the cursor follows. Painted splits are below overlays but above the residual compositor layers (transcript today; nothing after D.3). The compositor's focused-layer cursor path stays alive until D.3 retires the last `Component` implementor.
- **`Buffer::highlights_at` walks every namespace, not just `ns_highlights`** — the previous "convenience getter only sees `NS_HIGHLIGHTS`" comment treated highlights as if they were namespace-private. The diagram says otherwise: "All decoration is extmarks in named namespaces. Highlights, virtual text, conceals, signs, source-byte mapping for markdown/diff, click metadata, line backgrounds, folds, diagnostics — one store". A parser registering its own `"diff"` / `"syntax"` / `"selection"` namespace should have its highlights paint on the same pass. Cross-namespace ordering is `NsId` ascending so registration order is the paint order — the transcript registers `NS_SELECTION` after the well-known `ns_highlights` (id 0), so its spans paint on top of projection spans on the same cell. Within a namespace, BTreeMap iteration by extmark id keeps insertion order stable.
- **`TranscriptProjection` borrows the buffer instead of owning it** — moving the buffer into `Ui::bufs` is the whole point of the painted-split migration: `Window::render` reads `Ui::bufs[win.buf]`, so any owned-elsewhere buffer would force a per-frame mirror copy. `project()` takes `&mut Buffer` and writes into it; the `(generation, width, show_thinking)` cache markers stay on the projection struct.
- **Selection is a dedicated `NS_SELECTION` namespace** — the alternative was reusing the well-known `ns_highlights`. Two problems: `apply_to_buffer` calls `set_all_lines` which clears `ns_highlights` (so selection extmarks added before projection would die on the next project); and project() short-circuits on cached generations (so selection extmarks added after a no-op project would survive across frames without an explicit clear). A dedicated namespace solves both — `set_all_lines` doesn't touch custom namespaces, and we can `clear_namespace(ns_selection, …)` each frame regardless of whether project ran.
- **Soft-cursor compute stays gated on `transcript_owns_cursor`** — the old WindowView path skipped `set_soft_cursor` when content didn't own the cursor. The new path mirrors that: `tcursor.soft_cursor` is `None` when content isn't focused, and `cursor_kind` collapses to `None` accordingly so `Window::render`'s block-cursor cell paint short-circuits. No "always paint" regression.
- **`SoftCursor` moves to `app/transcript.rs`, not promoted to `ui`** — its only producer is `compute_transcript_cursor`; its only consumer is `sync_transcript_layer`, which now constructs `CursorKind::Block` from it directly. `ui::CursorKind` is the public seam — `SoftCursor` is a tui-internal helper so colocating it with the producer keeps the boundary clean.
- **`content/scrollbar.rs` deletes outright instead of porting tests to `ui::window`** — the `Scrollbar` struct's only consumer was `WindowView::draw_scrollbar`. Painted-split scrollbar math lives on `ui::ScrollbarState` (`max_scroll` / `max_thumb_top` / `scroll_from_top_for_thumb` / `thumb_top_for_click` / `is_thumb_at`), exercised by integration tests in `crates/ui/src/window.rs` (`render_paints_scrollbar_thumb_at_top_for_scroll_zero`, `render_skips_scrollbar_when_no_overflow`). The standalone roundtrip test that linked `Scrollbar::new` ↔ `ScrollbarState` was redundant once both paint paths went through `paint_scrollbar`.
- **`BufferView`-only Buffer cache deletes with `BufferView`** — `cached_highlights` / `cached_decorations` / `cache_tick` + `highlights_arc` / `decorations_arc` / `lines_arc` accessors served `BufferView::sync_from_buffer`'s Arc-clone fast path. With painted-split rendering reading `buf.highlights_at(idx)` per row directly, no caller needs the materialized cache. Removing it shrinks `Buffer` and removes a possible source of stale-cache bugs.
- **P1.d.5 splits into 5a/5b/5c/5d/5e/5f** — original 4-way split underestimated the EditBuffer→Buffer merge; 5c re-narrows to "persistent per-Window vim state → Window," 5d takes registers/dot-repeat/undo + `edit_buffer` merge, 5e drops the in-flight key state, 5f closes with cursor/selection consolidation + `vim/` deletion.
- **VimMode threads through helper params, not cached on PromptState** — single source of truth (`App.vim_mode`); borrow checker enforces "mode you read = mode you used."
- **`Vim::set_mode` / `begin_visual` / `MouseCtx.vim_mode` take `&mut VimMode`** — write-through keeps "VimMode lives on App" honest without copy-back-on-return.
- **`smelt.vim.mode` returns Debug-format string** — matches existing `smelt.win.mode` shape; Lua pattern-matches on `"Insert"` / `"Normal"` / `"Visual"` / `"VisualLine"`.
- **In-flight + persistent vim state share `VimWindowState`** — original 5c split kept them apart by lifetime, but both are per-Window data and only vim mutates either; coalescing avoids `&mut self.vim` + `&mut self.vim_state` co-borrow gymnastics in callers. The doc comment on the struct now describes both roles. `Vim` stays as the ZST enable marker so `Option<Vim>` keeps meaning "vim enabled here" without spawning a new bool field per Window.
- **`motions` and `text_objects` lift to top-level `crate::*` primitives, not folded into `text.rs`** — `text.rs` already houses pure text-motion helpers (`prev_char_boundary`, `line_start`, `word_forward_pos`, `vertical_move`); folding 332 + 158 LOC of vim-shaped motions and text-object selection into it would balloon the file past readability. Two siblings (`motions.rs` for cursor steps, `text_objects.rs` for object-shape selection) is the natural carve. They share `text.rs` for char-boundary / line-boundary / `char_class` primitives, so duplication still gets removed (the `prev_char_boundary` / `next_char_boundary` copies in motions deleted in favour of `text::*`).
- **`FindKind` moves with `motions`, not `vim`** — it's the parameter type for `motions::find_char` and parametrises a motion behaviour; the only reason it lived in `vim/mod.rs` was that motions was a private vim submodule. Now that motions is a top-level primitive, FindKind belongs next to its primary consumer. `lib.rs` re-exports it from `motions` (was `vim`); external callers see no change because the re-export path stays.
- **5f.2c lands as directory collapse only; "flatten dispatcher to recipe-style registrations" splits into a new 5f.2d** — the original 5f.2c plan paired both, but recipes are documented as Lua throughout `ARCHITECTURE.md` ("Recipes are pure Lua, defined in `runtime/lua/smelt/widgets/`") and the keymap-recipe registry lands with P3.b/P4. Building a Rust-side flat dispatch table now would either (a) be throwaway scaffolding the Lua registry replaces, or (b) entrench a Rust-side shape that diverges from the Lua surface. Neither is acceptable under the no-scaffolding rule. The directory collapse is independent of the recipe shape — `vim.rs` stays a single-file dispatcher with a giant match until 5f.2d hooks it into the (Lua) recipe registry.
- **Tui-side imports route to `ui::vim::*` / `ui::VimMode`, not a re-export under `tui::`** — the 1-line `crates/tui/src/vim/mod.rs` shim re-exporting four items from `ui::vim` was pure indirection. Deleting it means callers either say `ui::VimMode` (already re-exported at `ui` crate root from line 57 of `ui/lib.rs`) or `ui::vim::{handle_key, Action, VimContext}` (the rarely-used dispatcher entry points). One less hop, one less file, no behaviour change.
