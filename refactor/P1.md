# P1 — UI primitives

**Status:** in-progress **Started:** 2026-04-29 **Landed:** —

P1 rebuilds `ui` around the target primitives (`Buffer`, `LayoutTree`,
`Overlay`, `Window`, `Theme`, `UiHost`, `Ui` facade) and absorbs the
structural deletions P0 deferred. Sub-phases land independently, each
green at the boundary. Detail per sub-phase lives in the commit
message; this file is the index.

P1.f rewrites `Ui` to its target shape, splitting along field-by-field

## Decisions made

One bullet per non-obvious choice — full reasoning lives in the
commit message. If you'd re-litigate a choice in `P3+`, write it here.

- **Theme registry owns is_light/accent/slug as fields, not atomics.** Single owner via `Ui`.
- **Syntect light/dark hint stays a module-local atomic.** Threading `&Theme` through 14 sites was disproportionate.
- **`YankSubst` / `wrap_at` shipped without consumers.** Real consumers come with the transcript migration.
- **Builder constructors over struct-literal field-defaults (B.2).** `Direction` had no Default; builders became Vbox/Hbox.
- **Single `SeparatorStyle` (in `layout`).** Dialog's same-named enum deleted.
- **C.5 stops at design note before code ships.** Pulled `Window::render` forward instead of a Component-on-Overlay shim.
- **List/input keymap recipes live in Rust until P3.b.** Rust dispatch is fundamental; recipes go Lua when the registry takes shape.
- **`Anchor::ScreenBottom { above_rows }` over a `dock_bottom_full_width` builder.** Anchor positions, layout sizes.
- **`Overlay::blocks_agent` independent of `modal`.** Modal scopes input; blocks_agent gates engine drain.
- **List leaves fire `SelectionChanged` on cursor move.** confirm.lua needs live selection across leaf transitions.
- **C.9c splits into 5 sessions.** Original C.9c.2 was bigger than fits one session.
- **`Anchor::Win` gains `row_offset` / `col_offset`.** Mirrors `Anchor::Cursor`; lets notification attach above prompt.
- **Notification renders as Buffer extmarks.** `Notification` widget types die; `is_error: bool` is the only knob.
- **`PanelSpec.pad_left` + per-panel `separator` were already dead.** C.9c.1 drops them; visual restoration belongs to P1.d.
- **`WidgetEvent::{Submit, Cancel, SelectDefault}` retire.** No remaining emitters after legacy widgets deleted.
- **Cmdline keeps a literal `:` prefix in the buffer.** Cursor clamps to `>= PREFIX_LEN`; submit strips it.
- **No leaf-level callbacks for cmdline.** `App::cmdline_handle_key` runs ahead of compositor dispatch (avoids `&mut Ui` borrow conflict).
- **`render_with` closure returns `Option<(u16, u16)>` cursor override.** Overlay > layer priority; symmetric with overlay rect compute.
- **`paint_overlay` clears its rect first.** Overlays are conceptually opaque; cmdline's single-glyph `:` surfaced the leak bug.
- **Cmdline pre-intercept also gates global chord layer.** `cmdline_win.is_none()` guard; broadening to `active_modal()` would change confirm/help/btw behaviour.
- **Picker module lives in `tui`, not `ui`.** Thin compose-helper over generic primitives; P3.b moves it to `widgets/picker.lua`.
- **Drop `&& ctx.focused` gate on `cursor_line_highlight`.** Prompt-docked completer is non-focusable but needs selection paint.
- **Legacy float-resolution tests delete with the picker Component.** Coverage moves to overlay tests next door.
- **`WinConfig` collapses to `SplitConfig`.** Only one variant survived after C.9c.4.
- **Mouse modal gate moves to `overlay_hit_test`.** Filtered by `active_modal()`; non-modal overlays don't gate clicks.
- **Tui float→overlay rename pass.** "float" leaving the lexicon part of the deletion.
- **`Component` retirement defers to P1.d.** StatusBar / BufferView / WindowView consumers still need it through C.9c.5.
- **P1.d splits into six sub-phases.** D.1/D.2/D.3 (one Component impl each) / D.4 (trait retirement) / D.5 (vim decompose) / D.6 (completer).
- **Status line goes first in D.1.** Smallest wedge; `painted_splits` lands in isolation.
- **`painted_splits` is a `Vec<WinId>` registry on `Ui`.** Painting decision is host-policy, not Window-state.
- **Painted splits paint *before* overlays.** Preserves prior z-order (overlays on top of compositor layers + status).
- **Status-line spans cover every cell with exactly one span.** `highlights_at` paint order otherwise unstable.
- **Status buffer is `Nofile`, modifiable.** Window non-focusable to keep Tab cycling out of status row.
- **Status line stays in Rust; P4.c migrates to Lua spec.** D.1 only changes paint path, not what drives spans.
- **P1.d.2 splits into D.2a / D.2b.** Rendering primitive lands before prompt migration.
- **`CursorKind` is a kind enum; position lives on `Window`.** Two variants: `Hardware` (caret) and `Block { glyph, style }` (cell).
- **`viewport: Option<WindowViewport>` on `Window`.** Hosts repopulate per frame; symmetric with `cursor_kind`.
- **`painted_split_focus` lands as a third focus slot in D.2a.** Unification waits for P1.f.1.
- **Cursor priority overlay > painted-split > compositor-layer.** Mirrors modal-on-top in every other lookup.
- **`Buffer::highlights_at` walks every namespace.** Cross-namespace ordering is `NsId` ascending; transcript selection registers `NS_SELECTION` after `ns_highlights`.
- **`TranscriptProjection` borrows the buffer.** Owning would force per-frame mirror copy.
- **Selection is a dedicated `NS_SELECTION` namespace.** Avoids `set_all_lines` clearing extmarks + project's cache short-circuit.
- **Soft-cursor compute gated on `transcript_owns_cursor`.** Mirrors prior WindowView behaviour.
- **`SoftCursor` stays in `app/transcript.rs`.** Tui-internal helper; `ui::CursorKind` is the public seam.
- **`content/scrollbar.rs` deletes outright.** ScrollbarState math integration-tested in `ui::window` already.
- **`BufferView`-only Buffer cache deletes with `BufferView`.** Painted-split rendering reads `highlights_at` per row directly.
- **P1.d.5 splits into 5a/5b/5c/5d/5e/5f.** Original 4-way split underestimated EditBuffer→Buffer merge.
- **VimMode threads through helper params.** Single source of truth; borrow checker enforces mode-you-read = mode-you-used.
- **`smelt.vim.mode` returns Debug-format string.** Matches `smelt.win.mode` shape.
- **In-flight + persistent vim state share `VimWindowState`.** Both per-Window; coalescing avoids co-borrow gymnastics.
- **`motions` and `text_objects` lift to top-level primitives, not folded into `text.rs`.** Two siblings keep `text.rs` from ballooning.
- **`FindKind` moves with `motions`.** Parametrises `find_char`; lived in vim only because motions was a private vim submodule.
- **5f.2c lands as directory collapse only.** Recipe-style flatten splits to 5f.2d (gated on Lua keymap registry).
- **Tui imports route to `ui::vim::*` / `ui::VimMode` directly.** 1-line shim deleted; one less hop.
- **P1.f splits into F.1..F.6.** One field-by-field architectural move per session.
- **F.6 splits into F.6a / F.6b.** F.6a frees the `dispatch_event` slot via mechanical rename to `fire_win_event` (matches `UiHost::fire_win_event` from ARCHITECTURE.md); F.6b introduces `Ui::dispatch_event(Event)` over the unified terminal `Event` and folds key + mouse + resize + focus + blur fanout into it. Splitting because mouse routing alone is ~455 LOC of soft-wrap translation + scrollbar drag + click-count tracking that doesn't fit alongside a rename.
- **`focus()` method and `focus` field share a name.** Diagram specifies both as `focus`; parens disambiguate at call site.
- **F.2 keeps stable `OverlayId(u32)` over Vec-index handles.** Vec-index would invalidate IDs on close; live workloads carry 0–3 overlays.
- **F.4 captures via `Ui::set_capture(HitTarget)`, not via App-side scrollbar enum.** `HitTarget::Scrollbar { owner }` carries the WinId; tui maps owner directly to `PROMPT_WIN` / `TRANSCRIPT_WIN`.
- **F.4 auto-clears on overlay close when target was inside the overlay.** Gesture started in the closing overlay terminates with it; capture survives only when the captured target's owner is still alive elsewhere.
- **F.5 `CursorShape::Hidden` is a real variant, not `Option<CursorShape>`.** Default-constructible enum keeps `DrawContext::cursor_shape` field-shaped without an extra layer.
- **F.5 paint sites pass `(focus, cursor_shape)` rather than reading `Ui` from inside the closure.** `Ui::render`'s post-layer closure already snapshots `wins`/`bufs`/`term_size` to dodge the borrow; cursor state joins them.
- **F.5 `compute_cursor_ownership` suppresses prompt/transcript ownership on any focused overlay**, not just cmdline. The overlay-leaf default at end of `render_normal` then sets `Hardware`. Matches today's behaviour where `focused_overlay_cursor` always wins over `painted_split_cursor`.
- **F.6b `DispatchOutcome` replaces `KeyResult`.** `Consumed` / `Ignored` is now per-event (not per-key), so the same outcome carries the modal-Esc branch, the focused-window keymap branch, the resize branch, and the mouse pre-flight branch. Hosts read it once on every terminal event; per-event-kind enums would have multiplied without changing semantics.
- **F.6b mouse fold is a pre-flight, not a full fold.** Pre-P2, mouse routing carries heavy App-state coupling (prompt soft-wrap translation, transcript display rows, click-count tracking, scrollbar drag bookkeeping). `Ui::dispatch_event` claims the two Ui-shaped checks (wheel-on-overlay absorb, active-modal click-outside absorb) and returns `Ignored` for the rest; tui's `handle_mouse` keeps the App-state work. The full fold lands when P2 introduces the `Host` / `UiHost` traits.
- **F.6b resize fold makes `set_terminal_size` an internal effect of `Event::Resize`.** Tui's `handle_resize` no longer calls it directly — it dispatches the event and keeps its own `last_width` / `last_height` cache + `invalidate_for_width` post-work. App startup + the layout adapter still call `set_terminal_size` directly because they aren't event-driven.
- **F.6b dispatcher is `non_exhaustive`-safe.** The match's catch-all arm covers `FocusGained` / `FocusLost` / `Paste` plus any future crossterm variants — Ui has no state to update for any of them today; tui tracks `term_focused` and reads paste content App-side.
- **D.6 splits across sessions.** D.6a (virt-text painting primitive) is independent and lands now; D.6b (ghost-text storage → `"completer"` extmark on prompt buffer) follows; D.6c (picker sync) / D.6d (behaviour → keymap recipe) / D.6e (`completer/`+`attachment/` collapse) are gated on the Lua keymap registry from P3.b/P4.
- **D.6a `virtual_text_at` walks every namespace.** Same precedent as `highlights_at`: cross-namespace ordering is `NsId` ascending, within a namespace it's BTreeMap (extmark-id) order. Lets a `"completer"` namespace coexist with `ns_virt_text` (the convenience namespace `set_virtual_text` writes to).
- **D.6a primitive lands without consumer.** Same precedent as `YankSubst` / `wrap_at` in P1.a-2/3 — the consumer migration arrives in D.6b.
- **D.6a virt-text style merges `row_style` first via `merge_styles`.** Cursor-row highlight bg shows through when the extmark's resolved style omits bg, matching `merge_span_style`'s OR-the-attrs / fall-through-the-colors rule for `Style` / `Style` (vs `Style` / `SpanStyle`).
- **D.6b storage IS the extmark.** `App::input_prediction: Option<String>` retires entirely; the only ghost-text storage is the virt-text extmark in the `"completer"` namespace on the prompt buffer. `compute_prompt` reads-and-re-anchors it each frame because chrome rows can shift the input-row line index (queued grows, stash appears) and `set_all_lines` invalidates the previous anchor. `App::{set,clear,take}_prompt_completer` are the only mutation paths.
- **D.6b parks invisible storage at `total_lines`** (one past last). `Window::render` only walks `0..line_count` for `virtual_text_at`, so the extmark survives without rendering when the buffer is non-empty (typing usually clears it via `events.rs`, but the off-buffer park is the safety net for paths that don't). A custom-payload variant or sentinel namespace would be cleaner architecturally but adds enum surface for a one-off case; the off-buffer park is local to `compute_prompt`.
- **D.6b drops the render_loop prediction-suppression on `show_queued`.** Today's render_loop forces `prediction = None` during agent runs / compaction; with the extmark-as-storage model, `compute_prompt`'s "anchor visibly when `state.buf.is_empty()`" rule subsumes the agent-running case (begin_turn clears the extmark, so it's not present anyway). Compaction without a turn could surface the prediction visually, but predict.lua only re-fires on `turn_end` so the user sees the previous prediction continue — the trivial behavioural difference doesn't justify a `show_queued: bool` flag on `PromptInput`.
- **D.6b adds `"GhostText"` theme group** (dim-only, no fg/bg). Resolves through `populate_ui_theme`; lookups inside the buffer's `set_extmark` payload reference it by name so theme swaps propagate without touching the extmark.
- **D.6b leaves `EngineEvent::InputPrediction` and `App::predict_generation` alone.** Those are unused dead code (predict.lua flows through `engine.ask`, not `PredictInput`); cleaning them is a separate de-cruft pass, not part of the storage migration.
- **F.7 retires `current_win` outright** rather than aliasing onto `focus`. Confirmed zero callers outside `lib.rs`; the auto-set-on-first-open semantics didn't translate to focus (which is explicit and modal-aware), so re-pointing accessors at focus would have silently changed behaviour. Cleaner to delete and have callers use `set_focus` when they want focus.
