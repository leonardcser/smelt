## Scope

### 1. Saved-state cleanup + injectable constructor (landed)

Path A landed alongside Thread 2 in one edit.

**Thread 1: drop the doc lie.** `crates/core/src/state.rs` shrunk
from ~340 LOC to ~190 LOC. `PersistedSettings` (the 10-bool toggle
overlay) and the legacy field migration retired entirely; settings
no longer round-trip through JSON. `state::save_settings`,
`update_settings`'s persistence call, and the legacy
`vim_enabled` / top-level `show_thinking` migration paths all
deleted. `State` renamed to `SessionCache` with just the
genuinely cache-shaped fields: `mode`, `selected_model`,
`reasoning_effort`. `accent_color` dropped — themes are config,
they live in `init.lua` (the `/theme` and `/color` runtime pickers
are previewers; persistence comes from editing your config).

`ResolvedSettings` moved from `state.rs` to `core/config.rs`
alongside `SettingsConfig`, where it conceptually belongs.
`SettingsConfig::resolve(&self) -> ResolvedSettings` collapses the
two-phase merge into one direct call (no more
`PersistedSettings::resolve(&SettingsConfig)`).

**Thread 2: `TuiApp::new` is state-injectable.** Constructor now
takes `cache: SessionCache` as its final parameter; the internal
`state::State::load()` call is gone. `main.rs` calls
`SessionCache::load()` once via `startup::resolve` and threads it
through `ResolvedStartup` to `TuiApp::new`. Constructor is pure
modulo the `Ui::new() + terminal_size` setup — no filesystem
touch, no global state read.

Net: `state.rs` shrank ≥ 50%; the doc claim "all config in
`init.lua`" is true at the file level; `TuiApp::new` is
disk-clean and ready for L3 storybook fixture construction.

## Decisions made

- **Path A — drop persisted boolean settings entirely.** `init.lua`
  is the persistence story; runtime toggles (`/vim`, `smelt.settings.vim
  = true`) are transient by design. Config in init.lua is the user-facing
  contract.
- **Drop `accent_color` from the saved cache.** Themes are config —
  they live in init.lua. The `/theme` / `/color` pickers preview
  live; users wanting persistence edit their config. Same model
  Neovim uses for `:colorscheme`.
- **`ResolvedSettings` lives in `core::config`, not `core::state`.**
  Resolves directly from `SettingsConfig` with built-in defaults; no
  persistence layer between them.
