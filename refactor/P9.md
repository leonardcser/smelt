# P9 ‚Äî Make the architecture true

**Status:** in-progress **Started:** 2026-05-04 **Landed:** ‚Äî

P9 is the architectural-truth phase. Three sub-phases land the deferrals
that earlier phases dropped, plus the audit findings showing where Rust
still encodes Lua-shaped decisions. After P9, the code matches the rule
*"feature in Rust that could live in Lua is a bug."* Hygiene + parity
walk live in P10.

The earlier P9 decomposition (a‚Äìg, seven sub-phases for ~one migration)
is collapsed. Already-landed sub-phases stay logged below; remaining
work folds into three fat phases sequenced by dependency.

## Decisions made

- **Well-known window IDs out of `ui`** (`352de075`, `2026-05-04`) ‚Äî
  `PROMPT_WIN`, `TRANSCRIPT_WIN`, `PROMPT_EDIT_BUF` live in `app/well_known.rs`;
  `win_open_split` is collision-tolerant; `next_win_id` starts at `0`.
  Renamed `prompt_data.rs` ‚Üí `prompt_buf.rs` in the same change.
- **`buffer::SpanStyle` unified with `grid::Style`** (`de78d750`,
  `2026-05-04`) ‚Äî fixes latent strikethrough rendering bug; removes ~50 LOC
  of mechanical conversion. `merge_span_style` OR-merges `underline` and
  `crossedout` from extmarks.
- **Persisted transcript caches deleted** (`2125f6e0`, `2026-05-04`) ‚Äî
  `transcript_cache.rs` (`PersistedLayoutCache`, `RenderCache`,
  `ToolOutputRenderCache`) gone. Per-block caching is `Buffer::ensure_rendered_at(width)`
  once the pipeline migrates.
- **Lua `render` hook + `RenderCtx`** (`0a2bf51f`, `2026-05-04`) ‚Äî tool
  output rendering moved to a Lua callback that receives `RenderCtx`.
  *(Refined in P9.b: `RenderCtx` retires in favor of `smelt.buf.*` writing
  directly into a `Buffer` exposed across the crate boundary.)*
- **`BlockHistory::cache_dirty` deleted** (`b3ca0535`, `2026-05-04`) ‚Äî
  write-only dead field from the persisted-cache era.
- **`TranscriptSnapshot` moved to `tui`** (`b3ca0535`, `2026-05-04`) ‚Äî
  display projection is a tui concern; `core::content::transcript::Transcript`
  shrinks to `pub struct Transcript { pub history: BlockHistory }`.
  `BlockBufferCache` replaces direct `history.artifacts` reads with a
  per-block cache in tui. `BlockId::new` and `LayoutContext::new`
  constructors added for cross-crate use.
- **`Buffer` moves to `core`** *(P9.b, decided 2026-05-05)* ‚Äî see P9.b
  rationale below. Eliminates the `DisplayBlock` intermediate that survived
  every prior phase. Closes the deferral chain P1.a-tail ‚Üí P4.b ‚Üí old P9.b.
- **No more "too big to land" deferrals** *(2026-05-05)* ‚Äî earlier phases
  deferred the transcript pipeline three times. The replanning here folds
  it back in. Only legitimate reason to defer remains: a change that does
  not improve the codebase. Implementation size is not a reason.
- **`Buffer` + `BufferParser` + `Style` + `UndoHistory` landed in `core`**
  (`2026-05-05`) ‚Äî `core::buffer`, `core::style`, `core::undo` now host
  the data types. `BufId` and `LUA_BUF_ID_BASE` moved alongside `Buffer`.
  `tui::ui::buffer` / `undo` / `id` are thin re-export shims; no call
  sites changed.
- **`core::style::Color` is a frontend-neutral mirror, not a crossterm
  re-export** (`2026-05-05`) ‚Äî `core` has zero terminal deps. `Color`
  defines the same variant set as `crossterm::style::Color` (Reset, 16
  named ANSI slots, `AnsiValue(u8)`, `Rgb { r, g, b }`) but is owned by
  core. Tui converts at the SGR-emit boundary via
  `tui::ui::grid::to_crossterm_color` (a 16-arm match; the orphan rule
  blocks `From<core::Color> for crossterm::Color`). A future GUI
  frontend defines its own mapping for the named slots. The HlGroup-id
  follow-up (P9.e) replaces raw colors in extmarks entirely; this
  sub-step keeps the data shape neutral so that migration doesn't have
  to also disentangle a terminal dep.
- **`render_tool_body` exposes full `Buffer` userdata; `RenderCtx`
  retires** (`2026-05-05`) ‚Äî Lua tool `render` hooks now take
  `(args, output, width, buf_id)` and write into the buffer via
  `smelt.buf.*` plus the convenience renderers
  `smelt.{diff,syntax,bash,notebook,markdown,text}.render(buf, ‚Ä¶)`.
  Tui's `LuaRenderRenderer` mints an ephemeral scratch Buffer in
  `app.ui` per call, projects it back into the `SpanCollector`
  pipeline via `tui::content::to_buffer::buffer_into_collector`, and
  drops the buffer. The projector is the transient bridge that
  retires once per-block parsers also write into Buffer (P9.b step 3
  / P9.c). The architectural rule from ARCHITECTURE.md "Drawing
  context is full, not partial" now holds for tool render.

## P9.b ‚Äî `Buffer` to `core` (the keystone)

**Goal:** move the data types that *describe content* to `core`. Move the
data types that *describe paint* to `tui`. Today `Buffer` lives in
`tui::ui` even though it has zero terminal dependencies ‚Äî the result is
that `core`'s Lua render hook writes to a `DisplayBlock` intermediate,
`tui` projects that into a `Buffer`, and ~1500 LOC of bridge survives
every phase that tried to delete it.

### Move list

- **`crates/tui/src/ui/buffer.rs` ‚Üí `crates/core/src/buffer.rs`** (or
  `core::content::buffer`). Pure data move: lines, namespaces, extmarks,
  undo, soft-wrap cache, `BufferParser` trait. Re-export from
  `tui::ui::buffer` only as a transitional shim if needed; remove before
  P9.b closes.
- **`BufferParser` trait ‚Üí `core`.** Trait signature changes from
  `parse(&self, buf, source: &str, width)` to a closure-shaped form (or
  a generic over an associated input type) so block-variant parsers
  don't have to ignore `source` and stash the block on the struct.
- **`Theme` stays in `tui::ui`.** Buffer extmarks reference highlight ids
  (`HlGroup` newtype around a small int); resolution to `crossterm::Color`
  happens at paint time in `tui`. Cross-crate usage of `Buffer` doesn't
  need `Theme` to compile.
- **Parser *implementations* (markdown, diff, syntax, bash) stay in
  `tui::content`.** They pull from `tui::content::highlight::*` (syntect,
  inline markdown, diff LCS) which depends on tui-only crates. Tui
  registers them on Buffers via `Buffer::attach`.

### What this enables

- `core::lua::render_ctx` writes directly to `&mut Buffer`. `RenderCtx`
  collapses to whatever `smelt.buf.*` already exposes (lines + extmarks).
- `core::lua::runtime::render_tool_body` takes `&mut Buffer` instead of
  building a `DisplayBlock`.
- `transcript_present/` rendering glue deletes entirely.
- `SpanCollector`, `DisplayBlock`, `DisplayLine`, `layout_out.rs` delete.
- `BlockHistory::artifacts` deletes; per-block buffers live in tui's
  `BlockBufferCache` already introduced in 2026-05-04.
- Headless mode reads transcript content via `Buffer`, same surface as
  tui ‚Äî no parallel projection path.

### Order

1. ‚úÖ Move `Buffer` + `BufferParser` to `core` (2026-05-05).
2. ‚úÖ Migrate `render_tool_body` to hand Lua a `Buffer` userdata; delete
   `RenderCtx` (2026-05-05). Tui keeps a transient `Buffer ‚Üí
   SpanCollector` projector (`buffer_into_collector`) until step 3.
3. Delete `DisplayBlock`, `DisplayLine`, `SpanCollector`, `layout_out.rs`,
   the `transcript_present/` rendering glue.
4. P9.c starts from the new shape.

## P9.c ‚Äî Transcript pipeline as `BufferParser` impls

**Goal:** the path `Block ‚Üí BufferParser ‚Üí Buffer ‚Üí Window::render` works
end to end. This is the deferral that P1.a-tail / P4.b promised; with
`Buffer` in `core` (P9.b) it lands without a translator layer.

- **One parser per block variant** (`User`, `Thinking`, `Text`,
  `CodeLine`, `ToolCall`, `Exec`) under `tui::content::transcript_parsers/`.
  Each parser writes lines + extmarks into a per-block `Buffer`.
- **Width-independent parsing at ingest, width-dependent wrapping at
  paint.** Markdown AST, diff LCS, syntect tokens land as namespaced
  extmarks when the block first arrives. `Buffer::ensure_rendered_at(width)`
  re-flows on resize.
- **`TranscriptProjection` becomes a thin composition layer** ‚Äî
  concatenates per-block buffer lines, inserts gap rows, writes into the
  transcript display buffer. `BlockBufferCache` is the only cache.
- **User-bubble rendering deduplicates with prompt** ‚Äî `compute_prompt`'s
  queued-message rows call the same `UserBlockParser` the transcript uses.
- **`PromptInputParser` for the input area** ‚Äî replaces the three
  manual wrap call sites (`compute_prompt`, `PromptWrap::build`,
  `core/host.rs`). `PromptWrap` shrinks to a byte-map utility or deletes.
- **One `copy_range(buf, start, end)` primitive** ‚Äî replaces the eight
  divergent copy paths catalogued in `REFACTOR.md` (transcript cell-walk,
  prompt keybind, mouse yank, content vim yank, vim internal yank,
  kill-ring, mouse clipboard, whole-block yank). Each consumer passes
  its `Buffer` + byte range; no per-surface copy logic survives.
- **Responsive bar helper** ‚Äî `responsive_line(spans, width)` shared
  between `content/status.rs::spans_to_buffer_line` and
  `content/prompt_buf.rs::bar_row` (was old P9.f).

## P9.d ‚Äî De-Rustify Lua concerns

**Goal:** purge every place where Rust knows the name of a specific tool,
command, dialog, or mode. Audit (2026-05-05) found 11 instances; this
phase fixes them and forbids the pattern going forward.

### Tool registration grows callback fields

A Lua tool registers with a richer table:

```lua
smelt.tools.register({
  name = "bash",
  schema = { ‚Ä¶ },
  hooks = function(args, mode, ctx) ‚Ä¶ end,
  run = function(call_id, args, ctx) ‚Ä¶ end,
  summary = function(args) return args.description or args.command end,
  render = function(buf, args, output, width) ‚Ä¶ end,
  paths_for_workspace = function(args) return { args.command and ‚Ä¶  } end,
  elapsed_visible = true,
})
```

Rust calls these generically. No `match name { "bash" => ‚Ä¶ }` survives.

### Specific fixes

- **`engine/agent.rs:1179, 1443`** ‚Äî replace `if tc.function.name == "bash"`
  cmd_summary extraction with a generic `dispatcher.summary(name, args)`
  call routed into the Lua tool's `summary` callback.
- **`core/transcript_present/tools.rs`** ‚Äî gone. Tool block rendering
  becomes: Rust prints the pill + status; calls `tool.render(buf, args,
  output, width)`; that's it. No bash-specific multi-line wrap, no
  web_fetch prompt panel, no per-tool elapsed-time list.
- **`core/transcript_model.rs::ActiveTool::elapsed`** ‚Äî drop the
  hardcoded match; read `elapsed_visible` from the Lua tool def.
- **`core/permissions/workspace.rs::extract_tool_paths`** ‚Äî call
  `tool.paths_for_workspace(args)`. Drop the hardcoded `read_file |
  write_file | edit_file | glob | grep | bash` match.
- **`tui/lua/api/confirm.rs::render_title_into_buf`** ‚Äî delete.
  `runtime/lua/smelt/dialogs/confirm.lua` composes the title via
  `smelt.buf.add_highlight` + `smelt.bash.render` (a generic primitive
  Lua already calls). The "last Rust-side dialog primitive" disclaimer in
  the file goes with it.
- **`tui/lua/api/confirm.rs::_back_tab`** ‚Äî Rust steps reduce to
  `cycle_mode + decide`; the auto-allow heuristic moves into
  `dialogs/confirm.lua`.
- **`tui/lua/api/statusline.rs:180`** ‚Äî mode icons + labels read from a
  Lua-side table in `runtime/lua/smelt/modes.lua`; statusline source
  reads them via `smelt.cell` rather than Rust mapping `AgentMode` ‚Üí
  glyph.
- **`core/permissions/rules.rs:148`** ‚Äî Rust hardcodes which tools default
  to Allow in Apply / Yolo. The mode-default table moves to
  `modes.lua`; Rust calls `smelt.permissions.default_for(tool, mode)`
  generically. Custom modes work end-to-end.
- **`core/custom_commands.rs::CommandOverrides`** ‚Äî the typed
  `tools/bash/web_fetch` fields become a generic
  `HashMap<String, RuleOverride>`.

### `RenderCtx` retires in favor of full `Buffer` access

After P9.b, the Lua tool's `render(buf, args, output, width)` receives a
`Buffer` userdata directly. Plugins can `add_highlight`, `set_extmark`,
`set_lines`, `add_dim`, attach decorations ‚Äî the full surface. No
"render-context" subset.

The shipped Rust renderers (`smelt.bash.render`, `smelt.diff.render`,
`smelt.syntax.render`, `smelt.notebook.render`) stay as **convenience
primitives** that any plugin can call when it wants the standard look,
but the API is generic (`render(buf, opts) -> ()`), not packaged
("render the standard diff confirmation panel"). A plugin that wants a
different layout composes from primitives without going through them.

### Code rule (eternal ‚Äî added to ARCHITECTURE.md)

> **No tool/command/dialog name matching in Rust.** A `match name {
> "bash" => ‚Ä¶ }` (or `if name == "bash"`) over a Lua-registered
> identifier is a bug. The registration table carries the metadata; Rust
> calls through it generically.

## P9.e ‚Äî HlGroup-id model

**Goal:** Buffer extmarks carry semantic `HlGroup(u32)` ids, not raw
`Color` / `Style`. Theme is a paint-time resolver. Matches nvim. After
this, `core::buffer` knows nothing about colors; theme switches are
re-resolutions, not buffer rewrites.

- `core::buffer::HlGroup(u32)` minted by a registry; Extmark `Highlight`
  payload references it.
- `Theme::resolve(HlGroup) -> Style` lives in core; tui / GUI frontends
  each map `Style` (or just `HlGroup` directly) to their native paint.
- Plugins register groups by name (`smelt.theme.set("Warning", ‚Ä¶)`);
  call sites use names, not literal colors.
- Migration: `HlGroupOrStyle` enum during the flip, drop `Style`
  variant when every site lands. `core::style::*` survives only at the
  theme/paint boundary.

Deferred from P9.b because the Buffer move + crossterm-decoupling
already unblock GUI frontends; HlGroup is a separate 100+-site naming
pass that earns its own sub-phase.

## Deferrals

- **HlGroup-id migration ‚Üí P9.e.** Buffer carries raw `Style` in
  extmark payloads today; semantic decoupling lands after P9.c/d.
