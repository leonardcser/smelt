## P9.b — `Buffer` to `core` (the keystone)

`Buffer` is pure data with zero terminal deps; it belongs in core.
With it there, `RenderCtx` collapses, `DisplayBlock` deletes, and
headless reads transcript through the same surface as tui. `Theme`
and parser *implementations* (markdown / diff / syntax / bash) stay
in tui — they pull from tui-only crates (syntect, etc.).

1. ✅ Move `Buffer` + `BufferParser` to `core`.
2. ✅ `render_tool_body` takes `Buffer` userdata; `RenderCtx` deleted.
3. Delete `DisplayBlock`, `DisplayLine`, `SpanCollector`,
   `layout_out.rs`, `transcript_present/` rendering glue.
4. P9.c starts from the new shape.

## P9.c — Transcript pipeline as `BufferParser` impls

The path `Block → BufferParser → Buffer → Window::render` lands end
to end. Closes the P1.a-tail / P4.b deferral chain.

- One parser per `Block` variant under
  `tui::content::transcript_parsers/`; each writes lines + extmarks
  into a per-block `Buffer`.
- Width-independent parsing at ingest (markdown AST, diff LCS,
  syntect tokens as namespaced extmarks); `ensure_rendered_at(width)`
  re-flows on resize.
- `TranscriptProjection` shrinks to composition only;
  `BlockBufferCache` is the only cache.
- Prompt input goes through the same parser — deduplicates with
  user-bubble rendering; `PromptWrap` deletes or shrinks to byte-map
  utility.
- One `copy_range(buf, range)` replaces the eight divergent copy
  paths (transcript / prompt keybind / mouse / vim / kill-ring /
  etc.).
- `responsive_line(spans, width)` shared between
  `content/status.rs` and `content/prompt_buf.rs::bar_row`.

## P9.d — De-Rustify Lua concerns

Eternal rule (added to ARCHITECTURE.md): **no tool / command /
dialog / mode name matching in Rust over a Lua-registered identifier.**
The registration table carries the metadata; Rust calls through it.

Tool registration grows: `summary(args)`, `render(buf, args, output,
width)`, `paths_for_workspace(args)`, `pattern_from_args(args)`,
`elapsed_visible`. Audit (2026-05-05, refined 2026-05-05 against live
code) — sites + replacements:

- `engine/agent.rs:1179, 1443` cmd_summary → `dispatcher.summary`.
- `core/transcript_present/tools.rs:32, 43, 114` (bash + web_fetch
  branches) → gone with `transcript_present/`; tool block render is
  `tool.render(buf, …)`.
- `core/transcript_model.rs::ActiveTool::elapsed` → reads
  `elapsed_visible` from the tool def.
- `core/permissions/workspace.rs::extract_tool_paths` →
  `tool.paths_for_workspace(args)`.
- `core/permissions/mod.rs::decide_base` (`tool_name == "bash" |
  "web_fetch"`) → generic `tool.pattern_from_args(args) ->
  Option<String>` callback, then `check_tool_pattern(mode, tool,
  pattern)` for any tool. Bash + web_fetch lose their special-case
  branches; custom tools get pattern-based rules for free.
- `core/permissions/approvals.rs::is_auto_approved` (bash ruleset
  passed specially) → tools that opt into pattern rules return their
  ruleset via `permissions.ruleset_for(tool, mode)`; no bash branch.
- `tui/lua/api/confirm.rs::{render_title_into_buf, _back_tab,
  is_bash}` → delete; `dialogs/confirm.lua` composes title +
  auto-allow heuristic.
- `tui/lua/api/statusline.rs` `AgentMode → glyph` map → `modes.lua`.
- `core/permissions/rules.rs` mode-default table →
  `smelt.permissions.default_for(tool, mode)`.
- `core/custom_commands.rs::CommandOverrides` → generic
  `HashMap<String, RuleOverride>`.

## P9.e — HlGroup-id model

Buffer extmarks carry `HlGroup(u32)`, not raw `Style`. Theme is the
paint-time resolver; switches stop rewriting buffers.

- `core::buffer::HlGroup(u32)` minted by registry; `Theme::resolve`
  in core, native paint mapping in each frontend.
- Plugins register groups by name (`smelt.theme.set("Warning", …)`).
- Migration via `HlGroupOrStyle` enum, drop `Style` variant when
  every site lands. `core::style::*` survives only at the paint
  boundary.

Deferred from P9.b — separate 100+-site naming pass, independent of
the Buffer move.

## P9.f — Plugin API ergonomics (nvim parity)

Three gaps from the nvim audit
(`runtime/lua/vim/`, `src/nvim/api/keysets_defs.h`). Data layer
already has `NsId` + `set_extmark` + `ExtmarkOpts`; the work is at
the Lua boundary.

- **Expose namespace handles in Lua.** `smelt.api.create_namespace(name)`
  returns the existing `NsId`; same name returns same id (nvim
  semantics). Built-ins under `smelt.ns.*`. Today's
  `smelt.buf.add_highlight` writes to a default ns, hidden from Lua.
- **Unified extmark keyset.** Collapse `add_highlight` / `add_dim`
  to one `set_extmark(buf, ns, row, col, opts)` Lua binding over the
  existing data-layer call. Opts mirror nvim verbatim (`hl_group`,
  `virt_text`, `virt_lines`, `sign_text`, `conceal`, `priority`,
  `right_gravity`, `end_right_gravity`, `hl_eol`, `hl_mode`,
  `virt_text_pos`, `id`, `end_row`, `end_col`, …) plus smelt
  extensions (`yank`, `on_click`). Gravity needs ~30 LOC on the data
  type.
- **`runtime/lua/smelt/_meta/smelt.lua`.** Hand-written EmmyLua
  annotations for IDE / LSP. Editor-facing only.

Independent of P9.c/d/e.

## P9.h — Hook surface expansion

`Cells` is the right primitive; the *set* of pre-declared events is
thin vs pi-mono's ~30 hooks. Each new cell is one
`cells.set(name, payload)` at the right point — land incrementally.

New cells (event-shaped, fired tui-side): `session_before_compact`,
`session_before_fork`, `message_start`, `message_end`,
`tool_execution_start`, `tool_execution_end`, `input_submit`.

**System-prompt assembly: not a cell.** `smelt.prompt` already has
`set_section / remove_section`; grow it with `register_section(name,
fn, deps?)` mirroring `smelt.statusline.register`. The assembler
walks registered sections at compose time and concatenates in
registration order. `Arc<Mutex<…>>` mutate-payload over Cells is the
wrong pattern — closure-based source registration is what the
codebase already does for the statusline.

**`before_provider_request` / `after_provider_response`: redundant.**
Engine emits `TurnComplete` (already a cell); the request body is
assembled tui-side via the prompt-section register above, so a
pre-engine cell is one more `input_submit` subscriber's read of
`smelt.engine.compose_preview()` (a synchronous helper that runs the
same assembly engine sees).

Stale-context invalidation: tag `LuaTask`s with session id; cancel
on session boundary mismatch. `smelt.spawn(fn, { tag = "session" |
"global" })`. Reuses existing `CancellationToken` from P6.c +
existing `session_id` field on the LuaTask context.
