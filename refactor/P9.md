## Strategic concerns (audit 2026-05-05)

Plan-vs-code drift surfaced from a code audit; resolutions below.

### Tool render proliferation → `LayoutTree<BufId>`

`bash.lua` registers 11 callbacks. The five render callbacks
(`render` / `render_summary` / `render_subhead` / `header_suffix` /
`preview`) each mint a transient Buffer in Rust, call Lua, replay
into a `SpanCollector`. Five dispatch sites for "where in the tool
block does this paint go" — slot vocabulary owned by Rust.

**Direction (P9.r):** tool registers one `render(args, output, ctx)`
returning a `LayoutTree<BufId>`. Tool composes its own block layout
out of Buffers; Rust composer walks the tree and replays leaves into
the transcript display Buffer. Slots disappear; same primitive
dialogs and overlays already use. `preview` stays separate (different
surface, different lifecycle). `elapsed_visible` and `header_suffix`
fold into the tool's own header Buffer (extmarks). Live updates flow
via cell subscriptions in the render coroutine, not callback-per-frame.

Decision callbacks (`decide` / `needs_confirm` / `approval_patterns`)
stay separate. They each answer slightly different questions
(`decide` is the full ask/allow/deny composition; `needs_confirm`
returns the description shown to the user; `approval_patterns`
suggests "always allow" patterns). Cosmetic collapse possible later
if a clean shape emerges.

### "No defer on size" applied unevenly

Spirit was: don't procrastinate on architectural improvements.
P9.b/d/e/f/l all landed despite being heavy. P9.i (provider
middleware, ~500 LOC) sits open with no consumer — speculative
seam. **Restate: defer for absent consumer, not for size.** P9.i
moves to deferrals; P9.c open work that the entry conditions
already require stays open (don't call required work "deferred").

### P9.c is open work, not "follow-ups deferred"

P9.b's log called three pieces "clean follow-up sub-phases, not
blockers":

- per-`Block` `BufferParser` impls under
  `tui::content::transcript_parsers/`
- prompt rendering through the same parser
- one `copy_range(buf, range)` primitive

But the entry conditions for P10 demand all three.
`transcript_present/mod.rs` is 1330 LOC (grew from 1153);
`Window::mouse_yank_text` (`window.rs:801`) is still a separate
copy path. Resurrect P9.c as a live sub-phase.

### Pre-`BufferParser` stack still doubled with the new shape

P9.b/e landed additively — new shape in, old shape stays. Three
rings of leftover abstraction sit in parallel with the target:

1. **Two color-resolution layers stacked.** `content/display.rs`
   defines `SpanStyle` + `ColorValue::{Rgb,Ansi,Named,Role}` +
   `ColorRole { Accent, Slug, ToolPending, ReasonOff, … }` +
   `NamedColor` (~80 LOC). P9.e then introduced `HlGroup(u32)`
   resolved by `Theme::resolve(hl)` — same indirection, different
   mechanism. `HlGroup` is "the" model now; `ColorRole` /
   `ColorValue` / `SpanStyle` / `NamedColor` are pre-P9.e
   leftovers.
2. **`SpanCollector`'s accumulator duplicates the `Buffer` API.**
   `cur_text` + `cur_highlights: Vec<(u16, u16, HlGroup, SpanMeta)>`
   + `cur_decoration` is a mini-Buffer-of-one-line flushed
   line-by-line. `Buffer` already exposes `set_lines` +
   `set_extmark` + namespaces. P9.c parsers call the Buffer API
   directly; the 764-LOC `layout_out.rs` collapses.
3. **`core/src/transcript_present.rs` (139 LOC) is the
   pre-`BufferParser` trait shape.** `TranscriptPresenter` with
   `print_left` / `print_right` / `render_block` taking
   `&mut SpanCollector`. The new `tui::content::transcript_parsers/`
   tree (10 files, ~3400 LOC) already exists in parallel; the old
   trait survives only because nothing deleted it.

**P9.c scope is the deletion, not just the new files.**
`SpanCollector` + `SpanStyle` + `ColorRole` + `ColorValue` +
`NamedColor` + `transcript_present.rs` retire as one move
(~1000 LOC delete) when the BufferParser impls take over.
Parsers write through `buf.set_lines(...)` + `buf.set_extmark(ns,
row, col, opts)`; `ColorRole` variants become name-keyed `HlGroup`s
interned in the theme.

### Hardcoded permission defaults — same shape P9.d killed elsewhere

Audit 2026-05-06 surfaced a leak P9.d's closeout missed.
`core/permissions/rules.rs::install_tool_defaults` (line ~165) holds
a hardcoded list of 11 tool names (`read_file` / `edit_file` /
`write_file` / `glob` / `grep` / `bash` / `web_fetch` / `web_search` /
`read_process_output` / `stop_process` / `ask_user_question`) with
per-mode `Allow` / `Ask` defaults baked in. The doc comment admits
"real product UX overrides this via `smelt.permissions.set_rules`" —
which is the tell: Rust ships a fallback for tools it shouldn't know
about. Same shape P9.d killed at every other call site.

`DEFAULT_BASH_ALLOW` (rules.rs:217, P9.d's "accepted single
exception") is data, not shell-aware logic. `bash.lua` could own
the list. Worth re-litigating alongside `install_tool_defaults`
since both fall to the same fix. **P9.y** owns the cleanup.

### Cell typing vs settings typing asymmetry

Built-in cell name typing was deferred ("typo-as-no-op rare;
runtime checks already reject undeclared names"). Settings has no
equivalent guard — `smelt.settings.set("vimode", true)` silently
no-ops. P9.x adds reject-unknown to settings. Cells stay deferred
(already validated). The asymmetry was the bug, not the deferral.

### P9.r — Tool render returns `BlockLayout` (landed)

Single `render(args, output, ctx) -> BlockLayout` callback per tool
collapses the four slot callbacks (`render_summary` /
`render_subhead` / `header_suffix` / `elapsed_visible`). Tool composes
its block body from Buffers wrapped in `smelt.layout.leaf` /
`smelt.layout.vbox`; the Rust composer walks the tree and replays
each leaf's lines into the surrounding `LineBuilder` with `"  "`
gutter padding. `MAX_TOOL_BLOCK_ROWS` truncation applies to the
walked layout.

Shipped shape (bash):

```lua
render = function(args, output, ctx)
  local items = {}
  if ctx.status == "pending" then
    table.insert(items, smelt.layout.text("(timeout: 2m)"))
  end
  table.insert(items, smelt.layout.text(output.content,
    { is_error = output.is_error }))
  return smelt.layout.vbox(items)
end
```

Landed pieces:

1. **`smelt_core::content::block_layout::BlockLayout`** — minimal
   enum (`Leaf(BufId) | Vbox(Vec)`); no chrome / constraints (those
   are split-specific), no `Hbox` (deferred — see Deferrals).
   `BlockLayout::leaves()` walks the tree in declaration order.
2. **`smelt.layout.{leaf, vbox}`** Lua bindings in
   `core::lua::api::layout` wrap `BlockLayout` as
   `LuaBlockLayout` userdata. `_bootstrap.lua` adds
   `smelt.layout.text(content, opts)` sugar (mints buffer →
   `smelt.text.render` → `leaf`).
3. **`LuaRuntime::render_tool_layout(name, args, output, ctx)`**
   replaces `render_tool_body`. Returns `Option<BlockLayout>`;
   `ctx: ToolRenderCtx<'_>` carries `width / summary / status /
   elapsed_secs / call_id`. The 6 dropped runtime helpers
   (`tool_elapsed_visible`, `tool_has_render_summary`,
   `render_tool_summary_line`, `tool_has_render_subhead`,
   `render_tool_subhead`, `tool_header_suffix`) retire.
4. **Composer rewrite** in
   `tui::content::transcript_parsers::tools::render_tool`. Rust
   keeps painting the universal `⏺ status_pill name summary
   elapsed_pill` row 0 (elapsed shows when `>= 1s`, replacing the
   per-tool `elapsed_visible` opt-in). Body: if Lua returns a
   layout, walk it; else fall back to the default
   `print_tool_output`. The renderer parameter threaded through
   `layout_block_into` / `render_block` / `BlockBufferCache::ensure`
   /  `TranscriptProjection::project` is gone; `BlockHistory.body_renderer`
   field deleted.
5. **`ToolBodyRenderer` trait + `core/src/transcript_present.rs`
   deleted entirely**. `LuaRenderRenderer` shim in
   `app/transcript.rs` deleted with it.
6. **Lua tool migration** — `bash.lua` / `web_fetch.lua` / `glob.lua`
   / `grep.lua` / `read_file.lua` / `edit_file.lua` /
   `write_file.lua` / `notebook_edit.lua` / `web_search.lua` all
   updated. Single `render(args, output, ctx)` returning a layout.
7. **Confirm dialog** drops `tool_has_render_summary` /
   `render_tool_summary_line` calls; bash command paints as plain
   text in the dialog title (the preview pane keeps full bash
   highlighting via `_render_preview` →
   `LuaRuntime::render_tool_preview`).

Visual regressions accepted:

- **Bash command summary highlighting** in the row-0 transcript
  line is now plain text. The `print_tool_line` summary slot
  paints from the engine-recorded `summary` string only — no
  per-line callback. Bash highlighting still appears in the
  confirm dialog's preview pane and in the body subhead row.
- **Bash `(timeout: 2m)` suffix** moves from a row-0 right-edge
  pill to the first body row (one line below the summary).
  Same information, different placement.

Deferred from the original sketch:

- **Projection-layer fold** of `transcript_buf` /
  `transcript_snapshot` / `block_buffers` — promoted to **P9.s**
  (next sub-phase). Today every `build_snapshot` call mints a fresh
  `BlockBufferCache` and re-lays every block out — and the per-frame
  call sites in `app/transcript.rs` (~8 of them) each pay that cost
  independently of `TranscriptProjection`'s own cache. The layout
  walker didn't *cause* that drift; it surfaced it.
- **`preview` callback** still uses the old single-buffer shape
  (`smelt.confirm._render_preview` → `LuaRuntime::render_tool_preview`).
  The architecture hint that preview returns a layout for
  consistency would land alongside a confirm-dialog refactor and
  isn't worth the churn yet — preview has one consumer (the confirm
  dialog), one buffer, no composition needs.

Reconsidered and *not* deferred:

- **Per-leaf `BlockBufferCache` extension** and **cell-driven live
  updates** were originally listed as deferred perf optimizations.
  On re-examination both are non-issues: `update_tool_state` calls
  `BlockHistory::bump_generation` on every meaningful mutation, the
  whole-block cache invalidates at exactly that cadence, and
  `state.elapsed` is only set on tool finish (not live-updating).
  The layout walker runs at the same frequency as the pre-P9.r
  renderer trait. There's no regression to fix and no hot path
  cells could shave.

### P9.s — Snapshot folds onto `TranscriptProjection` (landed)

The transcript-display path had three layers that all walked the
same `BlockHistory` and produced the same per-block buffers:

1. **`BlockBufferCache`** (`content/block_buffers.rs`) — per-block
   layout cache keyed by `LayoutKey`. The actual cache primitive.
2. **`TranscriptProjection`** (`content/transcript_buf.rs`) — owns one
   `BlockBufferCache`; calls `cache.ensure(...)` to write the
   transcript display buffer that the renderer paints.
3. **`build_snapshot`** (`content/transcript_snapshot.rs`) — free
   function that **builds its own fresh `BlockBufferCache`** every
   call, walks history, and produces a `TranscriptSnapshot` for
   copy / yank / line-break / cell-snap / pane-focus paths.

The drift: `build_snapshot` is called ~8 times from
`app/transcript.rs` (`full_transcript_display_text`,
`transcript_line_breaks`, `block_text_at_row` x2,
`snap_col_to_selectable`, `snap_cpos_to_selectable`,
`copy_display_range`) plus `app/mouse.rs` and `app/pane_focus.rs`.
Each call recomputes the whole transcript layout from scratch —
even though the `TranscriptProjection` cache *already has the same
buffers* sitting one struct away.

**Landed.** `build_snapshot` is now `pub(crate) fn` in
`transcript_snapshot.rs` taking `&mut BlockBufferCache` as its first
argument; `TranscriptProjection` owns the cache and a lazy
`snapshot: Option<TranscriptSnapshot>` slot rebuilt only when
`(history.generation(), width, show_thinking)` no longer matches the
cached snapshot's. Both `project()` and `snapshot()` call a shared
`gc_if_stale(gen, width)` helper that clears the per-block cache and
the snapshot once on generation/width drift, so within a single
generation the two consumers share fresh per-block buffers without
recomputing.

The 9 call sites under `app/{transcript,mouse,pane_focus}.rs` now
route through `self.transcript_projection.snapshot(&mut
self.transcript.history, tw, show_thinking, &theme)`. The standalone
`build_snapshot` no longer mints its own `BlockBufferCache`; calling
it without a projection isn't possible from outside the module
(the function is `pub(crate)` and takes a borrowed cache).

**Outcome.** One `BlockBufferCache` per session; the per-frame
redundant relayout in 9 call sites collapses onto a single
`(generation, width, show_thinking)`-keyed snapshot. Net ~5 LOC
delete plus a substantial work reduction — every `build_snapshot`
call site previously walked every block at every width, now they
walk once per `(gen, width, show_thinking)` change.

### P9.n — Vim becomes Lua-extensible

`vim.rs` keymap dispatch is hardcoded Rust; users can't override
`dd` in init.lua even though every other interactive surface is
recipe-driven.

- **P1.d.5d** — registers / dot-repeat / undo move from
  `VimWindowState` to `Buffer` (per-buffer state lives where
  Buffer does).
- **P1.d.5f.2d** — vim dispatch table flattens onto the Lua keymap
  registry; `vim.rs` becomes the recipe of default keymap entries.

Framing is **Lua extensibility parity**, not "vim.rs is big" —
file size shrinks as a side effect. Lands after P9.f.

### P9.o — Prompt is a Window

`tui::input::PromptState` (~1170 LOC) is the only interactive
surface that isn't a Window over a Buffer with a keymap recipe.

- Edit state → Buffer.
- Cursor / scroll / selection → Window.
- Keymap → Lua via `widgets/input.lua`.
- Completer → ghost-text extmark + Overlay picker.
- Attachments stay as a small input-side concern;
  `core/src/attachment.rs` (237 LOC, `restructured` for "P1/P4 —
  extmarks + Lua input widget" but never touched) absorbs into the
  Lua input widget here.
- `tui/content/prompt_wrap.rs` (233 LOC, INVENTORY claimed P9.b
  delete-or-shrink that never landed) collapses with the rewrite.

**Bundled scope: UiHost TLS re-key.** Today
`crate::lua::with_app` / `try_with_app` reborrow `&mut TuiApp`
specifically — every UiHost-tier Lua binding (`buf` / `win` /
`overlay` / `prompt` / `confirm` / `statusline` / …) is keyed to
the concrete TuiApp struct. The Host tier already solved this in
P8.f (`try_with_host` over `&mut dyn Host` via TLS `CORE_PTR`).
P9.o flips the UiHost tier to match: a new TLS slot
(`UI_HOST_PTR`) holds `*mut dyn UiHost`, `try_with_uihost`
reborrows it, every `tui/src/lua/api/*.rs` binding migrates from
`with_app(|app| app.ui...)` to `with_uihost(|host| host.ui...)`.
TuiApp continues to impl UiHost; nothing functional changes —
the point is the seam. Once UiHost is type-erased at the TLS
boundary, anything UI-bearing can host UiHost-tier Lua bindings.
Same shape rule, mirrored.

Lands after P9.n (rides the same Lua keymap dispatch path).
Unblocks future visual-test work (see `TESTING.md` § L3).

### P9.x — Config binding fidelity (landed)

Mechanical reject-unknown / fidelity bugs in config-time bindings.

- **`provider.rs`** — `register` accepts each entry in `models`
  as a bare string or a table; reject-unknown over the 11 known
  per-model fields (`name` / `temperature` / `top_p` / `top_k` /
  `min_p` / `repeat_penalty` / `tool_calling` / `input_cost` /
  `output_cost` / `cache_read_cost` / `cache_write_cost`). `list`
  surfaces every field on each model row.
- **`mcp.rs`** — `register` reads `type` / `command` / `args` /
  `env` / `timeout` / `enabled`. Unknown top-level keys raise an
  error; unknown `type` values do the same so a future remote
  variant won't silently change config meaning.
- **`settings.rs`** — collapsed `set` / `toggle` / `snapshot` into
  field access via metatable. `smelt.settings.vim = true` /
  `local v = smelt.settings.vim` / `for k, v in pairs(smelt.settings)`
  route through `__newindex` / `__index` / `__pairs`. Unknown keys
  error at the access site. `SettingsConfig` field names renamed
  to match `ResolvedSettings` (`vim` / `show_prediction` /
  `show_slug`); `SETTINGS_KEYS` is the single source of known
  names. `LuaShared.settings_overrides` is now `HashMap<String,
  bool>`; `to_config()` propagates `set_bool` errors via
  `eprintln!` instead of silently discarding. `--set KEY=VALUE`
  (CLI) now parses booleans up front.

### P9.y — Permission defaults to Lua

`core/permissions/rules.rs` carries two leaks of the shape P9.d
killed everywhere else:

1. **`install_tool_defaults`** (line ~165) — hardcoded list of 11
   tool names with per-mode `Allow` / `Ask` defaults. Comment
   admits "real product UX overrides this via
   `smelt.permissions.set_rules`."
2. **`DEFAULT_BASH_ALLOW`** (line ~217) — list of safe bash
   prefixes (`ls`, `cat`, `pwd`, `grep`, …). P9.d accepted this
   as a "single named special case" but the list is *data*, not
   shell-aware logic; `bash.lua` can own it.

**Direction.** Defaults move into Lua. Each built-in tool's `.lua`
file declares its mode-defaults (`M.permission_defaults =
{ normal = "ask", apply = "allow", yolo = "allow" }`); `bash.lua`
declares `M.default_allow = { "ls", "cat", … }`. The Rust
permissions layer reads these through the existing
`smelt.permissions.set_rules` channel at registration time —
no new FFI surface needed. `install_tool_defaults` and
`DEFAULT_BASH_ALLOW` retire; `rules.rs` shrinks ~80 LOC.

Concrete changes:

- Each `runtime/lua/smelt/tools/*.lua` declares default
  permissions for the modes it expects.
- A `runtime/lua/smelt/_bootstrap.lua` (or similar) walks
  registered tools and populates `set_rules` on the *empty*
  default ruleset before user `init.lua` runs. User init can
  still override.
- `bash.lua` registers a default allow-list under its
  `subcommands.bash` bucket via the same channel.
- `install_tool_defaults` deletes; the `if mode == AgentMode::Apply`
  / `AgentMode::Yolo` branches go with it.
- The `if name == "bash"` arm in `build_subcommand_ruleset`
  deletes; the function becomes pure ruleset compilation.

After P9.y the verification grep returns zero exceptions; the
"single named special case" disappears.

### P9.i — Provider middleware (deferred)

Neutral `ProviderRequest` / `ProviderResponse` between
`EngineClient` and per-kind serializers; `Vec<Box<dyn ProviderMiddleware>>`
with return-payload semantics. ~500 LOC.

**Deferred** — no concrete consumer. Land when first plugin needs

## Decisions

Ordered by surprise value, not chronology.

- **No defer on size; defer on absent consumer.** Refines the
  earlier "no defer on size" rule. Speculative seams without
  callers wait; required architectural improvements don't.
- **Tool render returns `BlockLayout`, not five slot callbacks.**
  Tool composes its own block layout out of Buffers. Slot
  vocabulary (`render_summary` / `render_subhead` / `header_suffix`
  / `elapsed_visible`) retires; the four callbacks collapse into
  one `render(args, output, ctx) -> BlockLayout`.
- **`BlockLayout` is its own enum, not a `LayoutTree<L>` generic.**
  Splits/overlays use `tui::ui::LayoutTree<WinId>` with chrome /
  constraints / focus semantics; tool blocks are sequences of
  buffer leaves replayed into a `LineBuilder`. Different consumers,
  different needs. Forcing them to share the enum would couple two
  surfaces with no benefit. Architecture text "same primitive" is
  an aspirational alignment, not a requirement.
- **Tool decides custom row 0; Rust paints universal pill row.**
  The `⏺ status_pill name summary elapsed_pill` line stays
  Rust-painted (universal across all tools); the tool's
  `render` callback composes the body below. Consequence: bash's
  per-line summary highlighting and `(timeout: 2m)` row-0 suffix
  retire — bash highlighting still ships in the confirm preview
  pane, the timeout suffix moves to a body subhead row. Acceptable
  visual regression in exchange for retiring the slot vocabulary.
- **`elapsed_visible` flag becomes implicit at `>= 1s`.**
  Old per-tool opt-in (only `bash` / `web_fetch` set the flag)
  retired in the registration table. Composer paints the elapsed
  pill whenever `elapsed.as_secs_f64() >= 1.0`. Most tools
  complete faster than 1s and won't trigger it.
- **Permissions buckets generic by tool name.** `RawModePerms` /
  `RuleOverride` / `PermissionOverrides` carry a `subcommands:
  HashMap<String, RuleSet>` instead of hardcoded `bash` /
  `web_fetch` / `mcp` fields. Any tool that needs subcommand
  patterns registers a bucket by its own name. Closes the last
  shared-Rust tool-name match in the permissions data **shape**.
  Defaults still leak (`install_tool_defaults`,
  `DEFAULT_BASH_ALLOW`) — P9.y fixes that.
- **Permission defaults are Lua-owned.** Each built-in tool's
  `.lua` file declares its mode-defaults; `bash.lua` carries its
  own default-allow list. Rust permission layer reads through
  `smelt.permissions.set_rules` at registration time —
  `install_tool_defaults` and the `if name == "bash"` arm in
  `build_subcommand_ruleset` retire (P9.y).
- **`smelt.settings` collapses to field access via metatable.**
  `set` / `toggle` / `snapshot` retire; `smelt.settings.vim = true`
  reads / writes / iterates through one API. Unknown keys error at
  the access site. Same shape proposed for `smelt.theme` later.
- **Trust UX is `/trust` command, not startup dialog.** Interactive
  startup prompts couple loading to the event loop; deny-by-default
  + toast + opt-in slash command keeps loading sync.
- **Loader order: embedded → user init → global plugins → project
  plugins+init.** User init can override embedded defaults; project
  config is last so per-repo `<cwd>/.smelt/init.lua` always wins.
  Project content gated by SHA-256 hash + `/trust` mark.
- **Single-LuaRuntime autoload runs without TLS app pointer.**
  Verified safe — every embedded autoload module's top-level code
  is either a registration call (`smelt.{tools,cmd,au,cell}.register`,
  writes to `LuaShared`) or a pure data lookup. Anything reading
  live app sits inside callbacks fired after `start()`.
- **Anonymous HlGroup interning by content hash.** Theme switches
  no longer rewrite buffers — they mutate `Theme.styles[id]` once.
  Sites that haven't picked a name yet content-hash their `Style`
  into `__anon__/<hex>` keys.
- **Cell `(new, old)` via `mem::replace`.** `Cells::set_dyn` swaps
  in-place, stashes prior `Rc<dyn Any>` on `PendingFire`. Direct
  subscribers fire `(new, old)`; glob subscribers `(name, new, old)`.
- **`include_dir!` collapse for embedded Lua.** Side effect:
  three plugins embedded but never `require`'d (`plan_mode`,
  `predict`, `yank_block`) now autoload — fixed a latent bug
  where the `exit_plan_mode` tool wasn't installed.
- **No big-file decomp for size's sake.** `agent.rs` (1948),
  `ui/mod.rs` (~1600), `window.rs` (1198) stay as-is — single
  coherent structs, splitting fragments state.

## Deferrals

- **P9.i** (provider middleware) — no consumer. Land on first ask.
- **EmmyLua stubs** (`_meta/smelt.lua`) — post-P10 polish.
- **Plugin handle / lifecycle** (`smelt.plugin.define{...}` +
  teardown) — over-engineering for static-file plugin set. Revisit
  when an ecosystem needs hot reload or `/plugins disable`.
- **Reactive statusline binding** (`{ bind = cell, fmt = fn }`).
  Composer ticks once per main-loop iteration. Profile says don't
  bother yet.
- **Plugin manifest blocks** — no `/plugins` UI consumer.
- **Multi-mode `keymap.set({"n","v"}, ...)` sugar** — trivially
  composed in Lua.
- **`vim.ui.set_picker`-style override hook** — no second picker
  exists.
- **Built-in cell name typing** — `set_dyn` already rejects
  undeclared names; protection exists.
- **`smelt.cmd.read_md` bridge** — no consumer asked; .md and Lua
  command systems coexist fine.
- **`BlockLayout::Hbox` (side-by-side tool composition).** Dropped
  from the enum + Lua surface in P9.r since no built-in tool needs
  it and no plugin asked. Add when the first real consumer arrives
  (likely a custom diff or stats column tool) — the consumer will
  pin the design choices that are otherwise guesses today:
  - Column allocation: equal split / content-driven / user-weighted
    (`{ buf = b, weight = 2 }`)?
  - Separator: none / single column / styled bar?
  - Row-count mismatch (one column taller): pad blanks / center?
  - Per-column wrap behavior at narrow widths.
  Composer rewrite is non-trivial — `replay_layout` today walks
  leaves vertically; horizontal needs row-zip across columns with
  per-column visible-width tracking.
- **`engine_client.rs` collapse-or-bridge.** 58-LOC confirms-gate
  pass-through. INVENTORY notes target is "either full
  event-to-buffer/cell bridge or no wrapper at all"; today it's
  neither. The wrapper has one job (gate `recv` on
  `confirms_clear`) and it does it correctly. Re-evaluate after
  P9.r — if `EngineClient` doesn't grow event-bridge logic, fold
  `recv`/`try_recv` into `Core::recv` directly and delete the
  type.
