## Strategic concerns (audit 2026-05-05)

Plan-vs-code drift surfaced from a code audit; resolutions below.

### Tool render proliferation → `LayoutTree<BufId>`

`bash.lua` registers 11 callbacks. The five render callbacks
(`render` / `render_summary` / `render_subhead` / `header_suffix` /
`preview`) each mint a transient Buffer in Rust, call Lua, replay
into a `SpanCollector`. Five dispatch sites for "where in the tool
block does this paint go" — slot vocabulary owned by Rust.

**Direction (P9.r):** tool registers one `render(args, output, ctx)`
returning a `LayoutTree<BufId>`. Tool composes its own block layout
out of Buffers; Rust composer walks the tree and replays leaves into
the transcript display Buffer. Slots disappear; same primitive
dialogs and overlays already use. `preview` stays separate (different
surface, different lifecycle). `elapsed_visible` and `header_suffix`
fold into the tool's own header Buffer (extmarks). Live updates flow
via cell subscriptions in the render coroutine, not callback-per-frame.

Decision callbacks (`decide` / `needs_confirm` / `approval_patterns`)
stay separate. They each answer slightly different questions
(`decide` is the full ask/allow/deny composition; `needs_confirm`
returns the description shown to the user; `approval_patterns`
suggests "always allow" patterns). Cosmetic collapse possible later
if a clean shape emerges.

### "No defer on size" applied unevenly

Spirit was: don't procrastinate on architectural improvements.
P9.b/d/e/f/l all landed despite being heavy. P9.i (provider
middleware, ~500 LOC) sits open with no consumer — speculative
seam. **Restate: defer for absent consumer, not for size.** P9.i
moves to deferrals; P9.c open work that the entry conditions
already require stays open (don't call required work "deferred").

### P9.c is open work, not "follow-ups deferred"

P9.b's log called three pieces "clean follow-up sub-phases, not
blockers":

- per-`Block` `BufferParser` impls under
  `tui::content::transcript_parsers/`
- prompt rendering through the same parser
- one `copy_range(buf, range)` primitive

But the entry conditions for P10 demand all three.
`transcript_present/mod.rs` is 1330 LOC (grew from 1153);
`Window::mouse_yank_text` (`window.rs:801`) is still a separate
copy path. Resurrect P9.c as a live sub-phase.

### Pre-`BufferParser` stack still doubled with the new shape

P9.b/e landed additively — new shape in, old shape stays. Three
rings of leftover abstraction sit in parallel with the target:

1. **Two color-resolution layers stacked.** `content/display.rs`
   defines `SpanStyle` + `ColorValue::{Rgb,Ansi,Named,Role}` +
   `ColorRole { Accent, Slug, ToolPending, ReasonOff, … }` +
   `NamedColor` (~80 LOC). P9.e then introduced `HlGroup(u32)`
   resolved by `Theme::resolve(hl)` — same indirection, different
   mechanism. `HlGroup` is "the" model now; `ColorRole` /
   `ColorValue` / `SpanStyle` / `NamedColor` are pre-P9.e
   leftovers.
2. **`SpanCollector`'s accumulator duplicates the `Buffer` API.**
   `cur_text` + `cur_highlights: Vec<(u16, u16, HlGroup, SpanMeta)>`
   + `cur_decoration` is a mini-Buffer-of-one-line flushed
   line-by-line. `Buffer` already exposes `set_lines` +
   `set_extmark` + namespaces. P9.c parsers call the Buffer API
   directly; the 764-LOC `layout_out.rs` collapses.
3. **`core/src/transcript_present.rs` (139 LOC) is the
   pre-`BufferParser` trait shape.** `TranscriptPresenter` with
   `print_left` / `print_right` / `render_block` taking
   `&mut SpanCollector`. The new `tui::content::transcript_parsers/`
   tree (10 files, ~3400 LOC) already exists in parallel; the old
   trait survives only because nothing deleted it.

**P9.c scope is the deletion, not just the new files.**
`SpanCollector` + `SpanStyle` + `ColorRole` + `ColorValue` +
`NamedColor` + `transcript_present.rs` retire as one move
(~1000 LOC delete) when the BufferParser impls take over.
Parsers write through `buf.set_lines(...)` + `buf.set_extmark(ns,
row, col, opts)`; `ColorRole` variants become name-keyed `HlGroup`s
interned in the theme.

### Cell typing vs settings typing asymmetry

Built-in cell name typing was deferred ("typo-as-no-op rare;
runtime checks already reject undeclared names"). Settings has no
equivalent guard — `smelt.settings.set("vimode", true)` silently
no-ops. P9.x adds reject-unknown to settings. Cells stay deferred
(already validated). The asymmetry was the bug, not the deferral.

### P9.r — Tool render returns `LayoutTree<BufId>`

Single render callback per tool. Tool composes its block layout out
of Buffers; Rust composer walks the tree and replays leaves into the
transcript display Buffer.

Sketch:

```lua
M.render = function(args, output, ctx)
  local header = smelt.buf.create()
  smelt.bash.render(header, args.command)
  if ctx.status == "pending" then
    smelt.buf.set_extmark(header, NS_SUFFIX, 1, -1, {
      virt_text = "(timeout: 2m)", virt_text_pos = "eol", dim = true,
    })
  end

  local body = smelt.buf.create()
  smelt.text.render(body, output.content, { width = ctx.width })

  return smelt.layout.vbox {
    smelt.layout.leaf(header),
    smelt.layout.leaf(body),
  }
end

M.preview = function(args)  -- separate surface (confirm dialog)
  local buf = smelt.buf.create()
  smelt.bash.render(buf, args.command)
  return smelt.layout.leaf(buf)
end
```

Concrete changes:

1. `LayoutTree` (`crates/tui/src/ui/layout.rs`) gains a
   `Leaf(BufId)` variant (today's leaves are `Leaf(WinId)` for splits).
2. Tool registry stores one `render` callback returning a
   `LayoutTree`. Drop `render_summary` / `render_subhead` /
   `header_suffix` / `elapsed_visible` from the tool registration
   table.
3. `BlockBufferCache` extends to cache per-leaf-buffer at
   `(source_tick, width)`. Leaves invalidate independently — header
   re-renders on status change, body on output append, etc.
4. Live updates (elapsed ticks, status changes): tool subscribes to
   a per-block cell (`tool_status:<call_id>`) inside its render and
   updates extmarks on the header buffer. Standard cell pattern, no
   per-frame callbacks.
5. Sugar: `smelt.layout.text(string, opts?)` for one-line leaves
   without explicit Buffer creation. Covers the simple case.

`preview` stays separate (different surface, different consumer) but
returns `LayoutTree<BufId>` for consistency.

Lands together with — or just before — P9.c, since both touch the
transcript composer.

### P9.n — Vim becomes Lua-extensible

`vim.rs` keymap dispatch is hardcoded Rust; users can't override
`dd` in init.lua even though every other interactive surface is
recipe-driven.

- **P1.d.5d** — registers / dot-repeat / undo move from
  `VimWindowState` to `Buffer` (per-buffer state lives where
  Buffer does).
- **P1.d.5f.2d** — vim dispatch table flattens onto the Lua keymap
  registry; `vim.rs` becomes the recipe of default keymap entries.

Framing is **Lua extensibility parity**, not "vim.rs is big" —
file size shrinks as a side effect. Lands after P9.f.

### P9.o — Prompt is a Window

`tui::input::PromptState` (~1170 LOC) is the only interactive
surface that isn't a Window over a Buffer with a keymap recipe.

- Edit state → Buffer.
- Cursor / scroll / selection → Window.
- Keymap → Lua via `widgets/input.lua`.
- Completer → ghost-text extmark + Overlay picker.
- Attachments stay as a small input-side concern.

Lands after P9.n (rides the same Lua keymap dispatch path).

### P9.x — Config binding fidelity (landed)

Mechanical reject-unknown / fidelity bugs in config-time bindings.

- **`provider.rs`** — `register` accepts each entry in `models`
  as a bare string or a table; reject-unknown over the 11 known
  per-model fields (`name` / `temperature` / `top_p` / `top_k` /
  `min_p` / `repeat_penalty` / `tool_calling` / `input_cost` /
  `output_cost` / `cache_read_cost` / `cache_write_cost`). `list`
  surfaces every field on each model row.
- **`mcp.rs`** — `register` reads `type` / `command` / `args` /
  `env` / `timeout` / `enabled`. Unknown top-level keys raise an
  error; unknown `type` values do the same so a future remote
  variant won't silently change config meaning.
- **`settings.rs`** — collapsed `set` / `toggle` / `snapshot` into
  field access via metatable. `smelt.settings.vim = true` /
  `local v = smelt.settings.vim` / `for k, v in pairs(smelt.settings)`
  route through `__newindex` / `__index` / `__pairs`. Unknown keys
  error at the access site. `SettingsConfig` field names renamed
  to match `ResolvedSettings` (`vim` / `show_prediction` /
  `show_slug`); `SETTINGS_KEYS` is the single source of known
  names. `LuaShared.settings_overrides` is now `HashMap<String,
  bool>`; `to_config()` propagates `set_bool` errors via
  `eprintln!` instead of silently discarding. `--set KEY=VALUE`
  (CLI) now parses booleans up front.

## Decisions

Ordered by surprise value, not chronology.

- **No defer on size; defer on absent consumer.** Refines the
  earlier "no defer on size" rule. Speculative seams without
  callers wait; required architectural improvements don't.
- **Tool render returns `LayoutTree<BufId>`, not five slot
  callbacks.** Tool composes its own block layout out of Buffers;
  same primitive dialogs and overlays use. Slot vocabulary owned
  by tool, not Rust. Live updates flow through cells, not
  per-frame callbacks. Hot path stays pure Rust — Lua only fires
  at semantic boundaries (status change, output append, width
  resize).
- **Permissions buckets generic by tool name.** `RawModePerms` /
  `RuleOverride` / `PermissionOverrides` carry a `subcommands:
  HashMap<String, RuleSet>` instead of hardcoded `bash` /
  `web_fetch` / `mcp` fields. Any tool that needs subcommand
  patterns registers a bucket by its own name. Closes the last
  shared-Rust tool-name match in the permissions data shape.
- **`smelt.settings` collapses to field access via metatable.**
  `set` / `toggle` / `snapshot` retire; `smelt.settings.vim = true`
  reads / writes / iterates through one API. Unknown keys error at
  the access site. Same shape proposed for `smelt.theme` later.
- **Trust UX is `/trust` command, not startup dialog.** Interactive
  startup prompts couple loading to the event loop; deny-by-default
  + toast + opt-in slash command keeps loading sync.
- **Loader order: embedded → user init → global plugins → project
  plugins+init.** User init can override embedded defaults; project
  config is last so per-repo `<cwd>/.smelt/init.lua` always wins.
  Project content gated by SHA-256 hash + `/trust` mark.
- **Single-LuaRuntime autoload runs without TLS app pointer.**
  Verified safe — every embedded autoload module's top-level code
  is either a registration call (`smelt.{tools,cmd,au,cell}.register`,
  writes to `LuaShared`) or a pure data lookup. Anything reading
  live app sits inside callbacks fired after `start()`.
- **Anonymous HlGroup interning by content hash.** Theme switches
  no longer rewrite buffers — they mutate `Theme.styles[id]` once.
  Sites that haven't picked a name yet content-hash their `Style`
  into `__anon__/<hex>` keys.
- **Cell `(new, old)` via `mem::replace`.** `Cells::set_dyn` swaps
  in-place, stashes prior `Rc<dyn Any>` on `PendingFire`. Direct
  subscribers fire `(new, old)`; glob subscribers `(name, new, old)`.
- **`include_dir!` collapse for embedded Lua.** Side effect:
  three plugins embedded but never `require`'d (`plan_mode`,
  `predict`, `yank_block`) now autoload — fixed a latent bug
  where the `exit_plan_mode` tool wasn't installed.
- **No big-file decomp for size's sake.** `agent.rs` (1948),
  `ui/mod.rs` (~1600), `window.rs` (1198) stay as-is — single
  coherent structs, splitting fragments state.

## Deferrals

- **P9.i** (provider middleware) — no consumer. Land on first ask.
- **EmmyLua stubs** (`_meta/smelt.lua`) — post-P10 polish.
- **Plugin handle / lifecycle** (`smelt.plugin.define{...}` +
  teardown) — over-engineering for static-file plugin set. Revisit
  when an ecosystem needs hot reload or `/plugins disable`.
- **Reactive statusline binding** (`{ bind = cell, fmt = fn }`).
  Composer ticks once per main-loop iteration. Profile says don't
  bother yet.
- **Plugin manifest blocks** — no `/plugins` UI consumer.
- **Multi-mode `keymap.set({"n","v"}, ...)` sugar** — trivially
  composed in Lua.
- **`vim.ui.set_picker`-style override hook** — no second picker
  exists.
- **Built-in cell name typing** — `set_dyn` already rejects
  undeclared names; protection exists.
- **`smelt.cmd.read_md` bridge** — no consumer asked; .md and Lua
  command systems coexist fine.
