# P9 — Make the architecture true

**Status:** in-progress **Started:** 2026-05-04 **Landed:** —

Close the deferral chain (`P1.a-tail → P4.b → old P9.b`) and purge
places where Rust still encodes Lua-shaped decisions. Hygiene + parity
walk live in P10.

## Decisions made

- **Trust UX is a `/trust` command, not a startup dialog**
  (2026-05-05) — interactive startup prompts couple the loader to
  the event loop. Deny-by-default with a toast + opt-in slash
  command keeps loading sync at startup and lets the user opt in at
  the same time the toast appears.
- **`<config>/plugins/*.lua` autoload at startup** (2026-05-05) —
  built-in `<embedded>/plugins/*.lua` already auto-load via the
  embedded directory walk; mirroring that for `<config>/plugins/`
  closes the "what's a plugin?" question — drop a `.lua` in either
  location and it runs.
- **Loader order: embedded → user init → global plugins → project**
  (2026-05-05) — user init can override embedded defaults; project
  config is last so a per-repo `<cwd>/.smelt/init.lua` always wins.

- **Well-known win/buf ids out of `ui`** (`352de075`) — moved to
  `app/well_known.rs`; `next_win_id` starts at `0`; `prompt_data.rs` →
  `prompt_buf.rs`.
- **`buffer::SpanStyle` unified with `grid::Style`** (`de78d750`) —
  fixes strikethrough render bug; `merge_span_style` OR-merges
  `underline` + `crossedout`.
- **Persisted transcript caches deleted** (`2125f6e0`) —
  `transcript_cache.rs` gone; per-block caching is
  `Buffer::ensure_rendered_at(width)`.
- **Lua `render` hook seeded** (`0a2bf51f`) — tool output rendering
  routed through a Lua callback; `RenderCtx` retired in the next
  decision.
- **`BlockHistory::cache_dirty` deleted** (`b3ca0535`) — write-only
  dead field.
- **`TranscriptSnapshot` moved to `tui`** (`b3ca0535`) — display
  projection is tui-only; `BlockBufferCache` replaces direct
  `history.artifacts` reads.
- **`Buffer` + `BufferParser` + `Style` + `UndoHistory` to `core`**
  (2026-05-05) — shims followed; in a later same-day pass the
  `tui::ui::{buffer,undo,id}` shim files were deleted outright,
  `WinId` moved into `ui/mod.rs` directly, and `crate::ui::buffer::*`
  call sites collapse to `crate::ui::*` (everything they reach for
  is now re-exported from `ui/mod.rs`).
- **`core::style::Color` is a frontend-neutral mirror, not a crossterm
  re-export** (2026-05-05) — same variant set, owned by core; tui
  converts at SGR-emit via `to_crossterm_color`. P9.e replaces raw
  colors in extmarks with `HlGroup`.
- **`render_tool_body` hands Lua a `Buffer` userdata; `RenderCtx`
  retires** (2026-05-05) — `render` hook signature is
  `(args, output, width, buf_id)`; tui mints an ephemeral scratch
  Buffer and projects via `buffer_into_collector` until P9.c lands
  per-block parsers writing to Buffer directly.
- **No more "too big to land" deferrals** (2026-05-05) — only legit
  reason to defer is "doesn't improve the codebase." Size is not a
  reason.
- **Loader override search path** (2026-05-05) — module searcher
  checks `<cwd>/.smelt/runtime/<rel>.lua` →
  `<XDG_DATA_HOME>/smelt/runtime/<rel>.lua` → embedded. New
  `engine::data_dir()` mirrors the existing XDG accessors. Users
  override individual UX modules without forking the tree.
- **Cell `(new, old)` payload** (2026-05-05) — `set_dyn` swaps via
  `mem::replace` and stashes the old `Rc<dyn Any>` on `PendingFire`;
  drain pump projects both through the per-`TypeId` Lua projector so
  every subscriber sees the prior value as a second arg.
- **Tool `hooks` returns the decision** (2026-05-05) — `decide_base`
  + `is_auto_approved` bash branch + `extract_tool_paths` +
  `ActiveTool::elapsed` + `agent.rs:1179` cmd_summary are one
  mistake repeated. Lua hook returns
  `{ decision, summary?, confirm_message?, approval_patterns?,
  paths_outside_workspace? }`; Rust honors it. `needs_confirm +
  approval_patterns + preflight` collapse into `hooks`. Per-leak
  callbacks (`pattern_from_args` / `paths_for_workspace` /
  `elapsed_visible`) drop. P9.d rewritten.
- **`include_dir!` for embedded Lua** (2026-05-05) —
  `EMBEDDED_MODULES` (140) + `BOOTSTRAP_CHUNKS` (30) +
  `AUTOLOAD_MODULES` (30) collapse to one tree walked at startup.
  New P9.l.
- **Single LuaRuntime instance** (2026-05-05) — main.rs builds a
  fresh `LuaRuntime` to extract config, then TuiApp::new builds a
  *second* one and runs `init.lua` again. Two interpreters, two
  `LuaShared` instances; the first's populated state is thrown
  away. Side effects in init.lua (notifies, writes, network) fire
  twice. Fix: build once in main.rs, hand to `TuiApp::new`. New
  P9.m.
- **Vim becomes Lua-extensible** (2026-05-05) — vim.rs's keymap
  dispatch is a hardcoded Rust table; users can't override `dd` in
  init.lua even though every other interactive surface is
  recipe-driven. Per-buffer registers/dot-repeat also belong on
  `Buffer` (Buffer-in-core blocker landed in P9.b). Lands the
  deferred P1.d.5d + P1.d.5f.2d sub-phases. New P9.n. Framing is
  *Lua extensibility parity*, not "vim.rs is big" — file size
  shrinks as a side effect.
- **Prompt is a Window** (2026-05-05) — `PromptState` is the only
  interactive surface in the app that *isn't* a Window over a
  Buffer with a keymap recipe. Edit state → Buffer; cursor/scroll/
  selection → Window; keymap → Lua via `widgets/input.lua`;
  completer → ghost-text extmark + Overlay picker. Same "the new
  shape was always there" principle as P1.d. Lands deferred
  input/mod.rs carve. New P9.o.
- **No big-file decomp for size's sake** (2026-05-05) — sentrux
  flagged `agent.rs` (1948 prod), `ui/mod.rs` (~1600 prod),
  `window.rs` (1198 prod) as equality bottlenecks. They're single
  coherent structs (`Turn`, `Ui`, `Window`) — splitting would
  fragment state or scatter impls cosmetically. P9.d shrinks
  agent.rs organically. The other two stay as-is.
- **P9.l autoload includes `dialogs/`** — `dialogs/{rewind,resume,
  permissions}.lua` register slash commands at top level, so they
  must autoload alongside `tools/` / `commands/` / `plugins/`.
  `dialogs/confirm.lua` is filtered out because it ships earlier
  via `BOOTSTRAP_FILES`.
- **Plan refresh against live code** (2026-05-05) — audit confirmed
  that (a) `Buffer` data layer already carries `NsId` + `ExtmarkId` +
  three payload variants but the keyset is otherwise minimal
  (no priority / gravity / `sign_text` / `hl_eol` / `hl_mode` /
  `conceal` / `id` / `virt_text_pos` / `virt_lines`); (b) the
  `~/.config/smelt/runtime/` override path is already live; (c) the
  one remaining tool-name-match in user-facing dispatch is in
  `dialogs/confirm.lua` (not Rust, but same spirit); (d) statusline
  composer ticks once per main-loop iteration, not per render frame
  — fast enough at current load. P9 trimmed to six items: P9.d
  extends to cover the confirm preview; P9.f grows the data-layer
  keyset; P9.g drops the `tools/*.lua` and `commands/*.lua` user
  folders (plugins are the only user concept, and may register
  tools/commands themselves); P9.h shrinks to `(new, old)` payload;
  P9.i flips to return-payload middleware semantics. Cuts:
  plugin-handle/lifecycle, reactive-statusline binding, manifest
  blocks, multi-mode `keymap.set` sugar, picker-override hook,
  built-in-cell-name typing, "stale-context invalidation",
  `before/after_provider_request` event cells, prompt-section deps.
  EmmyLua stubs (`_meta/smelt.lua`) move to post-P10 polish.

## P9.b — `Buffer` to `core` (the keystone) ✅

`Buffer` is pure data with zero terminal deps; it belongs in core.
With it there, `RenderCtx` collapses, `DisplayBlock` deletes, and
headless reads transcript through the same surface as tui. `Theme`
and parser *implementations* (markdown / diff / syntax / bash) stay
in tui — they pull from tui-only crates (syntect, etc.).

1. ✅ Move `Buffer` + `BufferParser` to `core`.
2. ✅ `render_tool_body` takes `Buffer` userdata; `RenderCtx` deleted.
3. ✅ Delete `DisplayBlock`, `DisplayLine`, `DisplaySpan`. Rewrite
   `SpanCollector` to write directly into a `Buffer` with theme
   resolution at print time. `layout_out.rs` shrinks to the
   collector + `Outcome`. `transcript_present/mod.rs` switches from
   `layout_block(...) -> DisplayBlock` to
   `layout_block_into(buf, theme, block, state, ctx, renderer) -> Outcome`.
4. ✅ `apply_view_state` rewrites the Buffer in place (Collapsed /
   TrimmedHead truncate + ellipsis; TrimmedTail snapshots the tail,
   wipes, renders ellipsis, re-appends the kept content).
5. ✅ `BlockArtifact` + `BlockHistory.artifacts` deleted; the only
   per-block layout cache is `tui::BlockBufferCache`.
6. ✅ `Theme` moves to `core/theme.rs` so renderers can resolve
   colors at print time without pulling tui in. `tui::ui::Theme` and
   `tui::ui::buffer/undo/id` survive only as re-exports until the
   last consumer is rewritten — done in this push for the shim
   files but `tui/src/theme.rs` keeps the bridge layer (presets,
   OSC11 detect, populate_ui_theme).

**Decisions added in this push:**

- **Outcome carries width-pinning info** — `Outcome { line_count,
  layout_width, was_wrapped, max_line_width }` is what
  `SpanCollector::finish` returns. `is_valid_at(new_width)` is the
  cache-validity predicate.
- **`SpanMeta::default()` keeps `selectable: true`** — moved from
  `core::content::display::SpanMeta` to `core::buffer::SpanMeta`;
  derived `Default` would have flipped to `selectable: false` and
  silently broken every selection / copy test.
- **`ensure_rendered_at` clears the buffer first** — parsers
  expect to write from row 0. The append-mode SpanCollector is for
  transcript composition (multiple blocks → one buffer); for
  parser-driven re-render we wipe lines + extmarks first and let
  the collector seed from the empty buffer.
- **Tests get a `test_util::render_test` helper** — wraps a fresh
  `Buffer` + default `Theme` + `SpanCollector`, runs `fill`, and
  reads the buffer back into a `TestBlock { lines, outcome }` shape
  that mirrors the old `DisplayBlock`. Lets the markdown / inline
  parser / transcript-render tests survive the IR removal without
  rewriting their assertions.

## P9.c — Folded into P9.b ✅

P9.c was originally split from P9.b "for size." Per the new no-defer
rule, it landed in the same push. Concretely:

- Per-block layout cache (`BlockBufferCache`) replaces direct
  `BlockHistory.artifacts` reads.
- `TranscriptProjection` shrinks to a composition layer: walks
  `BlockHistory`, asks the cache for each block's Buffer at the
  current `LayoutKey`, concatenates lines + highlights + decorations
  into the transcript display Buffer.
- `TranscriptSnapshot` (the width-keyed plain-text projection used
  for selection / copy / cursor snapping) moves from `core` to
  `tui::content::transcript_snapshot` because it pulls per-block
  Buffers through the cache and reads `SpanMeta` off resolved
  highlights — both tui-side concerns now.

What didn't land in this push (deferred to a follow-up sub-phase
under the same banner):

- **Per-`Block`-variant `BufferParser` impls under
  `tui::content::transcript_parsers/`.** Today each block still
  routes through the central `layout_block_into` switch in
  `transcript_present/mod.rs`. The dispatch is generic; each arm
  could become a `BufferParser` impl writing into the per-block
  Buffer with no protocol change.
- **Prompt rendering through the same parser.** Prompt still has
  its bespoke layout path; unifying it with the user-bubble parser
  is the deferred half.
- **One `copy_range(buf, range)` primitive.** The eight divergent
  copy paths (transcript / prompt / mouse / vim / kill-ring / etc.)
  still exist; folding them onto a single Buffer-aware primitive is
  the natural follow-up.

These are clean follow-up sub-phases against the new shape, not
blockers for the rest of P9.

## P9.d — Tool hook owns the decision

Eternal rule: **no tool/command/dialog/mode name matching anywhere
shared** (Rust *or* Lua dispatch layer). Tool name matches live
only inside that tool's own `.lua` file.

The five `tool_name == "bash" | "web_fetch"` matches in
`permissions/`, `agent.rs`, and `transcript_model.rs` are the same
mistake repeated: Rust is the decision authority and reaches into
tool-shaped knowledge to decide. Fix: **the tool's hook returns the
decision**. Rust honors it.

### Tool callback table

```
{
  name, description, parameters,           -- LLM-facing schema
  execute(call_id, args, ctx),             -- runs the tool
  hooks?(args, mode, ctx) -> {             -- permission decision
    decision = "allow" | "needs_confirm" | "deny",
    summary?, confirm_message?,
    approval_patterns?, paths_outside_workspace?,
  },
  render?(args, output, width, buf),       -- transcript body
  preview?(args, buf),                     -- confirm-dialog body
  summary?(args) -> string,                -- one-liner for header
  modes?, override?,                       -- gating + shadowing
}
```

`needs_confirm + approval_patterns + preflight` collapse into
`hooks`. Per-leakage callbacks (`pattern_from_args` /
`paths_for_workspace` / `elapsed_visible`) drop — `hooks` covers
the path list, `summary` covers the elapsed-display case.

### Lua helpers (Rust FFI, no policy)

```
smelt.permissions.match_tool(mode, name)              -> "allow"|"ask"|"deny"
smelt.permissions.match_subcommand(mode, name, pat)   -> "allow"|"ask"|"deny"
smelt.permissions.is_runtime_approved(name, desc)     -> bool
smelt.permissions.bash_subcommands(cmd)               -> { "git status", ... }
smelt.permissions.default_for(tool, mode)             -> "allow"|"ask"|"deny"
smelt.path.outside_workspace(path)                    -> bool
```

Passive lookups against the rule store + runtime approvals. No
decision logic. The tool's `hooks` callback composes the decision.

### Net deletions in Rust

- `core::permissions::decide_base` — tool-name switch deleted.
  `Permissions::decide()` shrinks to a generic workspace post-check
  or vanishes (hooks return `paths_outside_workspace`).
- `core::permissions::approvals::is_auto_approved` — `tool_name ==
  "bash"` branch deleted. Becomes the `is_runtime_approved` Lua
  helper.
- `core::permissions::workspace::extract_tool_paths` — entire
  match-on-name function deleted. Tools know which arg holds the
  path.
- `core::transcript_model::ActiveTool::elapsed` — match deleted;
  derived from `tool.summary` or a per-tool flag.
- `engine/agent.rs:1179, 1443` — bash cmd_summary branches deleted;
  hook returns `summary`.
- `tui/lua/api/confirm.rs::{render_title_into_buf, _back_tab,
  is_bash}` — deleted; `dialogs/confirm.lua` composes title +
  auto-allow.
- `tui/lua/api/statusline.rs` `AgentMode → glyph` map → `modes.lua`.
- `core/permissions/rules.rs` mode-default table →
  `smelt.permissions.default_for(tool, mode)`.
- `core/custom_commands.rs::CommandOverrides` → generic
  `HashMap<String, RuleOverride>`.
- `runtime/lua/smelt/dialogs/confirm.lua::fill_preview` →
  `tool.preview(buf, args)` generically.

### Verification

After P9.d: `grep -E 'name *== *"(bash|read_file|write_file|edit_file|glob|grep|web_fetch|web_search|notebook_edit|exit_plan_mode|load_skill|ask_user_question)"'` on `crates/{core,engine,tui}/src/` and `runtime/lua/smelt/{dialogs,widgets}/` both return zero hits.

## P9.e — HlGroup-id model

Buffer extmarks carry `HlGroup(u32)`, not raw `Style`. Theme is the
paint-time resolver; switches stop rewriting buffers.

- `core::buffer::HlGroup(u32)` minted by registry; `Theme::resolve`

- Plugins register groups by name (`smelt.theme.set("Warning", …)`).
- Migration via `HlGroupOrStyle` enum, drop `Style` variant when
  every site lands. `core::style::*` survives only at the paint

Deferred from P9.b — separate 100+-site naming pass, independent of
the Buffer move.

## P9.f — Unified extmark keyset (nvim parity)

The data layer already carries `NsId` + `set_extmark` + three payload
variants (`Highlight | Decoration | VirtText`). The Lua surface
hides this behind `add_highlight` / `add_dim`. The keyset itself is
also missing nvim's everyday fields. Two pieces:

- **Extend `core::buffer::ExtmarkOpts`** with the full nvim keyset:
  `priority: u32`, `right_gravity: bool`, `end_right_gravity: bool`,
  `sign_text: Option<String>`, `hl_eol: bool`,
  `hl_mode: Replace | Combine | Blend`, `conceal: Option<String>`,
  `id: Option<ExtmarkId>` (re-target by id),
  `virt_text_pos: Eol | Overlay | Inline | RightAlign`,
  `virt_lines: Vec<Vec<(String, Option<String>)>>`. Paint paths
  honour `priority` (lowest first), gravity (cursor adjacent
  insertion shifts mark), `hl_eol`, `hl_mode`, `conceal`,
  `virt_text_pos`, `virt_lines`. ~250 LOC across `buffer.rs` +
  `to_buffer.rs` + `window.rs` paint.
- **Lua surface collapses to one call.** `smelt.api.create_namespace(name)
  -> NsId` (idempotent — same name returns same id, nvim semantics).
  `smelt.buf.set_extmark(buf, ns, row, col, opts)` — opts mirror
  nvim verbatim plus smelt extensions (`yank: YankSubst`,
  `on_click: fn`). `add_highlight` / `add_dim` retire. ~150 LOC in
  `tui/src/lua/api/buf.rs`.

- **Extract `Rect` to `tui/src/ui/geometry.rs`.** Today `grid.rs`
  imports `Rect` from `layout.rs` and `layout.rs` imports `Grid`
  from `grid.rs` — the lone structural cycle (DSM cluster of size
  2 with 2 internal edges). `Rect` is a primitive (x/y/w/h); it
  doesn't belong next to layout logic. ~30 LOC move; folded here
  because P9.f already touches `to_buffer.rs` + `window.rs` paint.

EmmyLua stubs (`_meta/smelt.lua`) move to post-P10 polish.
Independent of P9.c/d/e.

## P9.g — Plugin auto-discovery + project-local config ✅

Landed: new `smelt_core::trust` module hashes
`<cwd>/.smelt/{init.lua, plugins/*.lua, commands/*.md, runtime/**/*.lua}`
into a SHA-256 digest, persisted to `<XDG_STATE_HOME>/smelt/trust.json`
as `canonical_path → { hash }`. `LuaRuntime::load_global_plugins`
auto-loads every `.lua` under `<config>/plugins/` at startup;
`load_project_config` runs `<cwd>/.smelt/plugins/*.lua` then
`<cwd>/.smelt/init.lua` only when the trust hash matches. Both
`main.rs` and `TuiApp::start` invoke the loaders so config-time
registrations (providers, MCP, permission rules) and runtime-time
registrations (commands, tools, cells) see the same plugin set.

UX is a deny-by-default + opt-in slash command rather than an
interactive startup dialog: untrusted project content fires a
toast "project .smelt/ content not trusted; run /trust to load it";
running `/trust` records the digest and the next launch picks it
up. New `smelt.trust.{status,mark}` Host-tier API drives the
command. Avoids tangling startup with the dialog event loop while
preserving the spec's "trust gate / hash / re-prompt on edit"
semantics.

Order of loaders in `load_plugins()`:
embedded autoload → user `init.lua` → global `<config>/plugins/*.lua`
→ project plugins+`init.lua` (trusted only). Embedded runs first so
user code can override; project last so a per-repo override always
wins.

(Original sketch follows for context.)

User-facing convention is **`init.lua` + `plugins/`**, nothing else.
A plugin is a `.lua` file; it may register tools, commands, keymaps,
cells, statusline sources, anything. There is no separate `tools/`
or `commands/` user folder — those are *internal* organization of
the embedded built-ins (`smelt.tools.*`, `smelt.commands.*`), not a
user contract. If a user wants to ship a tool, they ship a plugin
that registers it.

- **Discovery.** At startup, after `register_api` and the embedded
  bootstrap chunks: scan global `<config>/plugins/*.lua` and load
  each (alphabetical). Then run `<config>/init.lua`. Then scan
  project-local `<cwd>/.smelt/plugins/*.lua` and load each. Then
  run `<cwd>/.smelt/init.lua`. Order: embedded built-ins → global
  plugins → user init → project plugins → project init. Later
  layers can override earlier (`smelt.tools.register` with
  `override = true` already supports this).
- **Trust gate.** `<cwd>/.smelt/` content runs only after first-load
  approval. Hash-based: SHA-256 over each file in the directory,
  persisted to `<XDG_STATE_HOME>/smelt/trust.json` as
  `path → sha`. Mismatch (file edited or new file added) re-prompts.
  Modal options: trust once / trust always / decline. ~80 LOC.
- **Built-in user folder for `commands/*.md`** stays — it's an
  existing feature (P4.e) consumed by the autoloaded
  `plugins/custom_commands.lua`. That plugin already scans
  `smelt.path.commands_dir()` and turns each `.md` into a slash
  command. The plugin extends the same scan to
  `<cwd>/.smelt/commands/*.md` after the project trust gate clears.
  No new user-facing folder convention; just a bigger glob inside an
  existing plugin.

Total: ~150 LOC. Independent of P9.c/d/e/f.

## P9.h — Cell `(new, old)` payload ✅

Landed: `Cells::set_dyn` swaps the slot in-place via
`std::mem::replace`, capturing the prior `Rc<dyn Any>` as
`PendingFire.prev`. The drain pump in `tui::TuiApp::drain_cells_pending`
projects both values through the registered Lua projector and calls
direct subscribers with `(new, old)`, glob subscribers with
`(name, new, old)`. Cell handle `:set(v)` and `smelt.cell.set` route
through the same path — Lua sees the prior value without mirroring
it in plugin state.

The deeper hook-surface ideas (typed built-in names, more event
cells, prompt-section deps, stale-context invalidation) defer
indefinitely — `smelt.prompt.set_section` already covers static
sections, and no concrete consumer has surfaced for the rest. Add
each only when a real plugin needs it.

## P9.i — Provider middleware

Plugin-extensibility seam pi-mono has, smelt doesn't. Without it,
redaction / prompt rewriting / A/B swaps require forking a provider.

- Introduce neutral `ProviderRequest` struct (messages, tools,
  sampling). Each kind-specific serializer (`anthropic` / `openai` /
  `codex` / `copilot`) reads it → wire body. Same for `ProviderResponse`
  on the way back.
- `EngineClient` holds `Vec<Box<dyn ProviderMiddleware>>` with
  `before_request(req: ProviderRequest) -> ProviderRequest` +
  `after_response(resp: ProviderResponse) -> ProviderResponse`.
  Return-payload semantics, not `&mut` — A/B swaps and full
  rewrites are first-class, type system enforces "you handled it."
  Middleware runs in registration order before, reverse after.
- Lua: `smelt.engine.use_middleware(before?, after?)` — same return
  shape.
- ~500 LOC (heaviest sub-phase). The neutral struct is the bulk of
  it; today serializers build wire bodies inline.

## P9.l — Embedded Lua tree ✅

Landed: `EMBEDDED_MODULES` (~140 lines) + `BOOTSTRAP_CHUNKS` (~30)
+ `AUTOLOAD_MODULES` (~30) collapse to one `include_dir!` of the
`runtime/lua/smelt/` tree. `BOOTSTRAP_FILES` is an 8-entry
order-sensitive list (semantic ordering: `_bootstrap` → `dialog` →
widgets → `cmd` → `dialogs/confirm` → `status` → `modes`).
Everything under `tools/`, `commands/`, `plugins/`, `dialogs/`
auto-loads at startup (sorted, with bootstrap files filtered out so
nothing executes twice). Every other `.lua` under the tree is
lazily `require`-able. New `pub fn smelt_core::lua::autoload_modules`
returns the runtime-derived list; tui's `load_autoload` consumes it.

Side effect: three plugins that were embedded but never required
(`plugins/plan_mode`, `plugins/predict`, `plugins/yank_block`) now
auto-load — the latent bug where Plan-mode prompt section + the
`exit_plan_mode` tool weren't installed is fixed. `/yank-block`
becomes always available.

(Original sketch follows for context.)

Three parallel hand-maintained registries today:

- `EMBEDDED_MODULES: &[(&str, &str)]` — 140 lines, `core/lua/runtime.rs:22`,
  every embedded `require`-able module.
- `BOOTSTRAP_CHUNKS: &[(&str, &str)]` — ~30 lines,
  `core/lua/runtime.rs:168`, chunks executed at `register_api`.
- `AUTOLOAD_MODULES: &[&str]` — ~30 lines, `tui/lua/mod.rs:515`,
  modules `require`'d at startup.

Adding any built-in `.lua` file requires editing two of these.
Replace with `include_dir!("runtime/lua/smelt")` (single dependency,
~5 LOC). One walker classifies by path:

- `_bootstrap.lua` / `dialog.lua` / `widgets/*.lua` / `cmd.lua` /
  `dialogs/confirm.lua` / `status.lua` / `modes.lua` —
  `BOOTSTRAP_CHUNKS` equivalent (run at `register_api`).
- everything else — `EMBEDDED_MODULES` equivalent (lazy `require`).
- `tools/*.lua` / `commands/*.lua` / `plugins/*.lua` — autoloaded at
  startup (replaces `AUTOLOAD_MODULES`).

Net delete: ~200 lines of `&[(&str, &str)]` boilerplate. Adding a
new built-in Lua file = drop file, recompile. Pairs naturally with
P9.g — auto-discovery on disk uses the same walker shape as
auto-discovery of the embedded tree.

## P9.m — Single LuaRuntime instance

Today main.rs builds a `LuaRuntime`, runs `init.lua` to populate
config registries (providers, MCP, permission rules, settings
overrides), pulls config out via `to_config()`, builds the engine,
then `TuiApp::new` builds a *second* `LuaRuntime` which calls
`load_user_config` again. `init.lua` runs twice with two separate
`LuaShared` instances; the first runtime's populated state is
thrown away (only `to_config`'s extracted values flow into the
engine config). Side effects in init.lua (notifies, file writes,
network calls, registrations of one-shot autocmds) fire twice.

Fix: build `LuaRuntime` once in main.rs, hand it into `TuiApp::new`.

```
main.rs                             tui::app::TuiApp::new
─────────────────────               ──────────────────────
LuaRuntime::new()
init.lua → LuaShared (config)
to_config() → engine
                          ───────►  takes LuaRuntime arg
                                    install TLS app pointer
                                    autoload (embedded plugins)
                                    global plugins
                                    project plugins
                                    (init.lua does NOT run again)
```

`TuiApp::new` takes a `LuaRuntime` parameter; deletes the
`LuaRuntime::new()` at `app.rs:473`. Autoload + global plugins +
project plugins still run from TuiApp::new (after TLS pointer
install) — only `load_user_config` moves to main.rs. ~50 LOC across
`main.rs`, `startup.rs`, `app.rs`, `tui::lua::mod`.

Independent of every other P9 sub-phase.

## P9.n — Vim becomes Lua-extensible

Smelt promises Lua extensibility everywhere except vim. `vim.rs`'s
keymap dispatch is a hardcoded Rust table (`handle_normal_char`,
`handle_visual_char`, etc.) that users can't override in init.lua —
remap `dd`, add `<leader>w`, anything. Per-buffer dot-repeat and
registers also belong on `Buffer` (matches vim semantics: per-buffer
not per-Window). The Buffer-in-core blocker landed in P9.b.

Lands the two deferred sub-phases:

- **P1.d.5d** — registers / dot-repeat / undo move from
  `VimWindowState` to `Buffer`. Per-buffer state lives where Buffer
  does, available to any consumer (vim recipe, future macro engine,
  plugin scripting).
- **P1.d.5f.2d** — vim dispatch table flattens onto the Lua keymap
  registry. `vim.rs` becomes "the recipe of default keymap entries
  that call into Buffer/Window primitives," like nvim's runtime
  `keymap.lua`. Users override entries via `smelt.keymap.set("n",
  "dd", fn)`.

Framing is **Lua extensibility parity**, not "vim.rs is big." File
size shrinks as a side effect, not as the goal. Lands after P9.f
(needs the unified extmark keyset for visual-mode highlight
extmarks) but otherwise independent.

## P9.o — Prompt is a Window

`PromptState` (1168 prod LOC in `tui/src/input/mod.rs`) is the only
interactive surface in smelt that *isn't* a Window over a Buffer
with a keymap recipe. Everything else (transcript, dialogs, picker,
cmdline) was migrated to that shape in P1.d. Prompt stayed because
it bundles edit state + completer + attachments + paste handling
into one god-struct. The carve:

- **Edit state → Buffer.** Lines, cursor, scroll already-Buffer-shaped.
- **Cursor / scroll / selection → Window** over the prompt Buffer.
- **Keymap → Lua** via `widgets/input.lua` (recipe-style, like
  every other Window).
- **Completer → ghost-text extmark + Overlay picker.** Primitives
  already exist (P1.c overlays, P9.f extmarks).
- **Attachments stay** as a small input-side concern that mutates
  the Buffer alongside paste handling.

Same principle as P1.d ("Window is the only interactive unit").
Removes the last shape outlier in the app. Lands the deferred
input/mod.rs carve. After P9.n (recipe-style flatten) so prompt
input rides the same Lua keymap dispatch path.

## Deferrals

- HlGroup migration → P9.e (post P9.b).
- EmmyLua stubs (`runtime/lua/smelt/_meta/smelt.lua`) → post-P10
  polish; not blocking on plugin extensibility.
- Plugin handle / lifecycle (`smelt.plugin.define{…}` + teardown)
  — over-engineering for the current plugin set. Plugins are static
  files; restart smelt to reload. Revisit only if a real ecosystem
  needs hot reload or `/plugins disable`.
- Reactive statusline binding (`{ bind = cell, fmt = fn }`).
  Composer ticks once per main-loop iteration, runs in microseconds.
  Doctrine says push-over-pull; profile says don't bother yet.
  Revisit when streaming hits real cost.
- Plugin manifest blocks (`smelt.plugin.manifest{name, version, …}`).
  No `/plugins` UI consumer; decoration without payoff.
- Multi-mode `keymap.set({"n","v"}, …)` sugar. Trivially composed in
  Lua: `for _, m in ipairs(modes) do smelt.keymap.set(m, …) end`. No
  Rust change needed.
- `vim.ui.set_picker`-style override hook. No second picker exists.
  Add when one shows up.
- Built-in cell name typing (Rust enum lookup). Typo-as-no-op is
  rare in practice; runtime checks already reject undeclared names.
