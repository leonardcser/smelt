## P9.b — `Buffer` to `core` (the keystone)

`Buffer` is pure data with zero terminal deps; it belongs in core.
With it there, `RenderCtx` collapses, `DisplayBlock` deletes, and
headless reads transcript through the same surface as tui. `Theme`
and parser *implementations* (markdown / diff / syntax / bash) stay
in tui — they pull from tui-only crates (syntect, etc.).

1. ✅ Move `Buffer` + `BufferParser` to `core`.
2. ✅ `render_tool_body` takes `Buffer` userdata; `RenderCtx` deleted.
3. Delete `DisplayBlock`, `DisplayLine`, `SpanCollector`,
   `layout_out.rs`, `transcript_present/` rendering glue.
4. P9.c starts from the new shape.

## P9.c — Transcript pipeline as `BufferParser` impls

The path `Block → BufferParser → Buffer → Window::render` lands end
to end. Closes the P1.a-tail / P4.b deferral chain.

- One parser per `Block` variant under
  `tui::content::transcript_parsers/`; each writes lines + extmarks
  into a per-block `Buffer`.
- Width-independent parsing at ingest (markdown AST, diff LCS,
  syntect tokens as namespaced extmarks); `ensure_rendered_at(width)`
  re-flows on resize.
- `TranscriptProjection` shrinks to composition only;
  `BlockBufferCache` is the only cache.
- Prompt input goes through the same parser — deduplicates with
  user-bubble rendering; `PromptWrap` deletes or shrinks to byte-map
  utility.
- One `copy_range(buf, range)` replaces the eight divergent copy
  paths (transcript / prompt keybind / mouse / vim / kill-ring /
  etc.).
- `responsive_line(spans, width)` shared between
  `content/status.rs` and `content/prompt_buf.rs::bar_row`.

## P9.d — Tool hook owns the decision

Eternal rule: **no tool/command/dialog/mode name matching anywhere
shared** (Rust *or* Lua dispatch layer). Tool name matches live
only inside that tool's own `.lua` file.

The five `tool_name == "bash" | "web_fetch"` matches in
`permissions/`, `agent.rs`, and `transcript_model.rs` are the same
mistake repeated: Rust is the decision authority and reaches into
tool-shaped knowledge to decide. Fix: **the tool's hook returns the
decision**. Rust honors it.

### Tool callback table

```
{
  name, description, parameters,           -- LLM-facing schema
  execute(call_id, args, ctx),             -- runs the tool
  hooks?(args, mode, ctx) -> {             -- permission decision
    decision = "allow" | "needs_confirm" | "deny",
    summary?, confirm_message?,
    approval_patterns?, paths_outside_workspace?,
  },
  render?(args, output, width, buf),       -- transcript body
  preview?(args, buf),                     -- confirm-dialog body
  summary?(args) -> string,                -- one-liner for header
  modes?, override?,                       -- gating + shadowing
}
```

`needs_confirm + approval_patterns + preflight` collapse into
`hooks`. Per-leakage callbacks (`pattern_from_args` /
`paths_for_workspace` / `elapsed_visible`) drop — `hooks` covers
the path list, `summary` covers the elapsed-display case.

### Lua helpers (Rust FFI, no policy)

```
smelt.permissions.match_tool(mode, name)              -> "allow"|"ask"|"deny"
smelt.permissions.match_subcommand(mode, name, pat)   -> "allow"|"ask"|"deny"
smelt.permissions.is_runtime_approved(name, desc)     -> bool
smelt.permissions.bash_subcommands(cmd)               -> { "git status", ... }
smelt.permissions.default_for(tool, mode)             -> "allow"|"ask"|"deny"
smelt.path.outside_workspace(path)                    -> bool
```

Passive lookups against the rule store + runtime approvals. No
decision logic. The tool's `hooks` callback composes the decision.

### Net deletions in Rust

- `core::permissions::decide_base` — tool-name switch deleted.
  `Permissions::decide()` shrinks to a generic workspace post-check
  or vanishes (hooks return `paths_outside_workspace`).
- `core::permissions::approvals::is_auto_approved` — `tool_name ==
  "bash"` branch deleted. Becomes the `is_runtime_approved` Lua
  helper.
- `core::permissions::workspace::extract_tool_paths` — entire
  match-on-name function deleted. Tools know which arg holds the
  path.
- `core::transcript_model::ActiveTool::elapsed` — match deleted;
  derived from `tool.summary` or a per-tool flag.
- `engine/agent.rs:1179, 1443` — bash cmd_summary branches deleted;
  hook returns `summary`.
- `tui/lua/api/confirm.rs::{render_title_into_buf, _back_tab,
  is_bash}` — deleted; `dialogs/confirm.lua` composes title +
  auto-allow.
- `tui/lua/api/statusline.rs` `AgentMode → glyph` map → `modes.lua`.
- `core/permissions/rules.rs` mode-default table →
  `smelt.permissions.default_for(tool, mode)`.
- `core/custom_commands.rs::CommandOverrides` → generic
  `HashMap<String, RuleOverride>`.
- `runtime/lua/smelt/dialogs/confirm.lua::fill_preview` →
  `tool.preview(buf, args)` generically.

## P9.e — HlGroup-id model

Buffer extmarks carry `HlGroup(u32)`, not raw `Style`. Theme is the
paint-time resolver; switches stop rewriting buffers.

- `core::buffer::HlGroup(u32)` minted by registry; `Theme::resolve`
  in core, native paint mapping in each frontend.
- Plugins register groups by name (`smelt.theme.set("Warning", …)`).
- Migration via `HlGroupOrStyle` enum, drop `Style` variant when
  every site lands. `core::style::*` survives only at the paint
  boundary.

Deferred from P9.b — separate 100+-site naming pass, independent of
the Buffer move.

## P9.f — Unified extmark keyset (nvim parity)

The data layer already carries `NsId` + `set_extmark` + three payload
variants (`Highlight | Decoration | VirtText`). The Lua surface
hides this behind `add_highlight` / `add_dim`. The keyset itself is
also missing nvim's everyday fields. Two pieces:

- **Extend `core::buffer::ExtmarkOpts`** with the full nvim keyset:
  `priority: u32`, `right_gravity: bool`, `end_right_gravity: bool`,
  `sign_text: Option<String>`, `hl_eol: bool`,
  `hl_mode: Replace | Combine | Blend`, `conceal: Option<String>`,
  `id: Option<ExtmarkId>` (re-target by id),
  `virt_text_pos: Eol | Overlay | Inline | RightAlign`,
  `virt_lines: Vec<Vec<(String, Option<String>)>>`. Paint paths
  honour `priority` (lowest first), gravity (cursor adjacent
  insertion shifts mark), `hl_eol`, `hl_mode`, `conceal`,
  `virt_text_pos`, `virt_lines`. ~250 LOC across `buffer.rs` +
  `to_buffer.rs` + `window.rs` paint.
- **Lua surface collapses to one call.** `smelt.api.create_namespace(name)
  -> NsId` (idempotent — same name returns same id, nvim semantics).
  `smelt.buf.set_extmark(buf, ns, row, col, opts)` — opts mirror
  nvim verbatim plus smelt extensions (`yank: YankSubst`,
  `on_click: fn`). `add_highlight` / `add_dim` retire. ~150 LOC in
  `tui/src/lua/api/buf.rs`.

EmmyLua stubs (`_meta/smelt.lua`) move to post-P10 polish.
Independent of P9.c/d/e.

## P9.g — Plugin auto-discovery + project-local config

User-facing convention is **`init.lua` + `plugins/`**, nothing else.
A plugin is a `.lua` file; it may register tools, commands, keymaps,
cells, statusline sources, anything. There is no separate `tools/`
or `commands/` user folder — those are *internal* organization of
the embedded built-ins (`smelt.tools.*`, `smelt.commands.*`), not a
user contract. If a user wants to ship a tool, they ship a plugin
that registers it.

- **Discovery.** At startup, after `register_api` and the embedded
  bootstrap chunks: scan global `<config>/plugins/*.lua` and load
  each (alphabetical). Then run `<config>/init.lua`. Then scan
  project-local `<cwd>/.smelt/plugins/*.lua` and load each. Then
  run `<cwd>/.smelt/init.lua`. Order: embedded built-ins → global
  plugins → user init → project plugins → project init. Later
  layers can override earlier (`smelt.tools.register` with
  `override = true` already supports this).
- **Trust gate.** `<cwd>/.smelt/` content runs only after first-load
  approval. Hash-based: SHA-256 over each file in the directory,
  persisted to `<XDG_STATE_HOME>/smelt/trust.json` as
  `path → sha`. Mismatch (file edited or new file added) re-prompts.
  Modal options: trust once / trust always / decline. ~80 LOC.
- **Built-in user folder for `commands/*.md`** stays — it's an
  existing feature (P4.e) consumed by the autoloaded
  `plugins/custom_commands.lua`. That plugin already scans
  `smelt.path.commands_dir()` and turns each `.md` into a slash
  command. The plugin extends the same scan to
  `<cwd>/.smelt/commands/*.md` after the project trust gate clears.
  No new user-facing folder convention; just a bigger glob inside an
  existing plugin.

Total: ~150 LOC. Independent of P9.c/d/e/f.

## P9.h — Cell `(new, old)` payload ✅

Landed: `Cells::set_dyn` swaps the slot in-place via
`std::mem::replace`, capturing the prior `Rc<dyn Any>` as
`PendingFire.prev`. The drain pump in `tui::TuiApp::drain_cells_pending`
projects both values through the registered Lua projector and calls
direct subscribers with `(new, old)`, glob subscribers with
`(name, new, old)`. Cell handle `:set(v)` and `smelt.cell.set` route
through the same path — Lua sees the prior value without mirroring
it in plugin state.

The deeper hook-surface ideas (typed built-in names, more event
cells, prompt-section deps, stale-context invalidation) defer
indefinitely — `smelt.prompt.set_section` already covers static
sections, and no concrete consumer has surfaced for the rest. Add
each only when a real plugin needs it.

## P9.i — Provider middleware

Plugin-extensibility seam pi-mono has, smelt doesn't. Without it,
redaction / prompt rewriting / A/B swaps require forking a provider.

- Introduce neutral `ProviderRequest` struct (messages, tools,
  sampling). Each kind-specific serializer (`anthropic` / `openai` /
  `codex` / `copilot`) reads it → wire body. Same for `ProviderResponse`
  on the way back.
- `EngineClient` holds `Vec<Box<dyn ProviderMiddleware>>` with
  `before_request(req: ProviderRequest) -> ProviderRequest` +
  `after_response(resp: ProviderResponse) -> ProviderResponse`.
  Return-payload semantics, not `&mut` — A/B swaps and full
  rewrites are first-class, type system enforces "you handled it."
  Middleware runs in registration order before, reverse after.
- Lua: `smelt.engine.use_middleware(before?, after?)` — same return
  shape.
- ~500 LOC (heaviest sub-phase). The neutral struct is the bulk of
  it; today serializers build wire bodies inline.

## P9.j — Loader override search path ✅

Landed: mlua searcher hook in `core/lua/runtime.rs` walks
`<cwd>/.smelt/runtime/<rel>.lua` →
`<XDG_DATA_HOME>/smelt/runtime/<rel>.lua` → embedded fallback. New
`engine::data_dir()` (`<XDG_DATA_HOME>/smelt`, default
`~/.local/share/smelt`) sits alongside `config_dir / state_dir /
cache_dir`. Default UX modules stay `include_str!`'d in the binary,
so `cargo install smelt` keeps working with no external state.

## P9.k — Honor `ToolExecutionMode::Parallel`

Already in protocol; agent loop runs everything sequentially. Tool
calls marked `Parallel` (read-only — `read_file`, `glob`, `grep`,
`web_fetch`, `web_search`) get `tokio::join_all` in the agent loop.
Throughput win for read-heavy turns.

## P9.l — Embedded Lua tree

Three parallel hand-maintained registries today:

- `EMBEDDED_MODULES: &[(&str, &str)]` — 140 lines, `core/lua/runtime.rs:22`,
  every embedded `require`-able module.
- `BOOTSTRAP_CHUNKS: &[(&str, &str)]` — ~30 lines,
  `core/lua/runtime.rs:168`, chunks executed at `register_api`.
- `AUTOLOAD_MODULES: &[&str]` — ~30 lines, `tui/lua/mod.rs:515`,
  modules `require`'d at startup.

Adding any built-in `.lua` file requires editing two of these.
Replace with `include_dir!("runtime/lua/smelt")` (single dependency,
~5 LOC). One walker classifies by path:

- `_bootstrap.lua` / `dialog.lua` / `widgets/*.lua` / `cmd.lua` /
  `dialogs/confirm.lua` / `status.lua` / `modes.lua` —
  `BOOTSTRAP_CHUNKS` equivalent (run at `register_api`).
- everything else — `EMBEDDED_MODULES` equivalent (lazy `require`).
- `tools/*.lua` / `commands/*.lua` / `plugins/*.lua` — autoloaded at
  startup (replaces `AUTOLOAD_MODULES`).

Net delete: ~200 lines of `&[(&str, &str)]` boilerplate. Adding a
new built-in Lua file = drop file, recompile. Pairs naturally with
P9.g — auto-discovery on disk uses the same walker shape as
auto-discovery of the embedded tree.

## Deferrals

- HlGroup migration → P9.e (post P9.b).
- EmmyLua stubs (`runtime/lua/smelt/_meta/smelt.lua`) → post-P10
  polish; not blocking on plugin extensibility.
- Plugin handle / lifecycle (`smelt.plugin.define{…}` + teardown)
  — over-engineering for the current plugin set. Plugins are static
  files; restart smelt to reload. Revisit only if a real ecosystem
  needs hot reload or `/plugins disable`.
- Reactive statusline binding (`{ bind = cell, fmt = fn }`).
  Composer ticks once per main-loop iteration, runs in microseconds.
  Doctrine says push-over-pull; profile says don't bother yet.
  Revisit when streaming hits real cost.
- Plugin manifest blocks (`smelt.plugin.manifest{name, version, …}`).
  No `/plugins` UI consumer; decoration without payoff.
- Multi-mode `keymap.set({"n","v"}, …)` sugar. Trivially composed in
  Lua: `for _, m in ipairs(modes) do smelt.keymap.set(m, …) end`. No
  Rust change needed.
- `vim.ui.set_picker`-style override hook. No second picker exists.
  Add when one shows up.
- Built-in cell name typing (Rust enum lookup). Typo-as-no-op is
  rare in practice; runtime checks already reject undeclared names.
